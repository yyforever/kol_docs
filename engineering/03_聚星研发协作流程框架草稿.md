# 聚星研发协作流程框架草稿

> 状态:v0.6.1 草稿
> 用途:把聚星的研发协作从"线性推进流程"升级为"Agent 时代的产品研发操作系统"
> 相关范围:聚星跨角色协作流程优化
> v0.6 主要变化(相对 v0.5):引入"产品流程 + Harness 流程"双流程架构;新增 Harness ownership 模型;把 eval、context engineering、品味编码显式化为一级机制
> v0.6.1 降噪修订:修正交叉引用漂移;入口检查改为分层(light / standard / heavy);把若干前沿一手材料的具体数字从"普适原则"降级为"参考起点";把 Harness owner 从"岗位结论"改为"责任模型 + 升级条件"
>
> 材料分层约定:文档中的具体规则分两类标注 ——
> - **行业共识 / 一手材料支撑**:有明确公开源可对照
> - **聚星当前经验规则 / 试运行假设**:基于本地经验或待验证的判断,优先级可随试跑调整

---

## 1. 北极星目标

这份流程的目标,不只是把角色串起来、减少交接损耗,而是建立一套能够持续:

- 低成本试错
- 并行跑原型
- 快速筛选方向
- 用真实项目验证
- 把有效做法沉淀成可被 agent 直接执行的能力包

的研发操作系统。

换句话说,我们最终要的不是"更顺的传统流程",而是:

> 让聚星能像一个持续生成原型、筛选方案、沉淀经验、快速交付的 Agent 研发组织一样运转。

并且在这个过程中,把"让下一轮更便宜"当成和"完成当前轮"同等重要的目标。

## 2. 协作流程为什么会演化

### 2.1 核心原理
协作流程不是抽象信仰,而是组织围绕当前主瓶颈分配注意力、责任和节奏的方式。

更具体地说:

- 软件开发方法论,本质上是在优化当时最贵、最慢、最危险的那个环节。
- 当某个环节的成本塌陷,瓶颈就会转移到下一个环节。
- 瓶颈一旦转移,协作流程就必须跟着重组,否则旧流程会开始优化已经不再重要的问题。

所以,决定协作方式的,不只是"编码成本",而是整条链路的成本结构:

- 变更成本
- 集成成本
- 反馈等待时间
- 协调成本
- 判断成本
- 信任成本
- 跨 agent / 跨会话的上下文一致性成本(Agent 时代新出现的)

底层逻辑始终是同一个:

1. 某个环节变便宜了
2. 新瓶颈暴露出来了
3. 流程开始围绕新瓶颈重组

### 2.2 演化历史
#### 第一阶段:阶段式 / 瀑布式组织
- 当写代码、改代码、部署和返工都很贵时,最自然的做法就是把工作切成顺序阶段,尽量在开工前把事情想清楚。
- 这一代流程优化的核心问题是:如何减少高成本返工。
- 后来被广泛引用的 Royce 论文,经常被简化成"纯线性瀑布",但原文其实已经提醒了纯顺序模式本身有风险,需要补反馈。

#### 第二阶段:敏捷
- 当编码、测试、版本控制和迭代本身开始变便宜,真正昂贵的就不再只是"写代码",而是"写了很久才发现方向错了"。
- 敏捷解决的是:如何在高不确定性环境里缩短反馈回路,让团队尽快知道自己是不是在做错事。
- 它不是反规划、反文档,而是反对用长周期和重交接去掩盖判断的不确定性。

#### 第三阶段:DevOps / 持续交付 / 精益
- 当开发团队内部的迭代速度提升后,新的瓶颈变成了集成、部署、跨团队 handoff,以及"写完之后多久才能看到真实反馈"。
- 这个阶段看起来方法很多,是因为不同团队卡住的子问题不同:
  - 有的卡在开发和运维之间的墙
  - 有的卡在部署和验证
  - 有的卡在需求膨胀
  - 有的卡在大团队协调
- 所以 DevOps、CI/CD、Lean、Shape Up、各种团队模型看起来很碎,但底层都在解决同一类问题:加速反馈,降低交付摩擦。

#### 第四阶段:Agent 时代(我们现在所处)
- 现在新的变化,不只是"代码更快写出来",而是整个执行层的边际成本都在快速下降:
  - 第一版实现
  - 变体生成
  - 文档草拟
  - 上下文整理
  - 初步 review
- 当这些动作越来越便宜,真正昂贵的环节就开始转向:
  - 问题定义
  - 方向选择
  - 质量判断
  - 试跑验证
  - 组织记忆
- 新瓶颈不是"做不出来",而是"做出来太多、判断跟不上"。
- 同时出现了一类以前不存在的新成本:**跨 agent、跨会话的上下文一致性成本**。这在过去用人脑就能吸收,现在必须工程化解决。
- 这意味着新的协作流程重点,不再是"如何组织生产",而是:
  - 如何组织判断、验证与沉淀
  - 如何持续维护 agent 的执行环境(上下文、工具、约束、评测)—— 这件事本身是一条独立流程

#### 第五阶段:未来可能的方向
- 如果未来一部分判断也被自动化了,瓶颈还会继续上移。
- 那时更稀缺的,可能是:
  - 品味
  - 信任
  - 价值取舍
  - 对用户和业务的真实理解
- 这也是为什么体验、信任和最终判断,在当前文档里必须被单独强调。

### 2.3 这和聚星当前要改什么有关
聚星现在面对的,不是"开发速度还不够快"这个老问题,而是一个新的结构性问题:

- 生成能力已经显著提速
- 但判断、筛选、验证和复用还没有同比提速
- 已经注册的 14+ 个 graph 意味着 agent 执行环境的复杂度,超过了任何单个工程师能顺手维护的范围

如果流程继续围绕"如何更高效地产出代码"来组织,团队很容易出现三个后果:

- 原型越来越多,但筛不动
- 做出来的东西越来越快,但方向错误暴露得不够快
- 每个 graph 独立维护低质量上下文,skill 库成为永远在"待归档"的坟场

所以这套流程要同时做两件以前不做的事:

- 产品流程:围绕判断、筛选、验证组织
- Harness 流程:持续编码 agent 的执行环境(上下文、工具、约束、评测)

这两条流程节奏不同、所有者不同、产出不同,必须分开设计,不能混成一条时间轴(详见 §4)。

## 3. 为什么我们现在要这样改

当前最核心的变化不是"AI 能帮忙写代码",而是:

- 执行成本已经显著下降,很多想法不需要先长时间讨论才能验证。
- 真正的瓶颈越来越少是"做不出来",越来越多是:
  - 怎么选方向
  - 怎么做取舍
  - 哪个方案体验更好
  - 怎么让用户真的敢用、会用、持续用
- 团队经验如果不沉淀成可复用的、可被 agent 直接加载的能力包,后面只能反复靠人肉同步。
- 同时,差异化正越来越多地从"用什么模型"转移到"如何包裹模型" —— agent 的执行环境(harness)正成为组织级别的持久竞争力(行业方向:Anthropic、Cognition、OpenAI 近期材料都把 agent 平台 / harness 作为差异化面来讨论)。

所以这套流程必须同时做到三件事:

- 做产品
- 做方法沉淀
- 做 harness 自身的演化

## 4. 双流程架构

本节是 v0.6 最重要的新增,下面所有章节都建立在这个区分上。

### 4.1 两条流程并行

| 维度 | Process A:产品流程 | Process B:Harness 流程 |
|---|---|---|
| 周期 | 天到周 | 持续、非项目化 |
| 驱动 | 业务目标、用户反馈、项目立项 | Process A 每一轮暴露的 harness 缺陷 |
| 产出 | 用户可见的功能、产品决策 | agent 的执行环境:上下文、工具、约束、评测、skill、playbook |
| 主要参与者 | 产品 / 设计 / 研发 owner + agent | Harness owner(兼任或专职,见 §6.1)+ 各 graph 的研发 |
| 节奏感 | 有明确起止、有时间盒 | 持续演化、每周有改动 |

### 4.2 为什么必须分开

- Process A 的节奏受业务压力控制,不可能停下来做基础设施优化。
- Process B 的价值靠复利累积,如果交给项目制团队,会被业务优先级永久压到最低。
- 两条流程混在一起时,agent 平台质量只能由各个 graph 工程师"顺手维护",结果就是 14 个 graph 各自维护低质量、互不共享的 harness。

### 4.3 两条流程的接口

- Process A 的每一轮,必须向 Process B 提交至少一条 harness 反馈(见 §5.11)。
- Process B 的每次改动,必须明确影响哪些 Process A 入口(哪些 graph、哪些角色的工作方式会被改变)。
- Harness owner 每周对齐一次 Process A 的 owner,确认本周 harness 改动是否有盲区。

## 5. 核心原则

### 5.1 原型优先,不是讨论优先
- 一个方向不要默认只做一个方案。
- 能低成本验证的,就尽量先出原型再讨论。
- 如果一个问题存在明显分歧,优先并行做 2 到 5 个轻量版本,而不是长时间口头争论。
- 但不是任何想法都直接进原型池,先要过一轮足够短、但足够清楚的入口检查(分层,见 §7.2)。

### 5.2 真实项目优先,不做空转流程
- 流程不是独立项目。
- 流程必须在真实项目中试跑,通过真实任务暴露问题。
- 试跑中发现的问题,反过来修正流程本身。

### 5.3 短周期判断,不做长周期空规划
- 关注下周、下下周怎么推进。
- 允许大方向存在,但不依赖长周期刚性 roadmap 才能行动。
- 任何跨月的大计划,都要拆成能在短周期内验证的具体动作。
- 每一轮都要有明确的投入上限和退出条件,不默认延期。

### 5.4 Read 并行、Write 串行
- 探索、调研、变体生成 —— 可以 N 路并行,甚至鼓励并行。
- 方案收口、实现落地、决策下注 —— 单线程 + 完整上下文。
- 违反这条原则是 agent 协作失败最常见的根本原因:并行的 write agent 彼此看不到对方决策,产出互相冲突。
- 这条原则贯穿 §6 所有阶段,特别体现在:原型池生成(Read 并行)vs 设计收口(Write 串行)vs 最终拍板(Write 串行)。

### 5.5 Skills 是一级产物,且必须嵌进流程才算沉淀
- 有效的方法、提示词、检查清单、角色 SOP,要沉淀成可复用文本。
- `skills / SOP / playbook` 不是边角料,而是研发系统的一部分。
- 每次试跑,不只是交付产品,也要看是否产生了新的流程资产。
- 但如果一个 skill 只是被写进文档库、不会在实际流程里被 agent 调用、引用或加载,它就不算真正沉淀。

### 5.6 体验和信任是流程目标的一部分
- 不只关注"功能能不能做出来",还要关注:
  - 用户敢不敢用
  - 第一次上手顺不顺
  - 多角色协作时是否容易理解和接管
- onboarding、可见性、可控性、交付结果的稳定性,都属于流程成功标准。

### 5.7 人类判断是核心资产,不是瓶颈开销
- 原型可以并行做很多,但最终还是要有人做选择。
- 流程里必须明确:
  - 谁拍板
  - 按什么标准选
  - 什么情况下继续并行
  - 什么情况下停止探索、进入收敛
- 原型失败不是例外,而是正常结果;关键是要尽快知道它该不该被砍掉。
- 判断要分层,不是所有决策都堆给同一个拍板人。
- 判断要被编码:能说清楚的品味,应该走上 `taste → rubric → eval → automation` 这条管线(见 §8.5),逐步变成组织资产而不是停在某个人脑子里。

### 5.8 设计要前置进原型池,而不是后置做美化
- 设计不是等 PRD 和代码都差不多了,再出来"补一层稿"。
- 当执行成本足够低时,设计应该直接参与原型生成、方向筛选和最终收口。
- 设计的核心价值,不只是画页面,而是判断体验是否成立:
  - 信息层次是否清楚
  - 首次上手是否顺
  - 信任感是否足够
  - 哪些细节决定用户愿不愿意继续用

### 5.9 设计判断要沉淀成规则,而不是只靠感觉
- 高质量设计判断里,有很大一部分其实可以被说清楚、记下来、复用。
- 能稳定复用的内容,要尽量沉淀成:
  - 参数规则
  - 组件约束
  - review checklist
  - prompt cards
  - `UI / design skills`
- 对某些具体决策,允许临时生成一次性调节工具(滑块、切换器、对照面板),用完就扔,但结论要被固化。

### 5.10 流程必须有显式回路,不是只会往前推
- 设计收口发现方案方向不对时,要能退回原型层,而不是只做局部修补。
- 试跑发现问题定义错了时,要能退回输入层,而不是只在当前版本上硬修。
- 新沉淀出来的 skill / SOP / template,要直接改变下一轮 pitch 和 agent 的工作方式,而不是只存起来。

### 5.11 每一轮必须让 harness 变强 1%
- Process A 的每一轮结束,除了交付产品结论,还必须产出至少一项 harness 改动:
  - 一条新工具被加入 / 一条旧工具被删除
  - 一条 CLAUDE.md 或 skill 规则被改写
  - 一个 eval case 被加入回归集
  - 一个 context 边界(write / select / compress / isolate)被重新划定
  - 一条 playbook 的 postcondition 或 forbidden action 被澄清
- 如果一轮结束时 harness 没有任何改动,视为这一轮没有产生组织复利。
- 这条原则把"短期交付"和"长期能力"之间的 tradeoff 内化到每一次迭代里,避免只有当下输出、没有沉淀的惯性。

## 6. 角色模型

当前流程先按以下角色协作,后续根据实际项目继续细化:

- 产品 / 需求 owner
- 设计
- 前后端研发
- 测试 / 验收
- 上线 / 运营协同
- 业务反馈输入方
- **Harness owner**(v0.6 新增,可兼任或专职,见 §6.1)
- Agent / automation 执行层

其中,Agent 不是"替代某个人"的角色,而是嵌在多个角色工作流中的执行层。

### 6.1 Harness ownership(v0.6 新增,分阶段演进)

Harness 的维护不能散落在各 graph 工程师的顺手活里,需要一个明确的 ownership 模型。但是否要马上设立专职岗位,取决于聚星当前规模;一开始用兼任即可,有明确信号时再升级。

**核心职责(始终要有人负责)**
- 维护 `CLAUDE.md / AGENTS.md / 全局 skill 库`
- 维护 eval harness(golden cases、回归集、evaluator 配置)
- 维护 context 架构(write / select / compress / isolate 策略)
- 维护工具集(哪些工具暴露给哪些 agent、定期审计)
- 跨 graph 识别重复模式

**承载方式,按阶段递进**

阶段一 —— 兼任 Harness owner(试跑期默认)
- 由现有研发兼任,明确时间配额(建议不少于 30%)
- 定期(每周或每 sprint)回答一次:这段时间 harness 有哪些改动、还有哪些缺口
- 有权跨 graph 提建议,跨 graph 的统一约束需通过 §4.3 的接口机制与 Process A owner 对齐

阶段二 —— 专职 Harness Engineer / Agent Platform Owner(触发式升级)
- 下列信号任意出现两个,启动升级讨论:
  - 跨 graph 审计发现重复能力或冗余工具 ≥ 3 处
  - 兼任者一个月内 ≥ 2 次被业务优先级挤走 harness 工作
  - 新增 graph 的 onboarding 时间持续增加
  - 某个 graph 的 harness 改动与其他 graph 冲突或被回退
- 升级后主要职责扩展到:跨 graph 平台化能力、公共组件抽取、eval 基础设施的自动化投入

**不独立完成的事**
- 不替代各 graph 工程师对自己业务领域的判断
- 不独自决定业务优先级
- 跨 graph 变更必须和对应 owner 对齐,而不是强推

**风险说明(聚星当前经验规则)**:如果 harness 职责长期无人稳定承担,14+ graph 的执行环境容易出现"各做各的、互不共享"的状态。这是聚星当前规模下需要主动防范的风险,建议阶段一就显式 owner 化。

### 6.2 设计

设计在这套流程里的职责,不应该再只理解成"静态稿输出",而应该明确包括:

- 参与原型生成与第一轮筛选
- 对关键体验面做最终收口
- 为 onboarding / trust / 信息架构负责
- 把有效的设计判断沉淀成规则、清单和 skills
- 在需要时,要求 agent / 研发快速生成临时调节工具来辅助决策

### 6.3 Agent / automation

Agent 的职责需要明确限定:

- 产品侧:起草 pitch、归纳用户反馈、整理候选方案
- 设计侧:生成方案变体、辅助做页面对比、生成临时调节工具
- 研发侧:生成第一版实现、补测试 / 排查 / review、做并行原型验证
- 共同约束:
  - Agent 可以先做第一轮,但不能自己决定进入下一环节
  - 任何要流转给下一角色的内容,都必须经过对应 owner 确认
  - 如果 agent 在某个环节连续两轮都需要大幅返工,应向 Harness owner 报告,评估是否是 harness 问题(而不是默默接受返工)

### 6.4 非传统角色的 agent 自助

前沿公司的观察:法务、市场、运营、数据科学家可以直接用 agent 出实现,不必每次都走研发。聚星在合适场景下,也应该允许:

- 任何 owner + 他自己的 agent pod 独立跑完 pitch → 原型 → 试跑
- 研发的角色重新定义为"提供底座、审高风险改动、维护 harness",而不是"所有实现必须流经研发"

## 7. Process A:产品流程主循环

本节承载 v0.5 的 §5,保留原有骨架,并把每个阶段和 Harness 的关系明确化。

### 7.1 输入与分流
- 明确输入来源:
  - 用户反馈
  - 销售 / 运营反馈
  - 内部策略调整
  - 技术问题 / 线上问题
  - 流程问题
- 判断输入属于哪一类:
  - 新需求
  - 优化项
  - 问题修复
  - 流程类事项
  - 需要原型验证的方向判断
- 事件驱动入口:某些输入源应该直接触发 agent 自动起一个 session,而不是先进待排队池 —— 比如线上 bug、特定用户反馈关键词、eval 回归失败。这条通路由 Harness owner 配置。

### 7.2 入口检查(分层)

任一方向在进入原型池前,先过一次入口检查。**具体要求按方向规模分三档,不是所有事情都要写三件** —— 统一硬门槛会把 bug fix 也拖慢,反而和 §5.1 "低成本试错"、§5.3 "短周期判断"冲突。

**Light — bug、轻量优化、探索性 spike**
- 只要一段 pitch 即可(一句话 + 预估时间盒也可)
- 不要求 spec 和 eval,除非发现会碰线上核心路径
- 典型场景:UI 小调整、已知 bug 修复、< 1 天的 spike

**Standard — 常规功能、明确范围的优化**
- pitch + 简化的 spec
- eval 可选;如果改动会影响已有 golden cases 必须跑回归
- 典型场景:已有 graph 的新能力、单 graph 内的重构

**Heavy — 新方向、新 graph、用户可见的大变化**
- pitch + spec + 3 到 5 条 golden eval cases(含至少一条 forbidden action)
- 向 Harness owner 报备,评估是否触发 context / 工具 / skill 变动
- 典型场景:新 graph 立项、跨 graph 的能力整合、主流程改写

**档位选择**:由发起人初判。L2 角色筛(§7.4)中的 owner 有权要求升档 —— 如发现某个 "Standard" 实际上碰了主流程,可要求补 eval。档位判断错了并不致命,但默认要就低不就高,避免用重门槛挤压实验速度。

---

**Pitch(所有档必备,短,结构化)**
- 要解决的问题
- 面向谁
- 为什么这一轮值得试
- 这轮投入上限
- 主要风险点
- 这轮明确不做什么
- 最小成功信号
- 时间盒

**Spec(Standard / Heavy 必备,薄,可机读)**
- 核心输入输出断言:给定 X,必须产出具备 Y 属性的 Z
- 1 到 3 条非目标 / 边界条件
- 如涉及用户可见面,附最小界面草图或 ASCII 流

**Eval(Heavy 必备,Standard 可选)**
- 每条至少包括:输入、期望行为、评判维度
- 至少一条 forbidden action(什么情况下必须不做)
- 这组 eval 在原型池期间持续运行,用来判断每个变体是否在退步

**为什么需要入口检查**
- 过滤掉还没想清楚的想法
- 让拍板人能快速比较多个候选方向
- 让原型池不会被无限扩张
- Heavy 档:让 L1 自动筛(§7.4)有客观判断依据,而不是停在形式层

### 7.3 原型池
- 对值得探索的方向,不直接收敛到单一方案。
- 建立一个轻量"原型池",允许同时跑多个版本。
- 原型池只接收已经过 §7.2 入口的方向。
- 原型生成阶段遵循 §5.4 "Read 并行",可以派 N 路 agent 独立生成变体。
- 原型不再重复回答 pitch 已经回答过的问题,而是重点验证:
  - 用户第一眼会看到什么
  - 首次使用路径是否走得通
  - 用户可见界面截图或 demo 是什么
  - pitch / spec 里没预见到的新问题是什么
  - 技术上是否有阻断风险
  - 按 §7.2 的 eval(Heavy 档必有、其他档如有)评分,当前达到 / 未达到什么程度

### 7.4 分层判断与拍板
对原型不做无限并行,判断也不集中压给同一个人。判断分三层:

**L1 自动筛(Harness 承担)**
- 对 Heavy 档:用 §7.2 提交的 eval 跑每个原型,得到客观分数
- 对 Standard / Light 档:跳过 eval 自动分,只做形式和重复性检查
- Agent + 规则检查 pitch / spec 完整性、是否与已有方向明显重复、是否达到最基本的进入标准
- 这一层要求 Harness owner 把规则固化进自动化脚本,而不是靠人肉 review

**L2 角色筛(各 owner 在自己领域)**
- 产品 / 设计 / 研发各自在自己的领域做快速判断
- 只回答:在我这一侧,这个方向是否成立
- 这一层是并行 Read,不是串行会议

**L3 方向拍板(单线程 Write)**
- 拍板人只在通过前两层过滤的候选里做选择
- 只处理少量值得下注的方向,而不是面对全部原始原型
- 遵循 §5.4:这一层必须单人单线程 + 完整上下文

每轮筛选至少回答:
- 哪个版本更容易打动用户
- 哪个版本更容易试跑
- 哪个版本能更快形成闭环
- 哪个版本的风险最可控

必须明确当前拍板人和拍板依据。每轮筛选结束后,只保留少量继续推进的方向,其余方向直接停止,不进入"先放着以后再说"的默认状态。

### 7.5 止损与重塑
- 进入本轮的方向,默认都有时间盒,不无限延长。
- 判断输入来自 §7.2 中提前定义好的 `最小成功信号 + eval 门槛 + 时间盒`。
- 在约定时间内,如果没有达到预设结果,只有三种处理方式:
  - 继续:说明它值得再投
  - 重塑:说明问题还对,但方案要重做
  - 停止:说明这轮先不继续
- 不允许"方向不清楚、结果也一般,但默认继续拖着做"。
- 失败原型也要留下最小结论:
  - 为什么被砍
  - 哪个判断原来错了
  - 哪些东西可以复用 —— 具体指哪些可以进入 skill 库、哪些 eval case 应该加入回归集

### 7.6 设计收口回路
- 对进入候选的 1 到 2 个方向,不要直接跳到全量工程化。
- 先在真实界面上做一轮高频设计微迭代,由设计 + agent + 研发共同完成。
- 这一轮遵循 §5.4 Write 串行:设计收口是单线程的,不再并发 N 个原型。
- 重点收口:
  - 信息层级
  - 页面密度
  - 关键文案
  - 首屏结构
  - 信任元素
  - onboarding 提示
- 如果某个决策需要连续试参数,直接让 agent 生成一次性调节工具,例如滑块、切换器、对照面板;确认结果后再固化进实现。
- 这一步输出的不只是"页面更好看了",还包括:
  - 关键页面对比截图
  - 设计决策记录
  - 可复用的规则和参数(进入 Process B)
- 如果设计收口发现不是局部细节问题,而是方案方向本身不成立,应回到 §7.3 原型池重选或重做,而不是只在当前方案上硬修。

### 7.7 研发与交付
- 被选中的方向进入短周期研发推进。
- 研发阶段不只关注功能实现,还要同步准备:
  - onboarding
  - demo / 使用说明
  - 验收路径
  - 多角色交接方式
- Agent 可以承担第一版实现、初步 review 和辅助测试。
- Generator / Evaluator 分离:让一个 agent 写实现、另一个 agent(带怀疑人格 prompt)评测,而不是同一个 agent 自评。Heavy 档方向强制做分离;Standard / Light 视情况而定(见 §8.1)。
- 正式交付前必须经过对应 owner 确认。

### 7.8 上线与试跑
- 优先在真实项目中试跑。
- 观察的不是"功能是否存在",而是:
  - 能不能顺利进入使用
  - 结果是否稳定
  - 协作是否顺
  - 用户是否愿意继续用
- 每次试跑至少要回答三件事:
  - 用户是不是真的会用
  - 这个版本是不是比其他候选更好
  - 这轮有哪些经验值得沉淀成规则或 skill(交给 Process B)
- 如果试跑发现错的不是当前版本,而是最初的问题定义,就回到 §7.1 输入与分流,重新定义问题,不在当前版本上继续叠补丁。

### 7.9 回收与沉淀
每轮试跑结束后,必须同时产出:

**产品侧**
- 产品结论
- 止损 / 重塑 / 停止决定
- 流程修正

**Harness 侧(§5.11 强制要求)**
- 至少一条 harness 改动:skill / eval / 工具集 / context 规则 / playbook 更新
- 改动必须交给 Harness owner 集成,不能只留在项目目录里

沉淀不是结束动作。新沉淀出来的 skill、模板和规则,必须反过来作用到下一轮的 §7.2 入口模板和 agent 执行上下文。

### 7.10 显式回路
- 设计回路:
  - §7.6 设计收口发现方案方向不对 → 回到 §7.3 原型池
- 验证回路:
  - §7.8 试跑发现问题定义错了 → 回到 §7.1 输入与分流
- 能力回路:
  - §7.9 沉淀出的 skill / SOP / eval / template → Process B 集成 → 直接升级下一轮 §7.2 入口和 agent 行为

## 8. Process B:Harness 流程

本节在 v0.5 完全没有。它把"agent 平台建设"从各 graph 工程师的顺手活,升级为一条正式流程。

### 8.1 Harness 的四大职能

Harness 是包裹 agent 的运行环境,做且只做四件事:

**Constrain(约束)**
- agent 能做什么的边界:工具集、权限、沙箱、速率限制
- 行业共识(一手材料):Anthropic 的 agent 工程材料反复强调要复查工具清单是否都必要 —— 工具越多,agent 选择时越容易分心。定期审计、砍掉冗余工具是值得坚持的方向。
- 聚星当前经验规则:具体砍多少按各 graph 实测结果定,不存在一刀切的比例。

**Inform(告知)**
- agent 应该做什么的上下文:CLAUDE.md / AGENTS.md、skill 库、rule 文件、示例
- 行业共识(一手材料):OpenAI 公开的 AGENTS.md 约 100 行,可作为参考起点,而非硬上限;不要复述模型训练时已知的内容。
- 聚星当前经验规则:按实测影响为准,超出参考起点但确实被 agent 用到的内容可以保留。

**Verify(验证)**
- 做得对不对:eval harness、hook、evaluator agent
- 行业共识(一手材料):Anthropic 在长会话 harness 设计里发现,同一 agent 对自身输出存在宽容偏差。对关键交付前的验证,值得让 evaluator 使用与 generator 不同的人格或角度(比如怀疑倾向的 prompt)。
- 聚星当前经验规则:Heavy 档方向强制做 generator / evaluator 分离;Standard / Light 视情况而定。

**Correct(纠正)**
- 错了怎么回:auto-fix、escalation、playbook 更新、context 回滚

### 8.2 Context Engineering 四桶

每个 graph、每个 skill 在 harness 层面必须明确回答四件事:

| 桶 | 回答的问题 |
|---|---|
| **Write** | 这次 session 产生的信息,什么要写到窗外(scratchpad / session memory / project memory / org memory)? |
| **Select** | 开始时,从哪里、按什么规则把上下文拉进来? |
| **Compress** | 什么时候压缩?压到多少 token?保留什么、丢弃什么? |
| **Isolate** | 哪些子任务用独立 agent 隔离上下文?主线程保留什么? |

聚星当前经验规则:聚星现有 14+ 个 graph,Harness owner 值得优先审计的事情之一是——这些问题在各 graph 是否已经被回答、回答是否一致。各 graph 用不同策略本身不是问题,但"没人回答过"是风险。

### 8.3 Skill 作为 Harness Component

v0.5 的 skill 元数据已经有:触发条件、预期输出、验证标准、owner、last_verified、last_used。

v0.6 在此基础上追加三项,让 skill 从"文档"升级为"可执行 harness 组件":

- **Constrain**:调用此 skill 时收紧哪些工具 / 权限
- **Context budget**:进来时最多消耗多少 token,结束时最多留多少
- **Postcondition + Forbidden action**:成功后世界必须是什么样;绝不能做的是什么

没有后两者的 skill,只能被人读、不能被 agent 安全调用。

超过多个 cycle 没被引用的 skill,进入待 review / 归档状态。

### 8.4 Eval Harness

- 每个 Heavy 档方向提交的 3-5 条 golden eval cases,进入全局 eval 回归集;Standard 档如果有自带 eval 也可进入。
- 回归集在 harness 改动和 graph 改动时自动跑;具体触发频率(每 PR / 每天 / 每周)由 Harness owner 根据"跑一次的成本 × 对产出的保护收益"决定,重点是有明确规则,不靠记性。
- 聚星当前经验规则:回归跌破基线时,要有明确的处置路径 —— 阻塞合并、或 owner 承认并留下记录 —— 不允许静默接受回退。
- Harness owner 负责维护回归集质量,定期淘汰过时或重复的 case,防止过拟合。
- 聚星当前经验规则:如果 eval 回归跑一次的成本高到会被日常跳过,就是危险信号 —— 要拆分、抽样或优化,而不是接受现状。具体时间阈值按实际情况定。

### 8.5 Taste → Eval 编码管线

品味和评测不是两件事,是同一件事的两个阶段:

1. 人在 Process A 中做了品味判断(这个设计更好 / 这个输出更自然)
2. 说出 why(taste articulation)—— 哪怕只是一句话
3. 把 why 写成 rubric(1-5 分 + 理由维度)
4. 把 rubric 固化为自动 eval 或 evaluator prompt

这条管线是 Process B 的主要产出路径之一。没有它,品味就永远停在某个人脑子里,无法 scale。

### 8.6 Harness 流程的节奏

- 每日:eval 回归自动跑、harness 监控报告
- 每周:Harness owner 和各 Process A owner 对齐一次,梳理本周 harness 改动和下周 backlog
- 每月:跨 graph 审计一次,识别重复模式并合并

### 8.7 Harness 反模式

以下几件事是 harness 质量退化的典型信号,出现时必须立即处理:

- 某个 graph 的 CLAUDE.md 在膨胀(超过 60-80 行)
- 工具清单在膨胀,但很少有 agent 真正用全
- skill 库在增长,但最近 30 天没有 skill 被 agent 实际调用
- eval 回归跑一次需要超过 30 分钟,导致被跳过
- 多个 graph 在独立实现同一类能力(检索、摘要、审核)
- evaluator 和 generator 用的是同一段 prompt(自我宽容陷阱)

## 9. 正式产物

### 9.1 Process A 每轮交付物
这轮项目必须交付的内容,包括:

- 需求框架 / 问题定义
- 入口检查产物(Pitch 所有档必备;Spec 在 Standard / Heavy;Eval 在 Heavy;见 §7.2)
- 原型说明与变体对比
- 方案筛选记录(含三层判断的理由)
- 关键页面截图 / 设计对比
- 设计决策记录 / 参数说明
- 使用说明 / onboarding 材料
- 验收标准
- 试跑反馈记录
- 关键决策记录
- 止损 / 重塑 / 停止结论
- 向 Process B 提交的 harness 改动清单(§5.11 强制)

其中,关键决策记录不追求大而全,但每轮至少记录 1 到 3 个关键分叉点:

- 当时有哪些可选方案
- 最后选了哪个,为什么
- 事后验证结果如何

### 9.2 Process B 可复用资产

只有满足下面条件之一的内容,才算真正的可复用资产:

- 能被 agent 直接加载进执行上下文(skill / rule / playbook)
- 能被人直接作为 checklist / template / 标准使用
- 能在下一轮 Process A 中被明确引用或自动触发

具体形式:

- Skill(带 Constrain / Context budget / Postcondition / Forbidden action,见 §8.3)
- Playbook(触发 / 步骤 / postcondition / 纠错指引 / 禁止事项)
- Eval cases(golden set + 回归集)
- Design rules / prompt cards / UI skills
- 工具白名单与工具审计记录
- Context 架构文档(四桶策略,见 §8.2)
- 流程问题清单与修正记录
- 之后还能继续引用的模板和标准

## 10. 当前最需要补齐的机制

### 10.1 原型池机制(Process A)
- 一个方向如何通过 §7.2 入口
- 同时最多并行几个版本
- 原型何时进入试跑
- 哪些方向需要设计 owner 提前介入收口

### 10.2 筛选机制(Process A)
- 谁拍板
- 用什么标准筛选
- 什么情况下继续探索
- 什么情况下进入收敛
- 设计质量和体验判断在拍板里占多大权重
- 哪些结果触发继续 / 重塑 / 停止
- L1 自动筛的规则和脚本归谁维护(Harness owner)

### 10.3 Harness 组件机制(Process B)
- 哪类经验必须沉淀成 skill / playbook
- 沉淀到哪里
- 谁维护
- 如何在项目中被 agent 自动加载
- skill 不是放在旁边的文档,而是要嵌进实际流程
- 每个 skill 至少要有:
  - 触发条件
  - 预期输出
  - 验证标准
  - owner
  - last_verified
  - last_used
  - Constrain
  - Context budget
  - Postcondition + Forbidden action
- 超过多个 cycle 没被引用的 skill,进入待 review / 归档状态

### 10.4 Eval 机制(Process B)
- 每个方向在 pitch 阶段就要提交 golden cases
- 全局 eval 回归集如何维护、淘汰、防止过拟合
- eval 失败如何阻塞合并
- taste → rubric → eval 的编码周期

### 10.5 Context / 工具管理机制(Process B)
- 每个 graph 的 write / select / compress / isolate 策略是否写下来
- 工具集是否定期审计、是否存在可砍的冗余
- 哪些能力在 graph 之间重复实现,能否合并

### 10.6 设计资产机制
- 哪些设计判断必须沉淀成参数、规则、组件约束或 checklist
- 哪些场景允许先生成一次性调节工具,再沉淀最终结论
- 设计规则放在哪里,谁维护,如何在项目中复用

### 10.7 信任与 onboarding 机制
- 新能力上线后,如何让人知道它能做什么
- 如何减少第一次使用门槛
- 如何让结果更可见、更可控

### 10.8 验证与复盘机制
- 每次试跑至少追哪几个最小指标
- 如何判断某个版本比另一个版本更好
- 哪些失败结论必须记录,避免反复踩坑

默认的最小指标集可以从下面四类里选,不要求每次全追,但至少要选:

- 结果信号:
  - 用户能否在无引导下完成核心操作
  - 用户是否更愿意继续使用某个版本
- 过程信号:
  - 试跑期间是否出现明显异常或阻断 bug
  - 交接给其他角色时是否需要额外解释
- Eval 信号:
  - golden cases 的分数变化
  - 回归集是否有退步
- Harness 信号:
  - 这一轮让 harness 变强了什么(§5.11)

## 11. 试跑方式

- 用实际项目采用这套新流程。
- 草稿过程中不追求一次定稿,而是边用边改。
- 近两周保持高频沟通,优先收集当前正在发生的问题。
- 试跑项目最好包含至少一个强用户可见的体验面,不要只拿纯内部逻辑链路试。
- 每次试跑后,不只问"产品哪里有问题",也问:
  - 流程哪里卡了
  - 哪个角色衔接不顺
  - 哪些经验值得沉淀成 skill
  - harness 哪里被发现是瓶颈
  - 哪个方向应该继续,哪个方向应该停

## 12. 近期落地建议

近两周优先做下面几件事:

1. 先选一个真实项目作为流程试跑样本。
2. 指定一位 Harness owner(可以是现有研发兼任,但要明确时间配额不少于 30%)。
3. 对该项目至少允许 2 个轻量方案并行比较,并让设计参与第一轮收口。
4. 对该项目的 pitch 强制附带 spec + 3 条 golden eval cases,作为 L1 自动筛的试跑素材。
5. 明确这轮试跑的拍板人、筛选标准和验收标准。
6. 每轮沟通后,把有效做法补成文本化产物,包括设计规则和体验判断 —— 由 Harness owner 统一归档。
7. 每周回看一次:
   - 哪些产品结论成立
   - 哪些流程问题重复出现
   - 哪些经验可以正式沉淀成 `skills / SOP / playbook`
   - harness 本周比上周好了多少

## 13. 短期目标

- 两周左右产出一版可复用的流程框架(v0.6 → v0.7)。
- 至少通过一个实际项目完成 Process A 试跑。
- 至少沉淀出一份流程类或设计类 `playbook / skill`,并被下一轮 pitch 明确引用或加载。
- Harness owner 交付一次跨 graph 审计,识别出至少 3 处可以合并的重复能力或可以砍掉的冗余工具。
- 把试跑中暴露的问题补回流程文档,使它从"讨论稿"进化为"可执行稿"。
