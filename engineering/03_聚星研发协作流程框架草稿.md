# 聚星研发协作流程框架草稿

> 状态：v0.3 草稿
> 用途：把聚星的研发协作从“线性推进流程”升级为“Agent 时代的产品研发操作系统”
> 相关范围：聚星跨角色协作流程优化

---

## 1. 北极星目标

这份流程的目标，不只是把角色串起来、减少交接损耗，而是建立一套能够持续：

- 低成本试错
- 并行跑原型
- 快速筛选方向
- 用真实项目验证
- 把有效做法沉淀成 `skills / SOP / playbook`

的研发操作系统。

换句话说，我们最终要的不是“更顺的传统流程”，而是：

> 让聚星能像一个持续生成原型、筛选方案、沉淀经验、快速交付的 Agent 研发组织一样运转。

## 2. 为什么要这样改

当前最核心的变化不是“AI 能帮忙写代码”，而是：

- 执行成本已经显著下降，很多想法不需要先长时间讨论才能验证。
- 真正的瓶颈越来越少是“做不出来”，越来越多是：
  - 怎么选方向
  - 怎么做取舍
  - 哪个方案体验更好
  - 怎么让用户真的敢用、会用、持续用
- 团队经验如果不沉淀成可复用的文本化能力包，后面只能反复靠人肉同步。

所以这套流程必须兼顾两件事：

- 做产品
- 做方法沉淀

## 3. 核心原则

### 3.1 原型优先，不是讨论优先
- 一个方向不要默认只做一个方案。
- 能低成本验证的，就尽量先出原型再讨论。
- 如果一个问题存在明显分歧，优先并行做 2 到 5 个轻量版本，而不是长时间口头争论。

### 3.2 真实项目优先，不做空转流程
- 流程不是独立项目。
- 流程必须在真实项目中试跑，通过真实任务暴露问题。
- 试跑中发现的问题，反过来修正流程本身。

### 3.3 短周期判断，不做长周期空规划
- 关注下周、下下周怎么推进。
- 允许大方向存在，但不依赖长周期刚性 roadmap 才能行动。
- 任何跨月的大计划，都要拆成能在短周期内验证的具体动作。

### 3.4 Skills 是正式产物，不是附属品
- 有效的方法、提示词、检查清单、角色 SOP，要沉淀成可复用文本。
- `skills / SOP / playbook` 不是边角料，而是研发系统的一部分。
- 每次试跑，不只是交付产品，也要看是否产生了新的流程资产。

### 3.5 体验和信任是流程目标的一部分
- 不只关注“功能能不能做出来”，还要关注：
  - 用户敢不敢用
  - 第一次上手顺不顺
  - 多角色协作时是否容易理解和接管
- onboarding、可见性、可控性、交付结果的稳定性，都属于流程成功标准。

### 3.6 人类判断依然是瓶颈
- 原型可以并行做很多，但最终还是要有人做选择。
- 流程里必须明确：
  - 谁拍板
  - 按什么标准选
  - 什么情况下继续并行
  - 什么情况下停止探索、进入收敛

### 3.7 设计要前置进原型池，而不是后置做美化
- 设计不是等 PRD 和代码都差不多了，再出来“补一层稿”。
- 当执行成本足够低时，设计应该直接参与原型生成、方向筛选和最终收口。
- 设计的核心价值，不只是画页面，而是判断体验是否成立：
  - 信息层次是否清楚
  - 首次上手是否顺
  - 信任感是否足够
  - 哪些细节决定用户愿不愿意继续用

### 3.8 设计判断要沉淀成规则，而不是只靠感觉
- 高质量设计判断里，有很大一部分其实可以被说清楚、记下来、复用。
- 能稳定复用的内容，要尽量沉淀成：
  - 参数规则
  - 组件约束
  - review checklist
  - prompt cards
  - `UI / design skills`
- 对某些具体决策，允许临时生成一次性调节工具，用完就扔，但结论要被固化。

## 4. 角色模型

当前流程先按以下角色协作，后续根据实际项目继续细化：

- 产品 / 需求 owner
- 设计
- 前后端研发
- 测试 / 验收
- 上线 / 运营协同
- 业务反馈输入方
- Agent / automation 执行层

其中，Agent 不是“替代某个人”的角色，而是嵌在多个角色工作流中的执行层。

`设计` 在这套流程里的职责，不应该再只理解成“静态稿输出”，而应该明确包括：

- 参与原型生成与第一轮筛选
- 对关键体验面做最终收口
- 为 onboarding / trust / 信息架构负责
- 把有效的设计判断沉淀成规则、清单和 skills
- 在需要时，要求 Agent / 研发快速生成临时调节工具来辅助决策

## 5. 流程主循环

### 5.1 输入与分流
- 明确输入来源：
  - 用户反馈
  - 销售 / 运营反馈
  - 内部策略调整
  - 技术问题 / 线上问题
  - 流程问题
- 判断输入属于哪一类：
  - 新需求
  - 优化项
  - 问题修复
  - 流程类事项
  - 需要原型验证的方向判断

### 5.2 原型池
- 对值得探索的方向，不直接收敛到单一方案。
- 建立一个轻量“原型池”，允许同时跑多个版本。
- 每个原型至少要回答：
  - 解决什么问题
  - 面向谁
  - 为什么值得试
  - 需要哪些资源
  - 用户第一眼会看到什么
  - 首次使用路径是否清楚
  - 用什么标准判断继续 / 淘汰

### 5.3 筛选与拍板
- 对原型不做无限并行。
- 每轮筛选至少回答：
  - 哪个版本更容易打动用户
  - 哪个版本更容易试跑
  - 哪个版本能更快形成闭环
  - 哪个版本的风险最可控
- 必须明确当前拍板人和拍板依据。

### 5.4 设计收口回路
- 对进入候选的 1 到 2 个方向，不要直接跳到全量工程化。
- 先在真实界面上做一轮高频设计微迭代，由设计 + Agent + 研发共同完成。
- 这一轮重点收口：
  - 信息层级
  - 页面密度
  - 关键文案
  - 首屏结构
  - 信任元素
  - onboarding 提示
- 如果某个决策需要连续试参数，直接让 Agent 生成一次性调节工具，例如滑块、切换器、对照面板；确认结果后再固化进实现。
- 这一步输出的不只是“页面更好看了”，还包括：
  - 关键页面对比截图
  - 设计决策记录
  - 可复用的规则和参数

### 5.5 研发与交付
- 被选中的方向进入短周期研发推进。
- 研发阶段不只关注功能实现，还要同步准备：
  - onboarding
  - demo / 使用说明
  - 验收路径
  - 多角色交接方式

### 5.6 上线与试跑
- 优先在真实项目中试跑。
- 观察的不是“功能是否存在”，而是：
  - 能不能顺利进入使用
  - 结果是否稳定
  - 协作是否顺
  - 用户是否愿意继续用

### 5.7 回收与沉淀
- 每轮试跑结束后，至少产出三类沉淀：
  - 产品结论
  - 流程修正
  - 新增 `skills / SOP / playbook`

## 6. 正式产物

这套流程里，正式产物不只包括 PRD 和代码，还包括：

- 需求框架 / 问题定义
- 原型说明
- 方案筛选记录
- 关键页面截图 / 设计对比
- 设计决策记录 / 参数说明
- 使用说明 / onboarding 材料
- 验收标准
- 试跑反馈记录
- 设计 rules / prompt cards / UI skills
- `skills / SOP / playbook`
- 流程问题清单与修正记录

## 7. 当前最需要补齐的机制

### 7.1 原型池机制
- 一个方向如何登记为候选原型
- 同时最多并行几个版本
- 原型何时进入试跑
- 哪些方向需要设计 owner 提前介入收口

### 7.2 筛选机制
- 谁拍板
- 用什么标准筛选
- 什么情况下继续探索
- 什么情况下进入收敛
- 设计质量和体验判断在拍板里占多大权重

### 7.3 Skills 沉淀机制
- 哪类经验必须沉淀
- 沉淀到哪里
- 谁维护
- 如何在项目中复用

### 7.4 设计资产机制
- 哪些设计判断必须沉淀成参数、规则、组件约束或 checklist
- 哪些场景允许先生成一次性调节工具，再沉淀最终结论
- 设计规则放在哪里，谁维护，如何在项目中复用

### 7.5 信任与 onboarding 机制
- 新能力上线后，如何让人知道它能做什么
- 如何减少第一次使用门槛
- 如何让结果更可见、更可控

## 8. 试跑方式

- 用实际项目采用这套新流程。
- 草稿过程中不追求一次定稿，而是边用边改。
- 近两周保持高频沟通，优先收集当前正在发生的问题。
- 试跑项目最好包含至少一个强用户可见的体验面，不要只拿纯内部逻辑链路试。
- 每次试跑后，不只问“产品哪里有问题”，也问：
  - 流程哪里卡了
  - 哪个角色衔接不顺
  - 哪些经验值得沉淀成 skill

## 9. 近期落地建议

近两周优先做下面几件事：

1. 先选一个真实项目作为流程试跑样本。
2. 对该项目至少允许 2 个轻量方案并行比较，并让设计参与第一轮收口。
3. 明确这轮试跑的拍板人、筛选标准和验收标准。
4. 每轮沟通后，把有效做法补成文本化产物，包括设计规则和体验判断。
5. 每周回看一次：
   - 哪些产品结论成立
   - 哪些流程问题重复出现
   - 哪些经验可以正式沉淀成 `skills / SOP / playbook`

## 10. 短期目标

- 两周左右产出一版可复用的流程框架。
- 至少通过一个实际项目完成试跑。
- 至少沉淀出一份流程类或设计类 `playbook / skill`。
- 把试跑中暴露的问题补回流程文档，使它从“讨论稿”进化为“可执行稿”。
