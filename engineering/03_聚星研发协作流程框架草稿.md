# 聚星研发协作流程框架草稿

> 用途:把聚星的研发协作从"线性推进流程"升级为"Agent 时代的产品研发操作系统"
> 相关范围:聚星全部活跃产品线(老平台 v7.3.x / 插件 v7.7.x / Skills v7.8.x / Agent 主线 v7.9.x / TTShop v7.10.x / 网红经纪人 / 邮件通道等公共能力),按产品线节奏分档切入(见 §12.3)
>
> 材料分层约定:文档中的具体规则分两类标注 ——
> - **行业共识 / 一手材料支撑**:有明确公开源可对照(2026 Q1-Q2 前沿)
> - **聚星当前经验规则 / 试运行假设**:基于本地经验或待验证的判断,优先级可随试跑调整
>
> 规则分层约定:所有**可执行规则**按"落地时效"标三档,防止同一条规则在不同章节被读成不同口径 ——
> - **【目标态】**:前沿已实证、聚星应该走到的终态(如:0 pre-merge human review;原型海 5-10 并行起点)
> - **【当前默认】**:聚星研发组织当前即可按此执行的规则(无需额外配置或能力建设就成立)
> - **【试运行例外】**:现阶段先从某个场景试跑的起步下限,跑顺后向【当前默认】或【目标态】收敛,不作为长期执行标准
>
> 读者判断某条规则当前是否适用,**看它被 tag 成哪档,不看它出现在哪个章节**。如果一条规则同时以三个 tag 出现,是它在不同阶段的"梯度":试运行例外 → 当前默认 → 目标态。

## 1. 目标

建立一套能让**聚星研发组织的产品化节奏完全发挥模型节奏**的产品研发协作方案。

展开这句话:

- 模型能力当前是**溢出**的 —— Anthropic 内部判断"模型能力的增长速度,已经开始超过我们把它变成产品的能力"(Felix Rieseberg, 2026-04),聚星没有理由面对的是不同的情况
- 所以组织真正的瓶颈,不是"模型不够强",也不是"代码写得不够快",而是**研发组织的产品化链路,能不能把模型能力以匹配的节奏吸收、转化、交付出去**
- 这套流程的全部设计,都围绕这一个瓶颈展开

具体到聚星,这套流程要能够持续:

- 低成本试错(执行 ≈ 0 成本,不再是前置假设的稀缺资源)
- 大规模并行原型(前沿默认做法)
- 快速对齐与筛选(alignment 是真正的稀缺瓶颈)
- 用真实项目和真实用户验证
- 把有效做法沉淀成可被 agent 直接执行的能力包
- **把"让下一轮更便宜"当成和"完成当前轮"同等重要的目标**

## 2. 协作流程为什么会演化

### 2.1 核心原理

协作流程不是抽象信仰,而是组织围绕当前主瓶颈分配注意力、责任和节奏的方式。

- 软件开发方法论,本质上是在优化当时最贵、最慢、最危险的那个环节。
- 当某个环节的成本塌陷,瓶颈就会转移到下一个环节。
- 瓶颈一旦转移,协作流程就必须跟着重组,否则旧流程会开始优化已经不再重要的问题。

决定协作方式的,不只是"编码成本",而是整条链路的成本结构:

- 变更成本
- 集成成本
- 反馈等待时间
- 协调成本
- 判断成本
- 信任成本
- **跨 agent / 跨会话的上下文一致性成本**(Agent 时代新出现的、不能用人脑吸收的成本类别)

底层逻辑始终是同一个:

1. 某个环节变便宜了
2. 新瓶颈暴露出来了
3. 流程开始围绕新瓶颈重组

### 2.2 演化历史

#### 第一阶段:阶段式 / 瀑布
当写代码、改代码、部署和返工都很贵时,最自然的做法就是把工作切成顺序阶段,尽量在开工前把事情想清楚。这一代流程优化的核心问题是:**如何减少高成本返工**。Royce 1970 的原文其实已经提醒纯顺序模式有风险、需要补反馈,后来被广泛简化成"纯线性瀑布"才成了反面教材。

#### 第二阶段:敏捷
当编码、测试、版本控制和迭代本身开始变便宜,真正昂贵的就不再只是"写代码",而是"写了很久才发现方向错了"。敏捷解决的是:**如何在高不确定性环境里缩短反馈回路**。它不是反规划、反文档,而是反对用长周期和重交接去掩盖判断的不确定性。

#### 第三阶段:DevOps / 持续交付 / 精益
开发团队内部迭代速度提升后,新瓶颈变成集成、部署、跨团队 handoff,以及"写完之后多久才能看到真实反馈"。DevOps、CI/CD、Lean、Shape Up 看似很碎,但底层都在解决同一类问题:**加速反馈,降低交付摩擦**。

#### 第四阶段:Agent 时代(我们现在所处)

上一阶段解决的是"加速反馈"。这一阶段解决的是另一类问题:**当执行本身的边际成本接近零,组织如何分配人的稀缺判断**。

行业前沿在 2026 Q1-Q2 已经实践到的状态(三家不同公司、不同方向的独立验证):

- **Anthropic Cowork**:内部约 100 个产品原型并行;不再"写 spec → 排期 → 执行",而是"全部候选并行做、焦点小组测、发布赢家";路线图 ≤ 1 个月(Cowork 工程负责人 Felix Rieseberg 访谈,2026 年 4 月)
- **OpenAI Harness Engineering 实验**:5 个月、3 → 7 个工程师、**0 行手写代码、0 人工 code review**,产出约 100 万行代码、1500 PR;Codex agent 之间互相 review(OpenAI 公开,2026 年 2 月)
- **Cursor 3**:传统 IDE 降级为 fallback,**Agent Console 成为主界面**,云端 agent 完全移出编辑器;Agent 用户已超过 Tab 自动补全用户 2 倍(2026 年 4 月发布)

这个阶段昂贵的环节是:

- 目标定义
- 判断与品味
- 对齐(alignment)
- 信任建设
- harness 质量

不是"做不出来",是"做出来太多,不知道选哪个";不是"代码写得慢",是"agent 跑偏了没人发现"。

协作流程的重点彻底转移:

- **从"组织生产"转向"组织判断、对齐、沉淀"**
- **从"加人扩张"转向"加 agent 扩张"**(执行边际成本 ≈ 0)
- **从"工程师写代码"转向"工程师建 harness"**(OpenAI 已经把 Harness Engineer 正名为新工种)
- **UX 服务对象反转**:界面不再是"让计算机更好工作",而是"让人能理解、控制、信任 agent"。Felix 原话:"过去我们设计界面是为了让计算机更好工作,人只是输入信息的角色;但现在反过来了"

同时出现一类以前不存在的新成本:**跨 agent、跨会话的上下文一致性成本**。这在过去用人脑就能吸收,现在必须工程化解决 —— 这是 harness engineering 作为独立学科的核心问题。

#### 第五阶段:可能的下一步
如果未来一部分判断也被自动化了,瓶颈会继续上移。那时更稀缺的可能是:品味、信任、价值取舍、对用户和业务的真实理解。这是为什么 §5 把"人聚焦最高杠杆位置"作为核心原则之一。

### 2.3 这个演化对聚星意味着什么

聚星当前的真实画像:

- **成熟的海外网红营销 SaaS**,多产品线并行推进
  - 老平台优化线(v7.3.x,工具型 SaaS 节奏)
  - Skills 能力层(v7.8.x)
  - 浏览器插件(v7.7.x)
  - TTShop 建联(v7.10.x)
  - **网红营销 Agent 主线(v7.9.x,Q2 P0)**
  - 网红经纪人(P2 调研)
  - 海外市场拓展优化包(P1)
  - 邮件通道优化(跨产品公共能力)
- 研发组织同时承担这些线上的交付和演进,节奏、形态、用户面差异很大

在这个结构里,**agent 主线是第一条直接撞到"产品化滞后于模型能力"这堵墙的产品线** —— 因为它的价值交付严重依赖 agent 运行时、skill、context engineering、evaluator 的工程化程度,而这些正是整个聚星研发组织还没有稳定肌肉的环节。

聚星当前真正面对的不是"开发速度还不够快"这个老问题,而是一个新的结构性问题:

- 模型能力本身在 2026 Q1-Q2 持续跃迁(Mythos / Cowork 级新能力),且能力增长速度已经超过行业的产品化能力
- 生成能力显著提速,但**判断、筛选、验证、复用**还没有同比提速
- 行业前沿(Anthropic / OpenAI / Cursor)已经走到"少数 owner + 大量并行 agent + 高质量 harness"的组织形态 —— 聚星不在这个位置,但这是研发操作系统要支持达到的目标状态

如果研发流程继续围绕"如何更高效地产出代码"来组织,会出现三个后果:

- 原型越来越多,但筛不动(alignment 瓶颈)
- 做出来的东西越来越快,但方向错误暴露得不够快(判断瓶颈)
- 每条产品线各自维护低质量上下文和 skill,组织没有沉淀(harness 瓶颈)

所以这套流程要同时做三件以前不做的事:

- **产品流程**:围绕判断、筛选、对齐组织(Process A,见 §4)
- **Harness 流程**:持续编码 agent 的执行环境,让下一轮更便宜(Process B,见 §4)
- **认知对齐**:让研发组织对"当前模型和 agent 能做什么、不能做什么"的直觉,跟上最新的模型能力曲线,而不是锁定在过去某一代模型的直觉上

> **背景风险(Felix Rieseberg 原话,2026-04)**:"许多 AI vertical 会被压缩,通用模型会吸收 narrow use cases"。聚星是**垂直 SaaS**(海外网红营销),对这条判断需要正面回应 —— 产品化滞后不只是"慢了",可能意味着**被通用 agent + skill 组合在 60 秒内复刻走**(如 LinkFoxClaw 应用 OpenClaw pattern 的例子)。这条风险不是本流程要解决的(战略层面的事),但研发组织应该知道自己为什么要在 Q2 把 agent 主线放 P0,以及为什么"产品化节奏完全发挥模型节奏"(§1)是个时间敏感的目标,不是锦上添花。

## 3. 第一性原理:Agent 时代的成本格局

如果不从第一性原理出发,这套流程很容易被误读成两种东西:**一种是"加 AI 提效"**(把 agent 当工具,流程不变),**另一种是"agent 当中心"**(让 agent 做决策,人退到幕后)。两个解读都不对,真正的方向是从成本格局推出的。

### 3.1 三个塌陷的成本

执行成本几乎清零。这不是修辞,是 2026 Q1-Q2 的实测数据。具体塌陷的是这三件事:

1. **执行**:写代码、生成变体、补测试、初步 review、文档草拟。OpenAI Frontier 团队 5 个月产出 **1M LOC、1B tokens/day、零手写代码、零 pre-merge human review**(Ryan Lopopolo,2026-02 公开),是这件事塌陷的极端验证。
2. **上下文整理**:agent 自己读代码、画图、归纳现状;Anthropic 内部把"调研一段陌生代码"的时间从小时压到分钟。
3. **失败回滚**:原型不喜欢就丢,Anthropic 内部 100 原型并行不构成预算约束(Felix 原话:"给我 10 分钟,我给你一个版本")。

### 3.2 三个保持稀缺的资源

但有些东西没有变便宜。**反而因为前三项变便宜,这些更稀缺了**:

1. **目标定义和品味**:从 100 个原型里挑哪个继续,这是人的工作;agent 不能替你判断"哪个体验更打动用户"。Felix 原话:"alignment 依然很难,而且高度依赖人 —— 这正是品味发挥作用的地方。"
2. **对齐(alignment)**:在多 owner、多 agent 并行的状态下,让大家的判断标准对齐 —— 这是组织治理的硬问题,agent 不解决。
3. **信任建设**:让用户(和团队内部)敢用、会用、持续用 agent 输出。这是产品能落地的最后一里。Felix 原话:"我们做的大多数按钮,其实是给人用的,而不是给机器用的。"

### 3.3 这个成本格局推出的协作必然形态

由"三个塌陷 + 三个稀缺",以下五件事不是设计选择,是成本结构的数学结果:

1. **执行先行,讨论后置**:既然执行 ≈ 0 成本,"先讨论再执行"就是浪费时间。前沿默认做法是**原型海** —— 有 10 个想法就跑 10 个,从结果倒推哪个对。
2. **Agent 互审,人审标准**:既然 agent 能做中等任务,"人审 agent 每一步"是把人的稀缺判断浪费在 agent 之间能解决的事情上。OpenAI 100 万行代码 0 人工 review 是这个原理的极端验证 —— 人的工作是定义 reviewer agent 的标准,而不是亲自当 reviewer。
3. **人聚焦最高杠杆位置**:目标定义、品味、对齐、信任建设。其他事情都该考虑能不能交给 agent 或 harness。
4. **Harness 是组织级一级资产**,不是各场景顺手活。组织产能 = `人 × agent 数 × harness 质量`,前两项加 agent 边际成本 ≈ 0,所以 **harness 质量是真正的扩张瓶颈**。
5. **加 agent 优先于加人**:执行边际成本 ≈ 0,但人的判断稀缺。在产能不够时,默认思路应该是"加 agent / 提升 harness",而不是"招更多工程师"。

### 3.4 聚星的特殊处境与迁移路径

聚星不在前沿位置,但有三重特殊压力:

- **多产品线同时存在**,节奏、形态、用户面差异大,不能用"一刀切流程"硬套
- **agent 主线是第一条被产品化滞后直接卡住的产品线**,但研发组织的 agent 工程化肌肉还在建
- **不能靠加人扩张产能** → 必须靠"加 agent + 提升 harness"扩张(§3.3-5)

这套流程是聚星的迁移路径,不是终点状态。具体阶段:

- **短期(本季度)**:按 §3.5 试验场规则选 1-2 条产品线试跑,把"少数 owner + N 个并行 agent"的工作模式在真实项目里跑通一次
- **中期(本半年)**:把"harness 是一级资产"落地(Process B 完整跑起来 + Harness owner 升级触发),把试跑出来的做法外推到其他产品线
- **长期(本年)**:流程在所有活跃产品线按各自档位稳定运行,研发组织的产品化节奏能持续跟上模型节奏

### 3.5 试验场的筛选规则

短期选哪条产品线(或哪个产品线切片)做试验场,**由一条规则决定,不由拍脑袋决定**:

> **挑那种能在 2 周内让多个原型撞到真实用户、并逼产研和另一个角色对齐判断的场景;避开被线上 SLA、跨月 rollout、或单角色闭环锁死的场景。**

这条规则是从 §3.1-3.3 的成本格局直接推出的:

- "多原型" —— 因为执行 ≈ 0,原型海是默认,试验场不能只有一解
- "真实用户" —— 因为信任建设不能在真空里验证(§3.2-3)
- "多角色对齐" —— 因为 alignment 是真正的稀缺瓶颈(§3.2-2),单角色闭环暴露不出来
- "2 周内闭环" —— 因为学习速度跟反馈周期呈反比,跨月 rollout 一个季度只学一次
- "避开 SLA 路径" —— 因为试错代价超过学习价值时,流程本身没法迭代

**规则的使用方式**:每季度 Harness owner 协同研发负责人开一次试验场评审,把当前活跃产品线(含可切片场景)过一遍这条规则,选 1-2 个最匹配的。试验场跑完一轮后,规则本身也要迭代(§5.10)。

## 4. 双流程架构

§3 推出的"产品判断 + harness 演化"两件事,节奏完全不同、产出完全不同、所有者完全不同。**把它们合并成一条时间轴,Harness 永远会被业务优先级压到最低**。所以从第一性原理出发,最小流程拓扑就是两条并行流程。

### 4.1 两条流程并行

| 维度 | Process A:产品流程 | Process B:Harness 流程 |
|---|---|---|
| 周期 | 天到周 | 持续、非项目化 |
| 驱动 | 业务目标、用户反馈、项目立项 | Process A 每一轮暴露的 harness 缺陷 |
| 产出 | 用户可见的功能、产品决策 | agent 的执行环境:上下文、工具、约束、评测、skill、playbook |
| 主要参与者 | 产品 / 设计 / 研发 owner + 各自激活的角色模板组合(§6.0) | Harness owner(兼任或专职,见 §6.2)+ 各场景的研发 |
| 节奏感 | 有明确起止、有时间盒 | 持续演化、每周有改动 |
| 评估指标 | 用户能否在无引导下完成核心操作 / golden cases 分数 | 跨场景重复度、CLAUDE.md 体积、skill 调用频率、回归集运行成本 |

### 4.2 为什么必须分开(第一性原理论证)

- **节奏冲突**:Process A 节奏受业务压力控制,停不下来做基础设施优化。Process B 价值靠复利累积,如果交给项目制团队,会被业务优先级永久压到最低。
- **资产形态不同**:Process A 产物是用户看到的产品;Process B 产物是 agent 加载的组件。混在一起,组件永远只会被当作"项目副产物",不会被当作组织级资产维护。
- **能力维度不同**:Process A 需要业务直觉、品味、用户感;Process B 需要系统性思维、跨场景抽象、context 工程。强行让同一个人同时做两件事,必有一件做不好。

### 4.3 两条流程的接口

- Process A 的每一轮,必须向 Process B 提交至少一条 harness 反馈(见 §5.10 "每轮 +1%")
- Process B 的每次改动,必须明确影响哪些 Process A 入口(哪些场景、哪些角色的工作方式会被改变)
- Harness owner 每周对齐一次 Process A 的各 owner,确认本周 harness 改动是否有盲区

## 5. 核心原则

§3 推出的成本结构,落到工作面上就是下面这些原则。每一条都标注了原理来源,不是设计偏好。

### 5.1 执行先行,讨论后置(原型海)
- 一个方向不要默认只做一个方案。能低成本验证的,就尽量先出原型再讨论。
- 一个问题存在明显分歧时,**优先并行做多个轻量版本**,不要长时间口头争论。
- 但不是任何想法都直接进原型池,先要过一轮入口检查(分层,见 §7.2)。
- **原理**(§3.1 + §3.3-1):执行 ≈ 0 成本,讨论是浪费;前沿默认做法是 100 原型并行。

### 5.2 Read 并行 / Write 串行
- 探索、调研、变体生成 —— 可以 N 路并行,鼓励并行。**并行能力随 agent 数量近似线性扩张,这是聚星扩产能的主路径,不是加人。**
- 方案收口、实现落地、决策下注 —— 单线程 + 完整上下文。
- **违反这条原则是 agent 协作失败最常见的根本原因**:并行的 write agent 彼此看不到对方决策,产出互相冲突。
- **原理**:Walden Yan(Cognition)的核心结论;Anthropic harness 设计也明确这一区分。

### 5.3 Agent 互审为默认,人做后置采样,不做逐 PR 复审

这一节把"agent 互审 + 0 pre-merge human review"拆成三档,避免读者把终局状态当成当前默认状态强推。

#### 规则梯度

"0 pre-merge human review" 的落地拆成**基础形态**和**完整形态**两层,避免读者把"聚星任何研发都应该走 7 角色流水线"当成当前默认:

- **【当前默认·基础形态】**:聚星任何用 Claude Code / Codex 的研发场景,**merge 前零 human review 已经在跑** —— 靠 agent 自带的 self-review + verify tests + subagent delegation(例如 Claude Code 里一个 task 自然拆成 plan → implement → verify 的 subagent),**不需要额外配置**。**这个档位对 Light / Standard / Heavy 三档全部立即生效**(Heavy 档在基础形态之上必须 + golden eval 回归集,见 §8.4),不是未来目标。
- **【目标态·完整形态】**:Heavy 档方向走 **7 角色模板流水线**(§6.0:CEO / Eng Mgr / Staff Engineer / Reviewer / Verify / QA / Release)+ **结构化 rubric**(每个 reviewer 角色都有独立的 rubric / forbidden actions / golden cases)。人只做定 rubric + post-merge sampling。这是 OpenAI Frontier 2026-02 实证的终态 —— 聚星目标是 2026 年下半年到达,**不是当前默认**。
- **【试运行例外】**:把基础形态升级到"结构化 evaluator(独立 rubric / forbidden actions / golden cases)+ 至少 3 个角色模板(Staff Engineer reviewer + Verify + 另一个业务相关 role)",只在当前选定的试验场(§3.5)先跑一个方向。跑顺后把角色模板表从起步 3 个扩展到完整 7 角色,推广到其他 Heavy 产品线 —— 见 §12.2 增强项。对应 §13 短期目标里的"角色模板表起步"节点。

#### Review agent 的默认规则(来自 OpenAI Frontier 实测)

- bias 向 merge(不卡住)
- **只 surface P0 / P1 / P2 级别的问题;抑制 P3 及以下 nice-to-fix 的 nitpick**(P0 = 最严重)
- 反馈以"可接受 / 延后 / 反驳"三选一方式给 authoring agent,迭代到所有 reviewer 满意为止

#### 在 code review 这一段,人的工作只剩两件

(注意作用域:这一条**只管代码审查**,不代替 §5.5 "人聚焦最高杠杆位置"的 4 件事,也不代替 §7.4 L3 方向拍板。)

- **定义 evaluator rubric**(rubric / forbidden actions / golden cases / gotchas)—— 这是 §5.5 品味的落点之一
- **做 post-merge sampling**(抽样识别 systemic issue,不是每个 PR 都看)—— 类似线上客服的抽检机制

#### 非代码交付如何套用本节(桥接规则)

聚星不少 Light / Standard 档工作不是代码交付(例:用户文档、合规材料、账号开户提审、对外演示视频)—— 这些没有"merge / PR review"的字面语义。**沿用同一原则,落地形式不同**:

| 代码交付 | 非代码交付的等价物 |
|---|---|
| Generator agent 出代码 | Generator agent 出文档 / 材料 / 视频脚本 |
| Reviewer agent 跑 rubric | Reviewer agent 对照 checklist + 公司语调 / 合规要求评分 |
| Verify agent 跑测试 | Verify agent 跑可读性 / 完整性 / 链接有效性自检 |
| post-merge sampling | owner 对发出后的产物做抽样 review(看用户实际是否能看懂、能用) |
| eval 回归集 | golden checklist 集合(如:用户文档 5 大必含项、提审材料 7 项合规清单) |

骨原理一致:**人定标准(rubric / checklist)+ agent 互审 + 抽样确认**,不依赖人逐字逐句把关。本节后续提到的"PR / code review / merge"等措辞,在非代码场景下读者按上表自行映射。

#### 原理(§3.3-2)

OpenAI Frontier 团队 5 个月、1M LOC、**0 手写代码、0 pre-merge human review**;12 月 3.5 PRs/人/天 → 1 月 GPT-5.2 后 5-10 PRs/人/天(同团队、零流程改动)。Anthropic 三 agent harness(planner / generator / evaluator)、GStack(CEO / Eng Mgr / Staff Eng / QA / Release)、Boris Cherny 的 verify-app + code-simplifier subagent 都是同一思路的不同落地。

#### 避开的反模式

- "关键节点再由人复审" —— 这是 SaaS 时代的瓶颈结构;人一旦回到逐 PR review,就又把自己放回了稀缺判断的错配位置
- evaluator 和 generator 用同一段 prompt —— 自我宽容陷阱,OpenAI 实测证明必须**人格 / 配置独立**

### 5.4 加 agent 优先于加人
- 在产能不够时,默认思路应该是:"加 agent / 提升 harness 质量",**不是招更多工程师**。
- 加人本身不一定错,但要先回答:这个工作能不能让一个 owner + 更多 agent 做完?如果能,那加人就是错配资源。
- **原理**(§3.3-5):执行边际成本 ≈ 0,而 onboarding 一个新人的成本是周到月级别。
- **极端实证**:Peter Steinberger(奥地利独立开发者)单人 + 一套 OpenClaw skill 库,2025-11 发布到 2026-02 期间达到 **250K GitHub stars**(史上最快开源项目之一);Garry Tan(YC CEO)单人用 GStack(23 角色模板) **60 天产出 600K LOC**。这两个数字在过去的组织架构里至少要 20-50 人团队才能达到。
- **聚星当前经验规则**:增员决策应该先经过 Harness owner 评估"是否是 harness 短板",而不是直接走招聘流程。

### 5.5 人聚焦最高杠杆位置
- 人的稀缺判断只用在四件事上:**目标定义、品味、对齐、信任建设**。
- 其他事情(执行、调研、互审、文档草拟、上下文整理)默认考虑交给 agent 或 harness。
- 流程里必须明确:谁拍板、按什么标准选、什么情况下继续并行、什么情况下进入收敛。
- 判断要分层(见 §7.4),不是所有决策都堆给同一个拍板人。
- 判断要被编码:能说清楚的品味,应该走 `taste → rubric → eval → automation` 这条管线(见 §8.5)。
- **原理**(§3.2 + §3.3-3):人是稀缺资源,把人放在能用 agent 替代的位置就是浪费。

### 5.6 体验和信任是一级目标(界面是给人用的,不是给机器用的)
- 不只关注"功能能不能做出来",还要关注:用户敢不敢用 / 第一次上手顺不顺 / 多角色协作时是否容易理解和接管。
- onboarding、可见性、可控性、交付稳定性,都属于流程成功标准。
- **设计原则锚点**(Felix Rieseberg):"我们做的大多数按钮,其实是给人用的,而不是给机器用的。过去我们设计界面是为了让计算机更好工作,人只是输入信息的角色;但现在反过来了。" —— 因为人现在的工作是理解、控制、信任 agent,而不是输入数据。
- **原理**(§3.2-3):信任建设是 agent 产品落地的最后一里。

### 5.7 短周期判断,不做长周期空规划
- 关注下周、下下周怎么推进。允许大方向存在,但不依赖长周期刚性 roadmap 才能行动。
- 跨月的大计划,都要拆成能在短周期内验证的具体动作。
- 每一轮都要有明确的投入上限和退出条件,**不默认延期**。
- **行业方向锚点**:Anthropic Cowork 整个产品路线图最长 1 个月(Felix 原话:"任何人如果告诉我他知道 AI 一年后会是什么样,我也不会太信服")。

### 5.8 真实项目优先,不做空转流程
- 流程不是独立项目。
- 流程必须在真实项目中试跑,通过真实任务暴露问题。
- 试跑中发现的问题,反过来修正流程本身。

### 5.9 Skills 嵌进流程才算沉淀;设计判断同样适用
- 有效的方法、提示词、检查清单、角色 SOP、设计规则、参数约束,都要沉淀成可复用文本。
- `skills / SOP / playbook` 不是边角料,而是研发系统的一部分。
- **如果一个 skill 只是被写进文档库、不会在实际流程里被 agent 调用、引用或加载,它就不算真正沉淀。**
- 设计判断同样适用:能稳定复用的,要沉淀成参数规则、组件约束、review checklist、prompt cards、UI/design skills,而不是停在某个设计师脑子里。
- 对某些一次性决策,允许临时生成调节工具(滑块、切换器、对照面板),用完即弃,但结论要被固化。

### 5.10 显式回路 + 每一轮必须让 harness 变强 1%
- **设计回路**:设计收口发现方向不对 → 回原型层(§7.6 → §7.3)
- **验证回路**:试跑发现问题定义错了 → 回输入层(§7.8 → §7.1)
- **能力回路**:沉淀 skill / SOP / template → 直接改变下一轮 agent 工作方式

每一轮 Process A 结束,除了交付产品结论,**必须产出至少一项 harness 改动**:
- 一条新工具加入 / 一条旧工具删除
- 一条 CLAUDE.md 或 skill 规则改写
- 一个 eval case 加入回归集
- 一个 context 边界(write / select / compress / isolate)重新划定
- 一条 playbook 的 postcondition 或 forbidden action 澄清

如果一轮结束时 harness 没有任何改动,**视为这一轮没有产生组织复利**。这条原则把"短期交付"和"长期能力"之间的 tradeoff 内化到每一次迭代里。

## 6. 角色模型

**人角色和 agent 角色是两个独立的维度,不是一张表** —— 这是整个 §6 的骨。

### 6.0 两类角色的划分

**A. 人角色(owner 类)**:谁对什么产出负责。这部分沿用聚星现有的组织结构,不因 agent 时代而消失。

- 产品 / 需求 owner
- 设计
- 前后端研发
- 测试 / 验收
- 上线 / 运营协同
- 业务反馈输入方
- **Harness owner**(可兼任或专职,见 §6.2)

**B. Agent 角色模板(role template 类)**:agent 在具体任务里扮演什么人格 / 专长。一个 human owner **同时指挥多个不同角色模板的 agent**(cognitive gearing),不是 1:1。

聚星初始推荐的核心模板表(按实际使用扩展):

| 模板 | 职责 | 典型激活时机 |
|---|---|---|
| **CEO / 方向模板** | 方向 review、是否值得做、问题定义质量 | Pitch 写完、进入原型池之前 |
| **Eng Manager 模板** | 架构、数据流、边界 case、拆分建议 | 原型进入实现之前 |
| **Staff Engineer 模板** | "paranoid code review" 人格,专门找生产问题 | 实现完成、merge 之前 |
| **Designer 模板** | 设计决策、参数判断、UI 对比 | 设计收口阶段(§7.6) |
| **QA 模板** | 测试生成、覆盖率、回归跑 | verify 阶段 |
| **Release 模板** | 打包、灰度、上线 checklist | 交付前闸门 |
| **Doc 模板** | onboarding 文档、使用说明、changelog | 交付并行 |
| **Reviewer 模板** | 对照 rubric 打分、输出 forbidden action 检查 | §5.3 agent 互审 |
| **Verify 模板** | 端到端 happy path 自动跑 | merge 前自动触发 |

**这张 agent 角色模板表本身是 harness 一级资产**(§8.3):
- 跨产品线复用(不是每条产品线各造一套)
- 由 Harness owner 维护
- 每个模板对应一个 skill folder(§8.3)

**概念来源锚点**(2026-03 到 2026-04 前沿实证):
- **GStack**(Garry Tan / YC CEO,2026-03-12):23 个 role-specialized agent,sprint 节奏 Think → Plan → Build → Review → Test → Ship → Reflect;GitHub 48 小时 10K 星;Tan 本人 60 天 600K LOC 产出
- **Boris Cherny**(Claude Code 作者):单人 5 个 Claude Code 并行 + 5-10 个 browser session = **同时 15 个 agent 各司其职**(code-simplifier、verify-app、不同专长 subagent)
- **OpenAI Frontier Harness Eng**:Codex agent 互审 + review agent bias toward merge + 只 surface > P2(见 §5.3)

**为什么要拆**:如果不拆,"多 agent 并行" 会被误读为"一个 owner 管 N 个通用 agent" —— 但 GStack 和 Boris 的实测都说明:**真正的并行能力来自角色专用化**,不是通用 agent 数量堆叠。不拆开,流程会卡在 1:1 使用模式。

### 6.1 所有 owner 共同的能力升级

所有 human owner 在 Agent 时代都要补一项核心能力:**cognitive gearing —— 能在多个角色专用 agent 之间正确激活 / 停用,并保持判断质量**。这不是"能不能用好一个 agent",是"能不能让一个 CEO 模板先做方向 review,再切换到 Eng Manager 模板拆任务,再让 Staff Engineer 模板做 review" 这种切换。

这一项很难直接培训,但可以通过 §10 的试跑机制反向验证 —— 如果某 owner 始终只能 1:1 用 agent(只用一个通用 Claude),那 §3.3-5(加 agent 优先于加人)对他就不成立。

具体来说,所有 owner 都要做到:

- 能给每个 agent 角色模板写清楚 onboarding(一个 skill folder,不是口头交代)
- 能在多个并行原型 / 输出之间快速做品味判断
- 能识别"这件事激活哪个角色模板最合适",并知道激活顺序
- 能识别"这个问题是 agent 可以做的,还是必须我自己做的"
- 为自己的 agent 团队维护角色专用 skill,不只是个人用法

**量化锚点参考**(详见 §10.9,口径以 §10.9 和 §7.3 梯度为准):入门 3-5 并行 / 合格 5-8 并行 / 熟手 8-15(对标 Boris Cherny 实测单人 15)/ 组织级上限 100+(Anthropic)。

### 6.2 Harness ownership(分阶段演进)

Harness 的维护不能散落在各场景工程师的顺手活里,需要明确的 ownership。但是否要马上设立专职岗位,取决于聚星当前规模;一开始用兼任即可,有明确信号时再升级。

**核心职责(始终要有人负责)**
- 维护 `CLAUDE.md / AGENTS.md / 全局 skill 库`
- 维护 eval harness(golden cases、回归集、evaluator 配置)
- 维护 context 架构(write / select / compress / isolate 策略)
- 维护工具集(哪些工具暴露给哪些 agent、定期审计)
- 跨场景识别重复模式

**承载方式,按阶段递进**

阶段一 —— 兼任 Harness owner(试跑期默认)
- 由现有研发兼任,明确时间配额(建议不少于 30%)
- 定期(每周或每 sprint)回答一次:这段时间 harness 有哪些改动、还有哪些缺口
- 有权跨场景提建议,跨场景的统一约束需通过 §4.3 的接口机制与 Process A owner 对齐

阶段二 —— 专职 Harness Engineer / Agent Platform Owner(触发式升级)
- 这是 OpenAI 已经正名的工种(harness engineering 作为独立学科)
- 下列信号任意出现两个,启动升级讨论:
  - 跨场景审计发现重复能力或冗余工具 ≥ 3 处
  - 兼任者一个月内 ≥ 2 次被业务优先级挤走 harness 工作
  - 新增 agent 场景的 onboarding 时间持续增加
  - 某个场景的 harness 改动与其他场景冲突或被回退
- 升级后主要职责扩展到:跨场景平台化能力、公共组件抽取、eval 基础设施的自动化投入

**不独立完成的事**
- 不替代各场景工程师对自己业务领域的判断
- 不独自决定业务优先级
- 跨场景变更必须和对应 owner 对齐,而不是强推

**风险说明(聚星当前经验规则)**:如果 harness 职责长期无人稳定承担,跨产品线的 agent 执行环境容易出现"各做各的、互不共享"的状态 —— 尤其当 agent 主线、Skills 能力层、老平台 Skill 化迁移并行推进时,这种分化会以倍数放大。这是聚星当前结构下需要主动防范的风险,建议阶段一就显式 owner 化。

### 6.3 设计

设计在这套流程里的职责,不应该再只理解成"静态稿输出",而应该明确包括:

- 参与原型生成与第一轮筛选(§5.1 原型海)
- 对关键体验面做最终收口
- 为 onboarding / trust / 信息架构负责(§5.6 体验和信任是一级目标)
- 把有效的设计判断沉淀成规则、清单和 skills(§5.9)
- 在需要时,要求 agent / 研发快速生成临时调节工具来辅助决策

设计师在 Agent 时代的核心稀缺性正在上升:**当 100 个原型一夜之间能跑出来,从中选出"用户敢用、愿意用、持续用"的那个,设计判断就成了关键瓶颈之一**。

### 6.4 Agent / automation

Agent 不是工具,也不是中心,而是**一组可按角色激活的 coworker**。Felix Rieseberg 原话:"就把 Claude 当成你的 coworker";GStack 把这条延伸为"你应该有一个 coworker 团队,每个人有专长,你像 CEO 一样调度他们"。

具体职责(按 §6.0 的角色模板激活):

- 产品侧:起草 pitch、归纳用户反馈、整理候选方案、生成原型变体 → 主要激活 **CEO 模板 + Doc 模板**
- 设计侧:生成方案变体、辅助页面对比、生成临时调节工具 → 主要激活 **Designer 模板**
- 研发侧:生成第一版实现、agent 互审、并行原型验证 → 按任务阶段顺序激活 **Eng Manager → Staff Engineer → Reviewer → Verify → Release** 五个模板(对齐 GStack Build → Review → Test → Ship 节奏;Light 档可裁剪掉 Reviewer/Release,Heavy 档全链路)
- 跨场景通用:所有 role template 都有独立 skill folder(§8.3),由 Harness owner 集中维护

**通用约束**:
- Agent 在 merge 前不需要人复审(§5.3 已升级为 0 pre-merge human review);但在**方向拍板(§7.4 L3)**和**试验场选择(§3.5)**这两件事上,人保留单线程决策权 —— 这两处是"品味落点",不是"PR 审查"
- 任何要流转给下一 role template 的 Heavy 档产物,必须通过 Reviewer 模板的 rubric 检查 —— **人定 rubric,不走人审**
- 如果 agent 在某个环节连续两轮都需要大幅返工,应向 Harness owner 报告 —— **这往往是 skill / rubric 缺失,不是 agent 问题**(默默接受返工 = 默默放过 harness 缺陷)

### 6.5 非传统角色的 agent 自助

Anthropic、OpenAI 等前沿公司的实测:**法务、市场、运营、数据科学家可以直接用 agent 出实现**,不必每次都走研发。Anthropic 法务团队用 Claude Code 做了"phone tree"系统;市场团队 2 个 agent 自动化几百条广告变体生成。

**极端实证**:Peter Steinberger(奥地利独立开发者、PSPDFKit 创始人)2025-11 用一次 coding session 做出 OpenClaw 原型,2026-02 被 OpenAI 招揽;60 天内 GitHub 250K 星(史上最快增长开源项目之一)。**一个人 + 一套角色专用 skill 的产出,可以超过传统 20 人团队**。这是 §3.3-5 "加 agent 优先于加人"最直接的验证。

聚星在合适场景下,也应该允许:

- 任何 owner 用一套**共享角色模板表**(§6.0)独立跑完 pitch → 原型 → 试跑,不必每次走研发代写
- 研发的角色重新定义为"提供底座、维护 harness 和角色模板表",**不是"所有实现必须流经研发"**

这样做的好处:
- 把研发从"瓶颈"角色解放出来,聚焦底座和 harness
- 让产品、设计、运营 owner 直接验证自己的判断,缩短反馈链
- **释放更多组织级并行度** —— §3.3-5(加 agent 不加人)只有在所有角色都能调用共享角色模板表时才真正成立

## 7. Process A:产品流程主循环

Process A 处理人的稀缺资源(判断、品味、对齐),节奏由业务驱动。

### 7.1 输入与分流(全渠道 kickoff)
- 明确输入来源:
  - 用户反馈
  - 销售 / 运营反馈
  - 内部策略调整
  - 技术问题 / 线上问题
  - 流程问题
- 判断输入属于哪一类:
  - 新需求
  - 优化项
  - 问题修复
  - 流程类事项
  - 需要原型验证的方向判断
- **全渠道 kickoff 规则梯度**(对标 Cursor 3 2026-04 范式;按规则分层约定 tag):
  - **【当前默认】**:聚星现有已经配置好的触发源继续按原样工作 —— 用户反馈 / bug 告警 / 产品会议等通过人手工进 agent session;已存在的自动化入口(如 eval 回归失败触发)照常。**不要求立即开通新渠道**。
  - **【试运行例外】**:在当前试验场(§3.5)先开通 **1 个新渠道**(推荐飞书 / 钉钉关键词触发或 Linear 特定标签 ticket 起 session),跑顺后再决策扩展。
  - **【目标态】**:完整的全渠道 kickoff 架构 —— 每一个工作实际发生的渠道都是 agent kickoff 入口,不挤到单一 session:
    - **飞书 / 钉钉**:关键词触发(如"查一下""帮我分析")直接起 session
    - **Gmail / 邮件**:特定 label 或发件人的邮件直接起 session
    - **GitHub / 线上监控**:issue 创建、eval 回归失败、线上 bug 告警直接起 session
    - **Linear / 任务系统**:特定标签的 ticket 直接起 session
    - **移动端 / web**:产品/设计/运营 owner 从手上设备随时起 session
    - **统一 console**:所有正在跑的 agent(本地 + 云端)汇总到一个 console,类似 Cursor 3 Agent Console;Harness owner 负责这个 console 的集成
- **规则骨**:**是否允许 agent session 跨渠道分散**是目标态的特征,聚星还不在这个位置,别强推。
- 渠道接入配置由 Harness owner 维护;新增渠道走 Process B(§8),不走 Process A。

### 7.2 入口检查(分层)

任一方向在进入原型池前,先过一次入口检查。**具体要求按方向规模分三档,不是所有事情都要写三件** —— 统一硬门槛会把 bug fix 也拖慢,反而和 §5.1、§5.7 冲突。

**Light — bug、轻量优化、探索性 spike**
- 只要一段 pitch 即可(一句话 + 预估时间盒也可)
- 不要求 spec 和 eval,除非发现会碰线上核心路径
- 典型场景:UI 小调整、已知 bug 修复、< 1 天 spike

**Standard — 常规功能、明确范围的优化**
- pitch + 简化的 spec
- eval 可选;如果改动会影响已有 golden cases 必须跑回归
- 典型场景:已有场景的新能力、单场景内的重构

**Heavy — 新方向、新场景、用户可见的大变化**
- pitch + spec + 3 到 5 条 golden eval cases(含至少一条 forbidden action)
- 向 Harness owner 报备,评估是否触发 context / 工具 / skill 变动
- 典型场景:新场景立项、跨场景的能力整合、主流程改写

**档位选择**:由发起人初判。L2 角色筛(§7.4)中的 owner 有权要求升档。档位判断错了并不致命,但默认就低不就高,避免用重门槛挤压实验速度。

---

**Pitch(所有档必备,短,结构化)**
- 要解决的问题
- 面向谁
- 为什么这一轮值得试
- 这轮投入上限
- 主要风险点
- 这轮明确不做什么
- 最小成功信号
- 时间盒

**Spec(Standard / Heavy 必备,薄,可机读)**
- 核心输入输出断言:给定 X,必须产出具备 Y 属性的 Z
- 1 到 3 条非目标 / 边界条件
- 如涉及用户可见面,附最小界面草图或 ASCII 流

**Eval(Heavy 必备,Standard 可选)**
- 每条至少包括:输入、期望行为、评判维度
- 至少一条 forbidden action(什么情况下必须不做)
- 这组 eval 在原型池期间持续运行,用来判断每个变体是否在退步

### 7.3 原型海

对值得探索的方向,**不直接收敛到单一方案,默认跑多个并行版本**。

- 并行规模没有硬上限,**真正的约束是 owner 的判断带宽,不是工程师手数**(§3.3-1 与 Anthropic 内部 100 原型并行实践)
- **并行规模梯度**(按规则分层约定 tag):
  - **【试运行例外】第一次跑原型海:≥ 3 个并行**,验证模式能跑通。试运行一两次后向下列档位收敛,不作为长期执行值。
  - **【当前默认】按 owner 的 cognitive gearing 成熟度分档(§10.9):入门 3-5 / 合格 5-8**。聚星绝大多数 owner 目前在这两档。
  - **【目标态】熟手 8-15 并行,起点 5-10**(Boris Cherny 实测单人同时管 5 Claude Code + 5-10 browser session = 15 个 agent)。聚星目标是让至少部分 owner 达到这档。
  - **【组织级上限参考】100 原型并行**(Anthropic 内部实测,Felix Rieseberg "给我 10 分钟,我给你一个版本";聚星整个研发组织可在一年内达到的理论上限,不是单人值)
- 同时跑多少个,由 owner 根据当前自己能消化的判断量决定;**判断带宽不够时,先提升 harness(补 skill / rubric / eval),再加并行数**,不是反过来
- 原型池只接收已经过 §7.2 入口的方向
- 原型生成阶段遵循 §5.2 "Read 并行",可以派 N 路角色专用 agent(§6.0)独立生成变体
- 原型重点验证:
  - 用户第一眼会看到什么
  - 首次使用路径是否走得通
  - 用户可见界面截图或 demo 是什么
  - pitch / spec 里没预见到的新问题是什么
  - 技术上是否有阻断风险
  - 按 §7.2 的 eval(Heavy 档必有、其他档如有)评分,当前达到 / 未达到什么程度

**关键经验(Anthropic 实践):"原型海"模式下,真正决定胜负的不是哪个原型最早写完,而是 owner 能在多快多稳的节奏下从原型海里挑出最值得继续的那个。** 这是 §5.5 品味的正面落点。

### 7.4 三层判断与拍板

对原型不做无限并行,判断也不集中压给同一个人。判断分三层:

**L1 自动筛(Harness 承担,本质是 agent 互审)**
- 对 Heavy 档:用 §7.2 提交的 eval 跑每个原型,得到客观分数;并由 evaluator agent(独立人格,见 §5.3)做一轮交叉 review
- 对 Standard / Light 档:跳过 eval 自动分,只做形式和重复性检查
- Agent + 规则检查 pitch / spec 完整性、是否与已有方向明显重复、是否达到最基本的进入标准
- 这一层的规则和 agent 配置由 Harness owner 维护,**人不参与逐个 review**

**L2 角色筛(各 owner 在自己领域)**
- 产品 / 设计 / 研发各自在自己的领域做快速判断
- 只回答:在我这一侧,这个方向是否成立
- 这一层是并行 Read,不是串行会议
- L2 的输入是 L1 已经过滤过的候选,owner 不需要看全部原型

**L3 方向拍板(单线程 Write)**
- 拍板人只在通过前两层过滤的少量候选里做选择
- 遵循 §5.2 Write 串行:这一层必须单人单线程 + 完整上下文

每轮筛选至少回答:
- 哪个版本更容易打动用户
- 哪个版本更容易试跑
- 哪个版本能更快形成闭环
- 哪个版本的风险最可控

必须明确当前拍板人和拍板依据。每轮筛选结束后,只保留少量继续推进的方向,其余直接停止,不进入"先放着以后再说"。

### 7.5 止损与重塑
- 进入本轮的方向,默认都有时间盒,不无限延长
- 判断输入来自 §7.2 中提前定义好的 `最小成功信号 + eval 门槛 + 时间盒`
- 在约定时间内,如果没有达到预设结果,只有三种处理方式:
  - 继续:值得再投
  - 重塑:问题还对,方案要重做
  - 停止:这轮先不继续
- 不允许"方向不清楚、结果也一般,但默认继续拖着做"
- 失败原型也要留下最小结论:为什么被砍 / 哪个判断原来错了 / **哪些可以进入 skill 库 / 哪些 eval case 应该加入回归集**

### 7.6 设计收口回路
- 对进入候选的 1 到 2 个方向,不要直接跳到全量工程化
- 先在真实界面上做一轮高频设计微迭代,由设计 + agent + 研发共同完成
- 这一轮遵循 §5.2 Write 串行:设计收口是单线程的,不再并发 N 个原型
- 重点收口:信息层级 / 页面密度 / 关键文案 / 首屏结构 / 信任元素 / onboarding 提示
- 如果某个决策需要试参数,直接让 agent 生成一次性调节工具(滑块、切换器、对照面板),确认后再固化进实现
- 输出不只是"页面更好看",还包括:
  - 关键页面对比截图
  - 设计决策记录
  - **可复用的规则和参数(进入 Process B,见 §8.5)**
- 如果设计收口发现不是局部细节问题,而是方向本身不成立,**应回到 §7.3 重选或重做**,而不是只在当前方案上硬修

### 7.7 研发与交付
- 被选中的方向进入短周期研发推进
- 研发阶段不只关注功能实现,还要同步准备:
  - onboarding
  - demo / 使用说明
  - 验收路径
  - 多角色交接方式
- **【当前默认·基础形态】Agent 互审默认开启**(见 §5.3):Claude Code / Codex 的内置互审(self-review + verify tests + subagent delegation)对 **Light / Standard / Heavy 三档全部立即生效**(Heavy 档在此之上必须 + golden eval 回归集,见 §8.4);**merge 前零 human review 是当前默认,不是迁移目标**。这一档**不依赖 §6.0 完整 7 角色模板表**。
- **【目标态·完整形态】**:Heavy 档方向走 7 角色模板流水线(§6.0:CEO / Eng Mgr / Staff Engineer / Reviewer / Verify / QA / Release)+ 结构化 rubric;聚星 2026 下半年达到。
- **【试运行例外】结构化 evaluator rubric + 至少 3 个核心角色模板**只在当前选定的试验场(§3.5)先跑 —— 见 §12.2;对应 §13 短期目标里"角色模板表起步"节点。
- Heavy 档方向在合并前必须通过 eval 回归集(§8.4),不允许静默接受回退
- 行业方向锚点:OpenAI Frontier 团队跑完整形态,5 个月产出 1M LOC、零手写代码、零 pre-merge human review;GPT-5.2 上线后同团队零流程改动,日产出翻倍(3.5 → 5-10 PRs/人/天)。聚星**基础形态即刻可用**,继续向完整形态的【目标态】迁移

### 7.8 上线与试跑
- 优先在真实项目中试跑
- 观察的不是"功能是否存在",而是:
  - 能不能顺利进入使用
  - 结果是否稳定
  - 协作是否顺
  - 用户是否愿意继续用
- 每次试跑至少要回答三件事:
  - 用户是不是真的会用
  - 这个版本是不是比其他候选更好
  - 这轮有哪些经验值得沉淀成规则或 skill(交给 Process B)
- 如果试跑发现错的不是当前版本,而是最初的问题定义,就回到 §7.1,**不在当前版本上继续叠补丁**

### 7.9 回收与沉淀

每轮试跑结束后,必须同时产出:

**产品侧**
- 产品结论
- 止损 / 重塑 / 停止决定
- 流程修正

**Harness 侧(§5.10 强制要求)**
- 至少一条 harness 改动:skill / eval / 工具集 / context 规则 / playbook 更新
- 改动必须交给 Harness owner 集成,不能只留在项目目录里

沉淀不是结束动作。新沉淀出来的 skill、模板和规则,必须反过来作用到下一轮的 §7.2 入口模板和 agent 执行上下文。

### 7.10 显式回路
- **设计回路**:§7.6 设计收口发现方向不对 → §7.3 原型海
- **验证回路**:§7.8 试跑发现问题定义错了 → §7.1 输入与分流
- **能力回路**:§7.9 沉淀出的 skill / SOP / eval / template → Process B 集成 → 直接升级下一轮 §7.2 入口和 agent 行为

## 8. Process B:Harness 流程

Process B 处理 harness 这个一级资产,节奏由复利驱动。

### 8.0 聚星 harness 栈与 Karpathy LLM Wiki 同构

Andrej Karpathy 于 2026-04-04 公开的 LLM Wiki 方法(4 原则:**Explicit / Yours / File over app / BYOAI**;3 层:**Raw / Wiki / Schema**)和聚星已经在跑的 `Global-Brain` Obsidian 知识库同构。这个对齐不是风格选择,是结构一致性 —— 聚星的 harness 不需要从零搭建,已经有底座。

| Karpathy 层 | 聚星对应 | 本流程文档位置 |
|---|---|---|
| **Raw Sources**(用户反馈、会议、外部 clip) | `Global-Brain/raw/` + source summaries | §7.1 输入与分流 |
| **Wiki**(LLM 生成并维护的结构化 md) | `Global-Brain/wiki/` entities / overviews / timelines | §9.2 可复用资产 |
| **Schema**(定义结构的配置文档) | `AGENTS.md` / `CLAUDE.md` / 本流程框架文档 | §8.1 Harness 四职能 |
| **Ingest / Query / Lint** 操作 | 日常 capture / 复盘 / 审计 | §5.10 每轮 +1% |

**对流程的骨含义**:
- **Skill 和 wiki 页都属于 file-based asset**(结构化 markdown,可选包含附属子文件):wiki 页多数是单个 .md;skill 在成熟后是 folder(SKILL.md + gotchas + scripts + assets);起步时 skill 也可能只是单个 md。存储形态有差异,但**生产 / 查询 / ingest 方式同源**(Karpathy 方法的核心),这是 §8.3 "grown not built" 能在数量级上成立的前提(100 个 skill 的生产成本 ≈ 100 个 wiki 页的生产成本)
- **任何 agent 都能加载**(BYOAI)—— Claude / Codex / 未来的任何模型都可以 ingest 聚星 harness,不绑定单一厂商
- **数据活过工具**(file over app)—— 即使某个 agent 客户端被淘汰,聚星的 skill / rule / playbook 依然是可读文件

**重要边界**:
- **File-over-app 适用于 harness 层**(内部 skill / rule / 组织记忆 / evaluator rubric / playbook)
- **File-over-app 不适用于用户数据层**(CRM 记录、合作历史、邮件线程等是聚星的产品资产,走产品数据库,不按 markdown folder 组织)
- 这条边界不要混 —— 聚星既是 SaaS 产品(对用户数据负责),又要 harness 可移植(对自己研发组织负责),两者并行不冲突

### 8.1 Harness 的四大职能

Harness 是包裹 agent 的运行环境,做且只做四件事:

**Constrain(约束)**
- agent 能做什么的边界:工具集、权限、沙箱、速率限制
- 行业共识(一手材料):Anthropic 的 agent 工程材料反复强调要复查工具清单是否都必要 —— 工具越多,agent 选择时越容易分心
- 聚星当前经验规则:具体砍多少按各场景实测结果定,不存在一刀切的比例

**Inform(告知)**
- agent 应该做什么的上下文:CLAUDE.md / AGENTS.md、skill 库、rule 文件、示例
- 行业共识(一手材料):OpenAI 公开的 AGENTS.md 约 100 行,可作为参考起点;不要复述模型训练时已知的内容
- 聚星当前经验规则:按实测影响为准,超出参考起点但确实被 agent 用到的内容可以保留

**Verify(验证)**
- 做得对不对:eval harness、hook、evaluator agent
- 行业共识(一手材料):同一 agent 对自身输出存在宽容偏差。对关键交付前的验证,值得让 evaluator 使用与 generator 不同的人格或角度
- 聚星当前经验规则:Heavy 档方向强制做 generator / evaluator 分离;Standard / Light 视情况而定

**Correct(纠正)**
- 错了怎么回:auto-fix、escalation、playbook 更新、context 回滚

### 8.2 Context Engineering 四桶

每个 agent 执行场景、每个 skill 在 harness 层面必须明确回答四件事:

| 桶 | 回答的问题 |
|---|---|
| **Write** | 这次 session 产生的信息,什么要写到窗外(scratchpad / session memory / project memory / org memory)? |
| **Select** | 开始时,从哪里、按什么规则把上下文拉进来? |
| **Compress** | 什么时候压缩?压到多少 token?保留什么、丢弃什么? |
| **Isolate** | 哪些子任务用独立 agent 隔离上下文?主线程保留什么? |

聚星当前经验规则:凡是涉及 agent 实际执行的场景(不论在哪条产品线),Harness owner 值得优先审计的事情之一是 —— 四桶问题在这些场景是否已经被回答、回答是否一致。不同场景用不同策略本身不是问题,但"没人回答过"是风险。

### 8.3 Skill:grown not built,folder 而不是单文件

Anthropic 官方 skill 指南(2026-03 公开)的原话是骨:**"Skills are grown, not built — you don't write a perfect Skill on day one"**。skill 元数据的 9 字段是成熟 skill 的目标状态,不是新 skill 的入门门槛。

#### Skill 是 folder,不是单个 markdown 文件

Anthropic 官方正式把 skill 定义为**可扩展的文件夹**:

```
skill-name/
├── SKILL.md             ← 主指令(trigger + 预期输出 + 核心 how-to)
├── gotchas.md           ← 真实使用中踩过的坑(最高价值部分,见下)
├── scripts/             ← 重复给 agent 的指令抽成脚本(≥ 3 次重复就抽)
├── assets/              ← 模板、示例、参考文档
└── reference/           ← progressive disclosure 子文件(skill 膨胀时按需加载)
```

单个 markdown 文件只是 skill 文件夹的一种退化形态(起步态);成熟 skill 一定是 folder。

#### Gotchas 是 skill 最高价值部分

Anthropic 官方:"The most valuable content in any skill is the Gotchas section — based on real problems Claude encounters when using your skill"。

- Gotchas 不是预先设计,是**在真实使用中累积**
- 每次 agent 出错 / 返工,都应该回头补一条 Gotcha
- **没有 Gotchas 的 skill 是不完整的** —— 因为模型训练时已经有的东西不需要写进去(参见 OpenAI AGENTS.md 100 行原则),skill 的独特价值恰恰是 "agent 自己想不到、或会想错的那些点"

#### 起步门槛(新 skill 最小集合)

新建一个 skill 只需要:

- **SKILL.md**:说清楚 trigger(什么情况下调用)+ 预期输出(调用完成后应该产出什么)
- **1 条 Gotcha**(哪怕只是占位"待真实使用中填充"),留出累积位置
- **owner 和 last_verified** 两个维护字段

其他字段是**成熟 skill 的目标状态**,在使用中逐步累积,不是第一天的硬门槛。

#### 成熟 skill 的目标字段(为目标状态,不是入门门槛)

- **触发条件 / 预期输出 / 验证标准**:基本三件
- **owner / last_verified / last_used**:维护属性
- **Gotchas**:最高价值部分,使用中累积
- **Constrain**:调用此 skill 时收紧哪些工具 / 权限
- **Context budget**:进来时最多消耗多少 token,结束时最多留多少
- **Postcondition**:成功后世界必须是什么样
- **Forbidden action**:绝不能做的是什么
- **Progressive disclosure index**:子文件路径和加载条件(skill 超过单文件可读容量时必有)

#### Skill 的核心三类(聚星优先使用)

- **执行类 skill**(instruction):告诉 agent 怎么做某件事(订机票、生成报告、起草 pitch)
- **评测类 skill**(reviewer):告诉 evaluator agent 怎么判断输出好坏(rubric + forbidden actions + gotchas)
- **修正类 skill**(playbook):告诉 agent 出错时怎么回退、怎么补救

**Anthropic 内部实测**:skill 最终归纳为 **9 个 category**,但大多数团队只用其中 2-3 种(不是因为其他不值得,是"不知道它存在")。聚星 Harness owner 应定期审视是否引入新 category(例如 onboarding / safety / formatting / meta-review / 数据转换 / 调度 等)。

#### Skill 的生命周期

- **新 skill**:满足本节"起步门槛"4 件(SKILL.md 含 trigger + 预期输出、1 条 Gotcha、owner、last_verified),即可直接上线,不走 9 字段审查
- **成长期**:每次 agent 使用出错 / 返工,补一条 Gotcha 或 script
- **成熟 skill**:9 字段齐全、progressive disclosure、Gotchas 持续累积
- **淘汰 skill**:超过多个 cycle 没被引用,进入待 review / 归档
- **删除 skill**:比新增 skill 更难(会破坏已有 agent 的依赖),每次删除必须在 Process B 中记录原因

#### 与角色模板表(§6.0)的关系

§6.0 的 agent 角色模板表里每一个模板(CEO / Staff Engineer / Reviewer / Verify 等)**都对应一个 skill folder**。角色模板表不是抽象概念,是一组可直接 ingest 的 skill。扩展角色模板 = 新建 skill folder。

### 8.4 Eval Harness

- 每个 Heavy 档方向提交的 3-5 条 golden eval cases,进入全局 eval 回归集;Standard 档如果有自带 eval 也可进入
- 回归集在 harness 改动和场景改动时自动跑;具体触发频率(每 PR / 每天 / 每周)由 Harness owner 根据"跑一次的成本 × 对产出的保护收益"决定,**重点是有明确规则,不靠记性**
- 聚星当前经验规则:回归跌破基线时,要有明确的处置路径 —— 阻塞合并、或 owner 承认并留下记录 —— **不允许静默接受回退**
- Harness owner 负责维护回归集质量,定期淘汰过时或重复的 case,防止过拟合
- 行业方向锚点:OpenAI Frontier Harness Engineering 实验(Ryan Lopopolo 团队,2026-02 公开)里,**eval harness + agent reviewer 的组合让团队完全不需要人工 code review**(5 个月、**1M LOC、0 手写代码、0 pre-merge human review**、1B tokens/day);从 12 月 3.5 PRs/人/天 → 1 月 GPT-5.2 后 5-10 PRs/人/天,**同团队零流程改动**,说明 eval + review agent 框架本身具备直接吸收模型能力跃迁的能力。聚星不必一步到位,但这是 §5.3 终局状态的可达到形态
- 聚星当前经验规则:如果 eval 回归跑一次的成本高到会被日常跳过,就是危险信号 —— 要拆分、抽样或优化,而不是接受现状

### 8.5 Taste → Eval 编码管线

品味和评测不是两件事,是同一件事的两个阶段:

1. 人在 Process A 中做了品味判断(这个设计更好 / 这个输出更自然)
2. 说出 why(taste articulation)—— 哪怕只是一句话
3. 把 why 写成 rubric(1-5 分 + 理由维度)
4. 把 rubric 固化为自动 eval 或 evaluator prompt(成为 §8.3 的"评测类 skill")

这条管线是 Process B 的主要产出路径之一,也是 §5.5 "人聚焦最高杠杆位置"能持续 scale 的关键 —— 没有它,每次品味判断都是一次性消耗;有了它,品味会逐步沉淀成组织资产。

### 8.6 Harness 流程的节奏

下面这组节奏更适合作为**聚星当前试运行的参考起点**,不是所有团队一上来都要配满:

- 每日或每工作日:对关键回归自动跑;如果暂时做不到全自动,至少要保证改动前后可手动触发
- 每周:Harness owner 和各 Process A owner 对齐一次,梳理本周 harness 改动和下周 backlog
- 每月:跨场景审计一次,识别重复模式并合并

### 8.7 Harness 反模式

以下几件事是 harness 质量退化的典型信号,出现时必须立即处理:

- 某个场景的 CLAUDE.md 在持续膨胀,已经没人能在几分钟内读完并抓住重点(60-100 行可作为警戒起点,不是硬上限)
- 工具清单在膨胀,但很少有 agent 真正用全
- skill 库在增长,但长期没有 skill 被 agent 实际调用
- **skill 没有 Gotchas section**(或 Gotchas 长期为空)—— 意味着 skill 的"真实使用价值"没有被累积,违反 §8.3 "grown not built"
- eval 回归跑一次已经长到团队日常会跳过
- 多个场景在独立实现同一类能力(检索、摘要、审核)
- evaluator 和 generator 用的是同一段 prompt(自我宽容陷阱)
- **Owner 始终只能 1:1 用 agent**(无法管理多个并行原型 / 不激活 §6.0 角色模板)—— 这意味着 §3.3-5"加 agent 不加人"对该 owner 不成立,需要先做 onboarding 而不是直接加任务
- **Agent 角色模板表(§6.0)长期停留在初始几条**(没有随试跑累积) —— 意味着 cognitive gearing 没有真正开始
- **有人仍然在 merge 前 review 代码**(按 §5.3 终局,这是反模式;除非 post-merge sampling 发现了需要 temporary 人工门禁的 systemic 问题)

## 9. 正式产物

### 9.1 Process A 每轮交付物

这轮项目必须交付的内容:

- 需求框架 / 问题定义
- 入口检查产物(Pitch 所有档必备;Spec 在 Standard / Heavy;Eval 在 Heavy;见 §7.2)
- **原型变体集** + 方案筛选记录(含三层判断的理由)
- 关键页面截图 / 设计对比
- 设计决策记录 / 参数说明
- 使用说明 / onboarding 材料
- 验收标准
- 试跑反馈记录
- 关键决策记录
- 止损 / 重塑 / 停止结论
- **向 Process B 提交的 harness 改动清单(§5.10 强制)**

其中,关键决策记录不追求大而全,但每轮至少记录 1 到 3 个关键分叉点:
- 当时有哪些可选方案
- 最后选了哪个,为什么
- 事后验证结果如何

### 9.2 Process B 可复用资产

只有满足下面条件之一的内容,才算真正的可复用资产:

- 能被 agent 直接加载进执行上下文(skill folder / rule / playbook)
- 能被人直接作为 checklist / template / 标准使用
- 能在下一轮 Process A 中被明确引用或自动触发

具体形式:

- **Skill folder**(不是单个 md 文件,见 §8.3):三种核心类型(执行 / 评测 / 修正);含 SKILL.md + Gotchas + scripts + progressive disclosure 子文件
- **Agent 角色模板表**(§6.0):每个模板对应一个 skill folder,跨产品线共享
- Playbook(触发 / 步骤 / postcondition / 纠错指引 / 禁止事项)
- Eval cases(golden set + 回归集)
- Design rules / prompt cards / UI skills
- 工具白名单与工具审计记录
- Context 架构文档(四桶策略,见 §8.2)
- 流程问题清单与修正记录
- 之后还能继续引用的模板和标准

## 10. 当前最需要补齐的机制

### 10.1 原型海机制(Process A)
- 一个方向如何通过 §7.2 入口
- 同时并行多少版本,由谁判断 owner 的判断带宽够不够
- 原型何时进入试跑
- 哪些方向需要设计 owner 提前介入收口

### 10.2 三层判断机制(Process A)
- 谁拍板
- 用什么标准筛选
- 设计质量和体验判断在拍板里占多大权重
- 哪些结果触发继续 / 重塑 / 停止
- L1 自动筛的规则和 evaluator agent 配置归谁维护(Harness owner)

### 10.3 Harness 组件机制(Process B)
- 哪类经验必须沉淀成 skill / playbook
- 沉淀到哪里、谁维护、如何在项目中被 agent 自动加载
- **新 skill 起步门槛(与 §8.3 严格对齐)**:SKILL.md(含 trigger + 预期输出)+ 1 条 Gotcha + owner + last_verified 四件,缺一不可;9 字段是成熟 skill 的【目标态】,不是入门门槛
- **Skill 是 folder**(不是单个 md):SKILL.md + gotchas + scripts + assets + progressive disclosure 子文件。单 md 是退化起步态,不是最终形态
- 超过多个 cycle 没被引用的 skill,进入待 review / 归档状态
- **Agent 角色模板表**(§6.0)的维护:新增 / 修改 / 淘汰模板走 Process B,由 Harness owner 批准

### 10.4 Eval 机制(Process B)
- 每个 Heavy 档方向在 pitch 阶段就要提交 golden cases
- 全局 eval 回归集如何维护、淘汰、防止过拟合
- eval 失败如何阻塞合并
- taste → rubric → eval 的编码周期

### 10.5 Context / 工具管理机制(Process B)
- 每个场景的 write / select / compress / isolate 策略是否写下来
- 工具集是否定期审计、是否存在可砍的冗余
- 哪些能力在场景之间重复实现,能否合并

### 10.6 设计资产机制
- 哪些设计判断必须沉淀成参数、规则、组件约束或 checklist
- 哪些场景允许先生成一次性调节工具,再沉淀最终结论
- 设计规则放在哪里,谁维护,如何在项目中复用

### 10.7 信任与 onboarding 机制
- 新能力上线后,如何让用户知道它能做什么
- 如何减少第一次使用门槛
- 如何让结果更可见、更可控

### 10.8 验证与复盘机制
- 每次试跑至少追哪几个最小指标
- 如何判断某个版本比另一个版本更好
- 哪些失败结论必须记录,避免反复踩坑

默认的最小指标集可以从下面四类里选,不要求每次全追,但至少要选:

- **结果信号**:用户能否在无引导下完成核心操作 / 用户是否更愿意继续使用某个版本
- **过程信号**:试跑期间是否出现明显异常或阻断 bug / 交接给其他角色时是否需要额外解释
- **Eval 信号**:golden cases 的分数变化 / 回归集是否有退步
- **Harness 信号**:这一轮让 harness 变强了什么(§5.10)

### 10.9 Owner 能力评估机制(Cognitive Gearing)
- 每个 owner 能并行管理多少 agent / 多少原型?能否在多个角色模板(§6.0)之间正确激活 / 停用?
- 哪些 owner 还停留在 1:1 用 agent 的状态?他们需要什么 onboarding?
- 用什么频次评估、谁负责评估(建议 Harness owner 每月一次)
- 这一项不是 KPI,是 §3.3-5 "加 agent 不加人"能否落地的前置条件

**量化参考锚点(2026-04 前沿实测)**:

| 水平 | 并行数 | 说明 |
|---|---|---|
| **入门** | 3-5 | 聚星当前起步基线 |
| **合格** | 5-8 | 对标 Anthropic 工程师日常工作(Felix Rieseberg:"3-8 个 Claude 窗口") |
| **熟手** | 8-15 | 对标 Boris Cherny 实测(5 个 Claude Code + 5-10 个 browser session,单人同时管 15 个 agent) |
| **组织级** | 100+ | 对标 Anthropic 内部 100 原型并行(整个公司范围,不是单人) |

**识别方式**:看 owner 每周处理的原型数 × 每个原型平均判断深度。如果 owner 的答复是"我今天只能看一个方向,其他顾不过来",说明 cognitive gearing 能力未建立,需要先补 skill 和角色模板(harness 侧),不是让他再努力。

## 11. 试跑方式

- 用实际项目采用这套新流程
- 草稿过程中不追求一次定稿,而是边用边改
- 近两周保持高频沟通,优先收集当前正在发生的问题
- 试跑项目最好包含至少一个强用户可见的体验面,不要只拿纯内部逻辑链路试
- 每次试跑后,不只问"产品哪里有问题",也问:
  - 流程哪里卡了
  - 哪个角色衔接不顺
  - 哪些经验值得沉淀成 skill
  - **harness 哪里被发现是瓶颈**
  - **owner 是否能管多个并行原型,瓶颈在哪**
  - 哪个方向应该继续,哪个方向应该停

## 12. 近期落地建议

### 12.1 试运行最低配

如果近两周就要开始跑,建议先只强制下面五件事,不要一上来把全部机制配满:

1. 选 1 个真实项目作为试跑样本,并先判定它属于 Light / Standard / Heavy 哪一档
2. 指定 1 位兼任 Harness owner,给固定时间配额(≥ 30%);先做到"有人持续维护",再讨论是否专职
3. **【试运行例外】强制至少跑 3 个并行原型**(即使 Light 档也试一次)—— 这是"原型海"模式跑顺的最小起点;跑顺后按 §10.9 "入门 3-5 / 合格 5-8"档位向前推,向 §7.3 【目标态】5-10 收敛
4. 在 Process A 里强制写清:拍板人是谁、这轮时间盒是什么、最小成功信号是什么
5. 每周做 1 次双复盘:
   - **产品侧**:这轮该继续 / 重塑 / 停止什么
   - **Harness 侧**:这轮具体升级了什么,还有什么缺口

### 12.2 增强项(有余力再加)

在最低配跑顺之后,再逐步加下面几项:

1. 如果试跑样本属于 Heavy 档,强制附带 spec + 3 条 golden eval cases,作为 L1 自动筛的试跑素材
2. **【试运行例外】升级到结构化 evaluator rubric**(§5.3 第三档:带独立 rubric / forbidden actions / golden cases)—— 注意:**基础形态 agent 互审已在 §5.3【当前默认】生效**,本增强项是从"自带互审"升级到"带独立 rubric 的 Heavy 档形态"。目的不是省时间,是**把人从 PR 循环里抽出来,放回定 rubric + post-merge sampling 的稀缺位置**,并为跨场景复用打基础
3. 让设计参与第一轮收口,把关键体验判断和参数规则留下来
4. 每轮沟通后,把有效做法补成文本化产物,包括设计规则和体验判断,由 Harness owner 统一归档
5. 评估每位 owner 的"能管多少 agent"基线,识别 onboarding 缺口

### 12.3 各产品线的切入深度

这套流程**适用于聚星全部产品线,但按产品线的天然节奏分档切入**,不搞一刀切。

| 产品线节奏特征 | 切入深度起点 | 可选加强 |
|---|---|---|
| **Light 节奏**(bug fix 主导 / 文档交付 / 单人独立完成) 例:插件用户文档 / TTShop 开户提审 | **【当前默认】**只用 §7.2 Light 档模板(一段 pitch),§5.3 基础形态 agent 互审即刻可用,其他机制暂不强制 | 出现用户可见体验面时,升 Standard |
| **Standard 节奏**(已上线模块的增量优化 / 多角色协作但反馈周期 ≤ 2 周) 例:消消乐 XXL 的邮件协作优化、Skills 某个模块的接入 | **【当前默认】**Light 档全量 + pitch/spec + 至少 3-5 个并行原型(§7.3 梯度"入门 3-5")+ 每周 1 次双复盘 + §5.3 基础形态 agent 互审 | §8.5 taste → eval 编码管线开始起跑 |
| **Heavy 节奏**(新产品线 / 新场景 / 跨产品线能力整合 / 用户可见大改版) 例:agent 主线、网红经纪人推进、老平台 Skill 化迁移 | **【当前默认】**Process A + Process B 全量适用,含 golden eval 回归集 + §5.3 基础形态 agent 互审;**【目标态】**升级到 §5.3 完整形态(7 角色模板 + 结构化 rubric) | 当某条 Heavy 线的 harness 改动影响到其他产品线时,Harness owner 升级触发(§6.2) |

**说明**:
- 哪条产品线属于哪类节奏,由该产品线的 owner 和 Harness owner 共同判定;节奏会随产品线演化变化,不是终身档位
- 试验场的首选来自 Standard 和 Heavy(这两档更容易同时满足 §3.5 的规则,Light 档一般不作为试验场)
- 某些跨产品线的推进(邮件通道优化、海外市场拓展优化包)**未来可能需要销售、运营、客户成功或客服团队配合**,届时再把接入点补进本节

## 13. 短期目标

- 两周左右产出一版可执行的流程框架
- 至少通过一个实际项目完成 Process A 试跑,**至少跑过一次 ≥ 5 个并行原型的方向**(【目标态】下限;如果首次试运行只能做到 3 个也算过关,见 §12.1)
- 至少沉淀出一份流程类、设计类或 evaluator 类 skill **folder**(不是单个 md),含 SKILL.md + 至少 1 条 Gotcha,并被下一轮 pitch 明确引用或加载
- **至少搭起一份聚星 agent 角色模板表(§6.0)的起步版**:至少含 3 个核心模板(比如 Staff Engineer reviewer + Verify + CEO 方向 review),各自有起步 skill folder
- Harness owner 交付一次跨场景审计,识别出至少 3 处可以合并的重复能力或可以砍掉的冗余工具
- **至少识别出 1 位 owner 能稳定管理 ≥ 5 个并行 agent / 原型的工作流**(§10.9 "合格"档下限;作为 §3.3-5 落地的样板;试运行若卡在 3 也接受,重点看迁移速度)
- 把试跑中暴露的问题补回流程文档,使它从"讨论稿"进化为"可执行稿"
