目标 + 规则约定(统领全图)

先看这一张。目标决定方向,规则约定决定怎么读后面所有数字,试验场规则决定第一步往哪走。

目标

建立一套能让聚星研发组织的产品化节奏完全发挥模型节奏的研发操作系统。
模型能力当前是溢出的,组织真正的瓶颈是把模型能力以匹配的节奏吸收、转化、交付出去的能力。这套流程的全部设计都围绕这一个瓶颈展开。

规则分层约定 · 所有可执行规则按"落地时效"标三档

【目标态】
前沿已实证、聚星应该走到的终态
如:0 pre-merge human review;原型海起点 5-10 并行;7 角色模板流水线
【当前默认】
聚星研发组织当前即可按此执行的规则,无需额外配置或能力建设就成立
如:Claude Code 内置 agent 互审;入门 3-5 并行;Light/Standard 档自动适用
【试运行例外】
现阶段先从某场景试跑的起步下限,跑顺后向【当前默认】或【目标态】收敛
如:第一次原型海至少 3 个;角色模板表起步 3 个;开 1 个新 kickoff 渠道

梯度方向:【试运行例外】 → 【当前默认】 → 【目标态】(按场景成熟度逐档收敛)

试验场筛选规则(§3.5)

挑那种能在 2 周内让多个原型撞到真实用户、并逼产研和另一个角色对齐判断的场景;
避开被线上 SLA、跨月 rollout、或单角色闭环锁死的场景。

推导自 §3 成本格局:多原型(执行 ≈ 0)/ 真实用户(信任不能在真空里建)/ 多角色对齐(alignment 是稀缺瓶颈)/ 2 周闭环(学习速度 ∝ 1/反馈周期)/ 避开 SLA(试错代价 < 学习价值)。

Skill = folder · grown not built(§8.3)

Anthropic 官方:"Skills are grown, not built — you don't write a perfect Skill on day one"。 Skill 是文件夹,不是单个 markdown。Gotchas 是最高价值部分,在真实使用中累积。

skill-name/ ├── SKILL.md # 起步必备 · 主指令(trigger + 预期输出) ├── gotchas.md # 起步必备 · 真实使用中踩过的坑(最高价值) ├── scripts/ # 重复给 agent 的指令 ≥ 3 次时抽成脚本 ├── assets/ # 模板 · 示例 · 参考文档 └── reference/ # progressive disclosure(skill 膨胀时按需加载)

4

角色模型 · 人 vs Agent 模板(两个独立维度) 骨

这是 §6 的骨:人角色和 agent 角色是两个独立的维度,不是一张表。一个 human owner 同时指挥多个不同模板的 agent(cognitive gearing),不是 1:1。

A. 人角色(owner 类)
谁对什么产出负责;沿用聚星现有组织结构,不因 agent 时代而消失
  • 产品 / 需求 owner
  • 设计
  • 前后端研发
  • 测试 / 验收
  • 上线 / 运营协同
  • 业务反馈输入方
  • Harness owner(可兼任或专职,见 §6.2)
B. Agent 角色模板(role template 类)
agent 在具体任务里扮演什么人格 / 专长;聚星初始推荐的 9 种核心模板(按实际使用扩展)
CEO 模板方向 review · 是否值得做
Eng Manager 模板架构 · 数据流 · 边界 case
Staff Engineer 模板"paranoid" code review · 找生产问题
Designer 模板设计决策 · 参数判断 · UI 对比
QA 模板测试生成 · 覆盖率检查
Release 模板打包 · 灰度 · 上线 checklist
Doc 模板onboarding · 使用说明 · changelog
Reviewer 模板对照 rubric 打分 · forbidden action
Verify 模板端到端 happy path 自动跑
为什么要拆? 如果不拆,"多 agent 并行"会被误读为"一个 owner 管 N 个通用 agent" —— 但 GStack(23 角色)和 Boris Cherny(单人 15 个 agent)的实测都说明:真正的并行能力来自角色专用化,不是通用 agent 数量堆叠。每个 agent 角色模板对应一个 skill folder(§8.3),角色模板表本身是 harness 一级资产。
Cognitive Gearing 量化锚点(§10.9)

owner 能并行管理多少个角色专用 agent · 能否在多个角色之间正确激活 / 停用。看 owner 每周处理的原型数 × 每个原型平均判断深度。

Agent 互审 · 0 pre-merge human review(§5.3 三档)

人不审 PR · agent 互审为默认 · 人只做定 rubric + post-merge sampling。 这条规则按"基础 / 完整"两层落地,不要把目标态当成当前默认强推。

self-review + verify tests + subagent delegation; 不需要额外配置。Light / Standard / Heavy 三档全部立即享用(Heavy 档在此之上 + golden eval 回归集)。
merge 前零 human review 已经在跑。

目标态 · 完整形态
7 角色模板流水线 + 结构化 rubric
CEO / Eng Mgr / Staff Engineer / Reviewer / Verify / QA / Release。 聚星 2026 下半年达到。
OpenAI Frontier 2026-02 实证终态。
Review agent 默认规则(OpenAI Frontier 实测)
  • bias 向 merge(不卡住)
  • 只 surface P0 / P1 / P2 级别的问题;抑制 P3 及以下 nice-to-fix 的 nitpick(P0 = 最严重)
  • 反馈以"可接受 / 延后 / 反驳"三选一方式给 authoring agent,迭代到所有 reviewer 满意为止
人在 code review 这一段的工作只剩两件: (1) 定义 evaluator rubric(rubric / forbidden actions / golden cases / gotchas)—— 品味的落点之一; (2) post-merge sampling(抽样识别 systemic issue,不是每个 PR 都看)。

作用域:这一条只管代码审查,不代替 §5.5"人聚焦最高杠杆位置"的 4 件事(目标定义 · 品味 · 对齐 · 信任建设),也不代替 §7.4 L3 方向拍板。
非代码交付桥接规则:用户文档 / 合规材料 / 开户提审 / 对外视频等没有"merge / PR review"字面语义。沿用同一原则,落地形式不同 —— Generator agent 出材料 → Reviewer agent 对 checklist 评分 → Verify agent 跑可读性 / 完整性 / 链接自检 → owner 抽样确认发出后产物。骨原理一致:人定 checklist + agent 互审 + 抽样确认。
原理(§3.3-2) OpenAI Frontier 团队 5 个月、1M LOC、0 手写代码、0 pre-merge human review; 12 月 3.5 PRs/人/天 → 1 月 GPT-5.2 后 5-10 PRs/人/天(同团队、零流程改动)。 GPT-5.2 上线后日产出翻倍 —— 说明 eval + review agent 框架本身具备直接吸收模型能力跃迁的能力。

7

各产品线切入深度(§12.3) 落地

这套流程适用于聚星全部产品线,但按产品线的天然节奏分档切入,不搞一刀切。 哪条产品线属于哪类节奏,由该产品线 owner 和 Harness owner 共同判定。

LIGHT 节奏
例:插件用户文档 / TTShop 开户提审
bug fix 主导 / 文档交付 / 单人独立完成
【当前默认】 只用 §7.2 Light 档模板(一段 pitch),§5.3 基础形态 agent 互审即刻可用,其他机制暂不强制。
出现用户可见体验面时升 Standard。
STANDARD 节奏
例:消消乐 XXL 邮件协作优化 / Skills 模块接入
已上线模块的增量优化 · 多角色协作 · 反馈周期 ≤ 2 周
【当前默认】 Light 档全量 + pitch/spec + 至少 3-5 个并行原型 + 每周 1 次双复盘 + §5.3 基础形态 agent 互审。
§8.5 taste → eval 编码管线开始起跑。
HEAVY 节奏
例:agent 主线 / 网红经纪人推进 / 老平台 Skill 化迁移
新产品线 · 新场景 · 跨产品线能力整合 · 用户可见大改版
【当前默认】 Process A + Process B 全量适用,含 golden eval 回归集 + §5.3 基础形态 agent 互审。
【目标态】 升级到 §5.3 完整形态(7 角色模板 + 结构化 rubric)。
harness 改动影响其他产品线时,Harness owner 升级触发。
试验场首选:Standard 和 Heavy(这两档更容易同时满足 §3.5 的规则);Light 档一般不作为试验场。 某些跨产品线的推进(邮件通道优化 / 海外市场拓展优化包)未来可能需要销售、运营、客户成功或客服团队配合,届时再补接入点。
