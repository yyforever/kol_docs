双流程总览

Process A（产品流程，天→周）跑业务;Process B（Harness 流程，持续） 跑 agent 平台演化。两条流程节奏不同、所有者不同,但通过接口持续互喂: A 每轮让 harness +1%,B 每次改动喂回 A 的入口与上下文。

为什么分开两条流程? Process A 的节奏受业务压力控制,停不下来做基础设施; Process B 靠复利累积,混进项目制会被永久压到最低。把它们分开, agent 平台质量才不会靠"顺手维护"。

Process B · Harness 流程 Agent 平台

Harness 是包裹 agent 的运行环境,做四件事:约束、告知、验证、纠正。 Process B 持续演化 agent 平台,让 Process A 下一轮更便宜。

Harness 四大职能
Constrain 约束
agent 能做什么
工具集 · 权限 · 沙箱 · 速率限制;定期审计、砍掉冗余工具。
Inform 告知
agent 应该做什么
CLAUDE.md · AGENTS.md · skill 库 · rule · 示例。保持精简。
Verify 验证
做得对不对
Eval harness · hook · evaluator(与 generator 人格分离)。
Correct 纠正
错了怎么回
Auto-fix · escalation · playbook 更新 · context 回滚。

Harness ownership · 分阶段演进
阶段一兼任 Harness owner
试跑期默认采用。现有研发兼任,时间配额 ≥ 30%。
工作节奏:每周/每 sprint 回答一次 — 这段时间 harness 有哪些改动、还有哪些缺口。
阶段二专职 Harness Engineer
下列信号出现 ≥ 2 个,启动升级讨论:
  • 跨 graph 重复能力 / 冗余工具 ≥ 3 处
  • 兼任者一个月内 ≥ 2 次被业务优先级挤走
  • 新 graph onboarding 时间持续增加
  • graph 之间 harness 改动冲突 / 被回退
Harness 反模式 · 出现即处理
CLAUDE.md 持续膨胀,复述模型已知内容
工具清单膨胀,很少 agent 真正用全
skill 库在增长,但 30 天内无 agent 调用
eval 跑一次太慢,导致日常被跳过
多个 graph 独立实现同一类能力(检索 / 摘要 / 审核)
evaluator 与 generator 用同一 prompt(自我宽容)
