## 一、周三下午 14:07，一条工单进了群

**2026 年 4 月的某个周三下午。**

美娜在客服后台看到当天第 7 条类似的留言：

> "用 Agent 跑完之后，我想把这几个达人单独留着，为什么没有收藏啊？"

她翻了一下前两周的工单——上周才 3 条，这周累计 12 条。**信号稳定了**。她把这批留言汇总成一段话，扔进产品群：

> "本周 12 个用户问收藏功能，重复率高、描述一致，不是噪音。"

三秒后，杨洋在群里 @了朱旭：

> "Skills 7.8.1 加一条 P0，收藏达人，下周能给方案吗？"

朱旭 30 秒回复：

> "能，下午给雏形。"

—— 如果你看完这段觉得平平无奇，那就对了。**一条来自一线用户的抱怨，被收敛、被拍板、被派活，聚星走完整个链条不到一分钟。**

两年前做同样的事，我们要多久？**一天起步。**

剩下的"实现"呢？朱旭那个下午没打开 IDE 写一行代码。他做的是**激活一整条由 agent 组成的"机器人工位流水线"**，自己只做**品味指导**（选方案、改子任务），其余交给 agent 之间自己互审收敛到满意、自动合并上线。**整条合并链路他一眼都没看过代码**——代码审查已经不是人的工位。

这就是这篇导读想告诉你的：**聚星研发中心已经不是 2023 年那个手工作坊。它正在变成一座工厂。**

## 二、如果你没来过聚星研发中心，它长这样

类比一下：**聚星研发中心 = 一座全自动的超级工厂**。

想象那种你在纪录片里见过的、最现代化的汽车或者电子产品工厂——里面有 200 多种不同专业的机器人，它们自己焊接、自己涂装、自己装电池、自己下线测试。人做的不是拧螺丝，而是**设计工位、维护机器人、抽检成品、判断要不要造新车型**。

把聚星往里头套：

| 工厂组件 | 聚星对应 |
|---|---|
| 厂区 | 聚星研发中心（你们办公室） |
| 产线（每条产一款车型） | 产品线（Skills / 消消乐 / Agent 主线 / 插件 ……） |
| 原料进货口 | 用户反馈、市场信号、运营输入 |
| 专业机器人（200 种） | **9 种 agent 角色模板**（CEO / Eng Mgr / Staff Engineer / Designer / QA / Verify / Release / Reviewer / Doc） |
| 工位操作协议（SOP） | **skill folder**（每个 agent 的"工位手册"：何时做、先查什么、不做什么、做完验什么） |
| 工厂档案室 / 产品知识库（事实） | **wiki**（Karpathy 式 Global-Brain：收藏 API 是什么、账单系统怎么结构、谁维护过） |
| 工厂的传送带、夹具、MES | **harness**（agent 运行环境 + CLAUDE.md + eval 集） |
| 质检站 | agent 自动互审 + 评测员按评分表（rubric）打分 |
| 门店反馈抽检 | 合并上线后再抽样（post-merge sampling · 看线上用户真实反馈） |
| 工位负责人 | 杨洋 / 朱旭 / 杨芳 / 娜娜 / 美娜 |
| 工厂工程师（管工装设备） | **Harness owner（徐伸）** |

> 📌 这 9 个模板里，**代码流水线走其中 7 个**（CEO / Eng Mgr / Staff Engineer / Reviewer / Verify / QA / Release）。Designer 和 Doc 是另外两条专用线——主文档 §5.3 里说的"7 角色流水线"就是这 7 个。

工厂圈有一句老话：**"It's the machine that builds machines."**
工厂本身是一台"造机器的机器"。聚星研发中心也一样：**agent 在生产产品，Harness 在生产 agent 的工作环境**。

## 三、为什么工厂要这样改

两个事实，一句话就能说明白。

**事实一：焊接（写代码）这件事变不要钱了。**
过去工厂 80% 的成本在人工焊接。现在机器人焊接 5 分钟搞定一台车的骨架。什么最贵？**设计新车型、判断这个新车型值不值得量产、下线以后客户用得爽不爽**——这些事机器人做不了，只有人做得了。

**事实二：上游零件升级越来越快，但产线吸收不过来。**
Anthropic、OpenAI 每个季度都在放出更强的零件（新模型）。**问题不是零件不够好，是产线跟不上**——新零件堆在仓库吃灰，就等于我们的竞争力天天缩水。

工程圈流传一条公式，说得很直白：

> **进步速度 = 迭代次数 × 每次迭代的进步幅度。**

把它翻译成聚星版：

> **聚星研发工厂的北极星目标，是让我们的产品化节奏，完全发挥模型的节奏。**

不是"加 AI 提效"那么小。是整个工厂组织方式的重构。

## 四、工厂怎么运转 —— 两个案例对比（Heavy / Standard）

**先看 Heavy 档**（全量 Process A 展开）：网红经纪人 0-to-1 调研的一周。
**再看 Standard 档**（日常迭代的压缩版）：Skills 收藏 feature 的 24 小时。

原则一样，尺度不同。看完两个案例你会明白：工厂的 SOP 骨架是 Heavy 档的完整形态，Standard 档是在这个骨架上做合理的压缩。

---

### 四A · Heavy 案例: 网红经纪人调研的一周

**网红经纪人**是聚星今年 P2 调研方向（主文档 §2.3 / §12.3）——全新产品线、0-to-1 探索、**结局未定**。这个案例里 Heavy 档必须 explicit 的 5 件事都会出现：pitch + spec + 标杆用例 + forbidden actions（§7.2）、原型海 8 并行（§7.3）、三层判断 + 止损（§7.4 / §7.5）、设计收口单线程（§7.6）、基础形态 5 角色 + 标杆用例回归（§5.3 / §7.7）。这些在 Standard 档会被压缩,Heavy 档必须都跑。

#### ⏰ 周一 09:00 · 立项决策 —— 杨洋 × 徐伸 共决（§12.3）

网红经纪人这个方向,过去几个月已经在杨洋和徐伸的战略讨论里出现过几次——三处迹象让它从 P2 调研档浮了上来:

- 合作过的腰部 KOL 偶尔在邮件里问"能不能给我们也开个自助入口"（Felix Rieseberg 所谓的"爬玻璃也要用"弱信号）
- 两家竞品在过去一个季度开始做 KOL-facing 产品,流量增速明显
- Q2 方向讨论里它作为 P2 被留了下来,但一直没立项

**上周五产品会议,杨洋和徐伸共同拍板**:这周做一次 Heavy 档快速探索,一周时间盒,结局 open——不押注 Q2 全量,只验证"值不值得做"。

> 📌 **Heavy 新产品线立项是 owner × Harness owner 共决**（主文档 §12.3:"产品线档位由该产品线 owner 和 Harness owner 共同判定"）。Heavy 会带动 harness 大量边界改动(新 skill 集 / 新 wiki / 新 Constrain),Harness owner 必须在立项阶段就在场,不能等 pitch 写完才报备。
>
> Heavy 0-to-1 阶段**没有客服信号源**——产品还不存在,用户没法提工单(对比 §四B 美娜的入料口)。所以信号来源是**战略判断 + 市场观察 + 内部讨论**的综合,由 owner + Harness owner 承担,不是客服聚合。

#### ⏰ 周一 10:30 · 立项 pitch —— 杨洋 品味 edit，agent 起草（§7.2 + §6.4）

杨洋花 1 小时完成 Heavy 档三件套——**不是手写,而是激活 CEO 模板 agent 起草 pitch、Doc 模板 agent 起草 spec**（§6.4:"起草 pitch / 归纳用户反馈 → 主要激活 CEO + Doc 模板"）。杨洋自己做的是**品味 edit 和最终 sign-off**,不亲手写文档：

**Pitch（一页）**：
- 问题：KOL 侧缺少自助产品,商务沟通靠邮件人力
- 面向谁：已合作过的腰部 KOL（头部有经纪人,尾部量太散）
- 最小成功信号：1 周内 50 个 KOL 触发 ≥ 2 次核心动作
- 时间盒：**2 周内发内测,第 4 周再决定要不要上全量**
- 不做什么：商家私信数据曝光、自动 KOL 评分

**Spec（薄,可机读）**：
- 输入：KOL 登录 → 看 offer 列表
- 输出：offer 状态（待接受 / 协商中 / 已签）+ 历史合作
- 边界：**不暴露其他 KOL 的竞品 offer**

**3 条标杆用例 + 2 条禁止动作（forbidden actions）**：
- ✓ 测 "登录 → 看 offer" 首屏走通
- ✓ 测 "拒绝 offer" 状态回写
- ✓ 测 "历史合作列表"
- ✗ 禁止：KOL agent 访问商家私信
- ✗ 禁止：自动给 KOL 打分 / 贴标签

> 📌 **Heavy 档的 pitch + spec + 标杆用例是强制产物**（§7.2）,不是可选。Standard 档可以 5 秒内嵌 pitch,Heavy 必须落成纸——因为要过下周 L1 评测员 agent 的自动筛。

#### ⏰ 周一 14:00 · Harness 边界评估 —— 徐伸（§7.2 Heavy 报备）

Heavy 档 pitch 必须向 Harness owner 报备。徐伸花 2 小时评估：

- 新产品线 → **需要新的 skill folder 集**（KOL-onboarding / KOL-offer-reply / KOL-data-boundary 等）
- 新用户类型 → **需要新的 wiki/concepts 页**（网红数据模型、KOL 通讯协议、腰部 KOL 特征）
- 工具边界 → **新的 Constrain 约束**（2 条 forbidden actions 必须工程上落到沙箱层面,不是只写进评分表）
- 旧 harness 怎么变 → **暂时新起一套 KOL-facing harness,先不和品牌端混**

徐伸评估完 push 一条回报："harness 改动中等,本周内起基座,不阻塞原型海。"

#### ⏰ 周二 全天 · 原型海（8 并行） —— 朱旭 + 杨芳 + 娜娜

朱旭和杨芳这周从各自的产线调过来支援 Heavy 调研。每人跑 4 条并行原型,共 8 条——对标主文档 §7.3 Heavy 起点 5-10。

每条原型由 3 个 agent 协作约 3 小时产出可运行 prototype：
- **CEO agent** 生成 pitch 变体（不同切入点：offer 中心 / 合作历史中心 / 消息中心 ...）
- **Eng Manager agent** 出每条的技术拆解
- **Staff Engineer agent** 落到可跑的半成品
- **Designer agent**（娜娜的那组）并行为 8 条生成 UI variants

**8 条并行的关键不是"都能做出来",是"用户真正看到什么"**。到周二下班,8 条都有了可点击 demo + 对应的评测分数。

> 📌 **对标 Anthropic 内部 100 原型并行**（主文档 §10.9 组织级 100+）。聚星起点 8 条,朱旭 + 杨芳各 4 条。这是 Felix 的"执行成本几乎为零"的直接落地——以前一个想法得排期 3 周,现在"给我 10 分钟给你一个版本"。

#### ⏰ 周三 10:00-17:00 · 三层判断 + 止损（§7.4 + §7.5）

8 条原型过三层筛。

**L1 自动筛（10:00-11:00）**：
评测员 agent 按 5 条标杆用例 + 2 条 forbidden actions 跑分。
- **3 条**踩了 forbidden action（2 条 leak 了商家私信、1 条自动给 KOL 打分）→ **止损**
- **5 条**过 L1 进入 L2

**L2 角色筛（14:00-15:30,并行独立）**：
- 杨洋（产品）review 5 条：方向是否站得住
- 娜娜（设计）review 5 条：体验面是否自然
- 朱旭（技术）review 5 条：架构是否有阻断

每人独立 read,写一段理由,**不开会**。

**L3 方向拍板（17:00,单线程 Write）**：
杨洋综合 3 人的评语,留下 **2 条 candidate**,3 条止损但留下结论。

> 📌 **止损不是失败**（§7.5）——这 3 条被砍的原型各自留了一条 Gotcha 进 skill folder：
> - "KOL-facing agent 默认不能访问 `wiki/concepts/商家私信.md`——常见错用"
> - "不要在 onboarding 里直接问 KOL '你的费率是多少'——会让 KOL 反感退出"
> - "自动打分看起来聪明,实际 50% KOL 觉得被评价,不如让他自己贴标签"
>
> 这些 Gotcha 下次任何 KOL-facing agent 激活都会自动加载。**止损产出的 harness +1% 不比成功小**。

#### ⏰ 周四 全天 · 设计收口（单线程 Write） —— 娜娜（§7.6）

**Heavy 档设计第一次变重头戏**。前 3 天娜娜都在并行 read,周四切进单线程 Write 模式,对剩下 2 条 candidate 做高频微迭代。

她关注的不是"按钮好不好看"：
- 信息层级：offer 列表按什么排序、状态怎么显示
- 关键文案：拒绝 offer 用"拒绝"还是"不合适"（信任感差异）
- 首屏结构：第一次打开看到什么,3 秒内能不能懂
- **信任元素**：KOL 是否确信"商家看不到我拒绝了多少"（隐私承诺显性化）
- Onboarding：第一次用要不要 tooltip 引导

娜娜让 Designer agent 生成**一次性调节工具**（滑块：offer 列表密度；对照面板：拒绝文案 A/B）,自己快速试。下午 4 点定稿,把规则写进 skill folder：
- "KOL-facing 首屏 3 秒内必须让用户看到 offer 数量和状态概览"
- "拒绝类动作用'不合适'不用'拒绝'——主观感受差 17%"

> 📌 **设计收口在全量工程化前**（§7.6）——agent 能把能做的做完,但**体验面的品味必须在 agent 动手前拍定**。Heavy 档这一步是**单线程、一整天、设计主导**；Standard 档是 15 分钟并行。差异在尺度,不在原则。

#### ⏰ 周五 全天 · 研发交付 + 标杆用例回归 —— 5 角色流水线（§5.3 + §7.7）

Heavy 档【当前默认·基础形态】：5 角色 agent 互审 + **强制标杆用例回归**。
- Eng Manager → Staff Engineer → Reviewer → Verify → Release

5 条标杆用例 + 2 条 forbidden actions 全过。Release 机器人按上线清单推内测灰度,**仅 50 个腰部 KOL 白名单**。

整天没一个人在合并前审过代码。

#### ⏰ 周末 · 50 KOL 试跑 —— 朱旭 + 杨芳 + 杨洋 轮值 on-call（§7.8）

真实 KOL 开始用。**Heavy 0-to-1 阶段没有现成的客服入口接反馈**,三人轮流 on-call——朱旭、杨芳对自己刚上线的系统保持 hot,杨洋做最终观察整理。production 面板设了自动告警,异常第一时间有人处理。

按主文档 §7.8 观察 4 点：
- **能不能顺利进入**：38/50 首次登录成功,12 个卡在短信验证 ← 朱旭周六发现并记录
- **结果是否稳定**：2 条 KOL 反映 offer 列表偶尔 3 秒才加载 ← 杨芳周六晚告警触发
- **协作是否顺**：20/50 尝试"和商家协商"但没找到入口 ← inline 反馈表汇总
- **是否愿意继续用**：周日前 27 个 KOL 触发 ≥ 2 次核心动作 ← production 埋点

周日晚上**杨洋汇总三人观察**,整理成一份 review 备忘,周一早上终评会用。

> 📌 Heavy 新产品线 0-to-1 阶段,监测是团队事——**不是 owner solo 盯**。研发 owner 对自己的上线系统保持 hot 是本能,需求 owner 做观察整理。主文档 §7.8 说"观察 4 点",这 4 点必须有人盯,但不必是一人。

#### ⏰ 下周一 09:30-14:00 · post-merge + 终评 + 结局 open

**杨洋 09:30 做上线后抽样**：抽 15 条 KOL 使用日志看,抽到了**一条边缘 forbidden action 违规**——某个 KOL agent 在推荐历史合作时,错误载入了商家私信的片段（L1 评测员没 catch,因为是边缘 case）。

**11:00 终评会议**（杨洋 + 徐伸 + 娜娜,不叫朱旭 / 杨芳——现在是判断级,不是执行级）：

摆上桌的三条路：
- **继续**：50 KOL 里 27 个真用了,信号 OK,下周把 forbidden 漏洞堵上,第 3 周考虑全量
- **重塑**：协商入口缺失是方向问题不是细节,回 §7.3 原型海重选
- **止损**：腰部 KOL 实际活跃度没到预期,这个方向再跑 2 周可能还是这样

讨论 1 小时,决定**继续**——但加一个硬条件：**下周再评一次,如果 50 KOL 的 2 周留存 < 40%,止损**。

> 📌 **结局 open 是 Heavy 调研档的特征**（§7.5 三选一：继续 / 重塑 / 止损）,不是"第一版一定要上"。主文档 §5.7："短周期判断,不做长周期空规划"——杨洋没说"第 4 季度前全量",他说"下周再决定一次"。这和 Felix 的"路线图 ≤ 1 个月"对上。

#### ⏰ 下周一 14:00-17:00 · harness sink（§7.9）

徐伸收周 review：这一周 harness 变了什么？
- **3 条新 skill folder** 合入全局索引：KOL-onboarding / KOL-offer-reply / KOL-data-boundary
- **2 条新 wiki/concepts 页**：`wiki/concepts/腰部 KOL 特征.md` / `wiki/concepts/KOL 通讯协议.md`
- **1 条新 forbidden action** 进评测员 rubric：边缘 case 里 KOL agent 不能引用商家私信片段（事后抽样发现的漏洞）
- **3 条止损的 Gotcha** 固化进 KOL-facing skill folder
- **娜娜 2 条设计规则** 进 Designer skill folder

**一周结束：一个决策 + 10 条 harness 改动**。主文档 §5.10 要求"每轮 +1%"——这一周 +10%。

### 四B · Standard 案例: Skills 收藏 24 小时

好,现在把镜头从 Heavy 切换到 Standard——**同样的 SOP 骨架,压缩到 24 小时的小迭代**。Heavy 档里那些 explicit 的步骤（pitch 成纸、原型海 8 并行、设计一整天收口）在 Standard 会被压缩：pitch 内嵌在杨洋 5 秒拍板里、CEO agent 3 方案替代原型海、设计 15 分钟并行 Eng Mgr。**原则一样,尺度不同**。

让我们跟着"用户要收藏达人"这条需求,看它在工厂里走完一圈。每一个工位上都有一个你认识的人。

#### ⏰ 14:07 · 入料口 —— 美娜的工位

**美娜是这座工厂最前沿的传感器。**

她不写代码、也不设计界面。她每天和 200 条用户工单打交道，把它们分类、去重、打标签，找出哪些是真需求、哪些是个例。

**这就是全渠道 kickoff 的第一步**——需求不是谁在群里喊一嗓子就进来，是有人把噪声收敛成信号。

美娜这次上报前，做了两件事：
1. 翻了上周工单，确认"收藏"关键词从 3 条跳到 12 条，**信号稳定**；
2. 在标记备注里写"Skills 用户为主，原型页触发，非个例"，**给下游节省 5 分钟甄别**。

她的"工位手册"里沉淀着这类收敛规则：**"同一诉求 < 5 条 = 噪音，5-10 条 = 观察，> 10 条 = 上报。"** 这条规则是她过去半年摔过几次跤沉淀下来的。

> 📌 美娜这套收敛规则固化进一个 skill folder（她负责维护，但放在全工厂共用的 skill 索引里，任何 agent 都能按 task 自动发现）。下次 **Doc agent** 要生成工单周报时，这条规则会被自动加载进它的上下文。

#### ⏰ 14:08 · 方向拍板 —— 杨洋的工位（需求 owner）

杨洋看到群消息，5 秒钟拍板。

杨洋做的**唯一一件事**是 L3 判断：
- 这事进哪个版本？**→ Skills 7.8.1。**
- 优先级？**→ P0，挤掉原本排在第 3 的"收藏夹排序"。**
- 边界？**→ 只做最基础的添加/移除，排序留下一轮。**

**杨洋不去想怎么实现、不去看代码。** L1（需求有没有意义）和 L2（怎么拆任务）不是他的工位——那是朱旭和他的 agent 要做的。

杨洋的活只有一个字：**挑**。挑哪个做、不做哪个、什么时候做。这是**品味的高杠杆位**。

> 📌 这对应 §7.4 的三层判断。杨洋这一段"版本 / 优先级 / 边界"**就是 Standard 档要求的最小 pitch 内容**（§7.2）——不用写成正式文档,但内容必须落下来,不能口头带过。你只要记住"杨洋管值不值得做，不管怎么做"就够了。

#### ⏰ 14:10 → 15:35 · 产品线一号工位 —— 朱旭开工

朱旭接到 @ 后，**没有打开 IDE**。他打开的是一条由 5 个机器人工位组成的产线：

**1. CEO 机器人（3 分钟）**
"'Skills 接入收藏'这个需求的边界是什么？可能有几种做法？哪种值得先跑？"
CEO agent 给了 3 方案。朱旭选了"复用现有收藏 API + Skills 侧加参数"——不换架构，当天能收口。

**2. Eng Manager 机器人（15 分钟）**
拆成 5 个子任务：参数解析、收藏夹 id 映射、防重复提交校验、错误码、埋点上报。朱旭改了 2 条放行。**这份 subtask 列表实际上就是 Standard 档"简化 spec"的作用（§7.2）——能机读、可机执行、和 pitch 对应得上**。

> 📌 **这 15 分钟同时,娜娜并行在做设计收口**（见下一段 14:13→14:28）。Eng Manager 拆的 subtask + 娜娜定的体验参数,两头的产物都会汇进 Staff Engineer 的执行上下文。

**3. Staff Engineer 机器人（1 小时，14:28→15:28）**
这个 agent 用**多疑型人格**（paranoid persona）——主动怀疑自己的实现、边写边质疑自己。**它同时吃到 Eng Mgr 拆好的 5 个子任务,以及娜娜留在 skill folder 里的设计约束（按钮位置、tooltip 延迟、onboarding 文案）**。

激活时它按任务路由匹配到 Skills 的 **skill folder**（操作协议），SKILL.md 里 `pre-check`（预检清单）字段列了两个必查的 **wiki 页**（事实本体）：`wiki/concepts/收藏API.md` 和 `wiki/concepts/账单系统.md`。两个 wiki 页自动被加载进上下文——第二个里第一句写着：

> "**防重复提交的那个字段（业界叫"幂等 key"）和账单系统绑死，修改前必须先出一份架构决策记录（ADR）评估影响。**"

**这条约束在 agent 动手前就是它的默认输入**。它没机会去尝试重写——直接走了"参数层加一层映射"的路，没碰接口本体。**朱旭全程不在场，杨芳全程不在场，群里没有一条消息**。

> 📌 **这是 skill × wiki 两层接力的胜利**，不是人的警觉，也不靠群里问同事。
> - **skill folder** 定"协议"（什么时候做、去查哪个 wiki 页、不做什么、做完验什么）——由 Harness owner 管
> - **wiki** 放"事实"（接口现状、依赖关系、谁维护过）——由领域 owner 管
> - agent 按协议自动拉事实，不需要人来中转
>
> 如果"账单系统绑防重字段"这条事实只存在某个人脑子里或某条群消息里，今天就是个事故。所以聚星在 §8.3 把 skill 和 wiki 严格分开：事实不许进 skill folder，否则 skill 会膨胀成事实库的冗余复制品，双边要改就漂了。

**角色模板（CEO/Eng Mgr/...）全厂共享，每条产线在此之上累积自己领域的 skill folder 和 wiki 页——但整个工厂共用一套 skill 索引，agent 按 task 能跨产线发现到对应的 skill folder，不靠人告诉它**。通用 × 特化——这是工厂能扩展到多产品线的关键。

#### ⏰ 14:13 → 14:28 · 体验收口工位（并行于 Eng Manager 拆任务）—— 娜娜

按主文档 §7.6，**设计收口在全量工程化之前，不是之后**。agent 能把能做的做完，但**体验面的品味判断必须在 agent 动手前拍定**——否则 Staff Engineer 会按自己的默认实现走,出来的东西"能用但不好用"。

所以娜娜的窗口是**朱旭启动 Eng Manager agent 那 15 分钟**。她在另一个工位上并行开工,看这个方向的 3 个体验问题：
- 收藏按钮放在达人卡片的哪个位置不挡信息？
- 第一次使用要不要 tooltip 引导？
- 跟 Skills 既有视觉语言协调吗？

娜娜之前在**插件 7.7.1** 上线时沉淀过一条规则：
> "新按钮第一次出现，tooltip 加 1.2 秒延迟——不然用户来不及注意到。"

她让 Designer agent 加载这条规则,生成设计稿——**第一版就对了**。关键参数（按钮位置、tooltip 延迟、onboarding 文案）**直接写进 skill folder**,Eng Manager agent 拆任务时自动 load 到,Staff Engineer 14:28 开工时自然吃到。

**娜娜和朱旭全程没开一次会**——skill folder 就是他们的异步协作接口。她的品味判断以"参数 + Gotcha"的形式进入 agent 的执行上下文,agent 按约束实现。

> 📌 **这是 §7.6 设计收口的 agent 时代落地**：设计师不是先画一周稿子再扔给研发,而是在 agent 拆任务的 **15 分钟窗口**里并行定体验面。设计的杠杆位从"画稿产出"迁到"**关键体验面的品味判断**"（§5.5 + §5.6）——execution 变便宜了,品味反而变贵了。

#### ⏰ 15:28 → 15:35 · 质检工位 —— Reviewer + Verify

**Reviewer 机器人（2 分钟）**：对照**评分表**（rubric）打分，报出 2 条 P1、3 条 P2 级别的问题。P3 的琐碎小毛病自动压掉。按规则它把反馈以"可接受 / 延后 / 反驳"三选一发回给 Staff Engineer agent；两个 agent 来回迭代 3 轮，全部收敛。
**Verify 机器人（5 分钟）**：把**正常使用流程**（happy path）端到端自动跑一遍，Skills 既有的**标杆用例**（golden cases）全过。

两个 agent 互相满意以后，Release 机器人按**上线清单**自动推一轮**灰度**（canary）。**整条合并链路里朱旭没看一眼代码，也没抽检任何 P1 问题**——那是 agent 之间自己收敛的事，不是人的杠杆位。

> 📌 **这就是"合并前不做人工审"（0 pre-merge human review）的落地**。人在合并前环节**不出现**。人的两个杠杆位在别处：**提案（pitch）阶段定评分表**、**事后合并上线后再抽样**（post-merge sampling，见下面次日场景，以及主文档 §5.3 / §8.5）。

#### ⏰ 次日凌晨 02:15 · 工装维护工位 —— 徐伸

**徐伸是这座工厂的工程师——Harness owner。**

他不管任何一条产线的代码。他管的是**工装、传送带、MES、质检站**——也就是所有产线**共用**的那套东西：harness 运行环境、CLAUDE.md 主规约、评测用例集、评分表模板、互审机制。

凌晨 02:15，harness 的自动日报给徐伸发了条告警：

> "Skills 产线今天的 Reviewer agent 误报 1 次：把一条正常实现判成 P1。"

徐伸打开 Skills 的评分表文件，加了一条误报样本，重新跑一遍评测，确认评分表被收敛回来。

**如果没有他，产线上的机器人会慢慢变笨。** 评分表会逐步漂移、skill folder 会无序膨胀、评测用例集会陈腐。

> 📌 这是为什么说 Harness owner 是"工厂里最需要懂工厂的人"——他不生产车，他让这座工厂能**持续生产**下去。按主文档 §6.2 的分阶段演进：**先兼任、再触发式升级为专职**——跨场景能力开始重复、兼任 owner 被业务挤走等信号出现时，自然升专。是条件触发，不是日历触发。

#### ⏰ 次日 09:30 · 合并上线后再抽样（post-merge sampling）—— 杨洋的另一侧杠杆位

昨天下午两个 agent 互相满意后自动合并、Release 机器人推了灰度，晚上跑得稳、夜里全量上线。**整条路径上没有人在合并前审过任何代码**。

次日早上杨洋打开**上线后抽样**面板，按 §5.3 的规则随机抽了 3 条昨天合并的代码看。"收藏"这条他皱了下眉：

> "收藏上限到了，只返错误码，没给友好提示。"

这不是程序 bug——Reviewer agent 按当前评分表判得没错。**但"用户会懵"属于 L3 的品味判断**，评分表里原来没有这一条。

杨洋**没去改那份已经上线的代码**（这一条不严重，重写成本不划算）。他做的是把"用户可见错误必须带友好提示"写进 Skills 的评分表。下次 Reviewer agent 再看到类似实现，自己会判 P1，自己会让 Staff Engineer 修。

> 📌 **这是 §8.5 "品味 → 评分表 → 评测"（taste → rubric → eval）管线的一次实操**：人的一次品味判断，固化成下次 agent 自己能用的规则。人不回到合并前做审查；人把自己的品味升级成组织资产。

#### ⏰ 次日 10:00 · 闭环 —— 美娜的工位，反馈回来了

回到美娜的工位。她翻当天的工单：
- 昨天：12 条"不能收藏"。
- 今天：**0 条**。
- 新冒出 2 条"收藏能不能按时间排序"——**新的、更细的信号**。

她按她的收敛规则打标（暂时归入观察）。

**这条需求的一生，从她开始，在她这里闭环合上。**

## 五、6 个工位在两档下的对比

同样 6 个人,Heavy 和 Standard 尺度下动作差异很大。这张表帮你把两个案例里同一个人做的事对上。

| 工位 / 人 | Heavy 档(网红经纪人一周) | Standard 档(Skills 收藏 24 小时) | 对应 SOP |
|---|---|---|---|
| 入料口 · **美娜** | **不参与**（0-to-1 探索无客服信号源,信号由 owner 战略判断承担） | 本周 12 条工单收敛成稳定信号;次日观察反馈闭环 | §7.1 全渠道 kickoff |
| 方向拍板 · **杨洋** | 立项 × 徐伸 共决(§12.3);CEO/Doc agent 起草 pitch+spec,杨洋品味 edit(§6.4);L2 产品 review + L3 拍板(关键时点);周末 on-call 轮值之一 + 周日整理 review 备忘;下周一 post-merge 抽样 + 终评 | 5 秒拍板 P0;次日做合并上线后抽样,把"用户会懵"升级进评分表 | §7.4 L3 + §7.8 + §8.5 |
| 产品线一号 · **朱旭** | 跨过来 run 4 条原型(8 并行的一半)+ L2 技术 review + 周末 on-call 轮值 | 独立激活 5 agent 流水线,agent 互审收敛自动合并 | §7.3 原型海 + §5.3 执行规程 |
| 产品线二号 · **杨芳** | 跨过来 run 另 4 条原型 + 周末 on-call 轮值 | 跑邮件邀约;她过去沉淀到 wiki 的事实今天仍服务朱旭 agent(无需群里问人) | §7.3 原型海 + §8.3 skill × wiki |
| 体验收口 · **娜娜** | 原型海并行设计 + 整天单线程收口 + L2 筛 | 15 分钟并行 Eng Manager 定体验面,参数进 skill folder 由 agent 自动吃到 | §7.6 设计收口 + §5.6 体验信任 + §8.5 品味沉淀 |
| 工装维护 · **徐伸** | 在 pitch 阶段就介入边界评估;周末 harness sink(3 skill + 2 wiki + 1 forbidden) | 凌晨收告警,升级 Skills 的评分表 | §6.2 Harness ownership + §8 |

**一张骨架,两种尺度**。Heavy 档所有环节 explicit;Standard 档把不痛的环节压缩掉,但 5 个杠杆位——**pitch(有内容即可) / 原型 / 三层判断 / 设计收口 / 合并上线后抽样**——一个都不能省。

**两档都一样的铁律**：没有一个工位在"审所有代码"。人只做评分表定义 + 事后抽样,agent 互审到满意就自动合并。

## 六、三色规则标签 —— 教你怎么读文档

执行规程里所有数字、规则，都会带一个三色标签。你看到时不要慌，按标签读就行：

- 🟨 **【目标态】** = 工厂升级到这里就爽，但现在到不了。是年底要达到的状态。
- 🟢 **【当前默认】** = 今天就能按这个做，什么都不用额外配。
- 🟦 **【试运行例外】** = 现阶段先在试验场小跑，跑顺了再向前收。

举 3 个例子：

| 话题 | 🟢 当前默认 | 🟦 试运行例外 | 🟨 目标态 |
|---|---|---|---|
| 并行原型数 | 入门 3-5 / 合格 5-8 | 第一次至少 3 个 | 熟手 8-15 个 |
| Agent 互审 | Claude Code 内置互审 | 升级到结构化评分表 | 7 角色全链路 |
| 全渠道 kickoff | 已有渠道保持 | 试开 1 个新渠道 | 飞书/Linear/Gmail 全渠道 |

**认错标签是最常见的误读来源**。别人说"我们现在应该 X"时，先问一句：这是 🟨🟢🟦 哪个？

## 七、你手头的项目属于哪一档

也是三色，不过这次是产品线的节奏分档：

| 档位 | 特征 | 聚星真实例子 | 当前默认切入深度 |
|---|---|---|---|
| 🟢 **Light** | 小 bug 修复 / 文档 / 单人独立 | 插件 7.7.1 文档 · TTShop 开户提审 | 一段提案（pitch）+ §5.3 基础互审即可，其他机制不强制 |
| 🟡 **Standard** | 已上线模块增量 · 多角色 · ≤ 2 周 | **消消乐 XXL0312 多邮箱**（已上线） · Skills 7.8.1 | + 提案 / 规格书 + 至少 3-5 并行原型 + 每周双复盘 |
| 🔴 **Heavy** | 新产品线 · 用户可见大改版 | agent 主线 · 网红经纪人 · 老平台 Skill 化迁移 | Process A + B 全量 + 标杆用例回归；【目标态】向 7 角色流水线靠 |

**XXL0312 是一个跑完整轮的完整案例**：2026-03-11 立项，2026-04-10 上线（多 Gmail、多企业邮箱、邮箱池轮发）。29 天内完成从立项到上线。这是目前已经稳定的 Standard 节奏。

Skills 7.8.1（也就是你上面看到的那条收藏需求）是正在进行的 Standard 项目。Agent 主线是未来的 Heavy 项目——产品设计还在草稿（01+02），还没到交付阶段。
