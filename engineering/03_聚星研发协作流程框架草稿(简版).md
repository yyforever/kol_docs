## 一、周三下午 14:07，一条工单进了群

**2026 年 4 月的某个周三下午。**

美娜在客服后台看到当天第 7 条类似的留言：

> "用 Agent 跑完之后，我想把这几个达人单独留着，为什么没有收藏啊？"

她翻了一下前两周的工单——上周才 3 条，这周累计 12 条。**信号稳定了**。她把这批留言汇总成一段话，扔进产品群：

> "本周 12 个用户问收藏功能，重复率高、描述一致，不是噪音。"

三秒后，杨洋在群里 @了朱旭：

> "Skills 7.8.1 加一条 P0，收藏达人，下周能给方案吗？"

朱旭 30 秒回复：

> "能，下午给雏形。"

—— 如果你看完这段觉得平平无奇，那就对了。**一条来自一线用户的抱怨，被收敛、被拍板、被派活，聚星走完整个链条不到一分钟。**

两年前做同样的事，我们要多久？**一天起步。**

剩下的"实现"呢？朱旭那个下午没打开 IDE 写一行代码。他做的是**激活一整条由 agent 组成的"机器人工位流水线"**，自己只做**品味指导**（选方案、改子任务），其余交给 agent 之间自己互审收敛到满意、自动合并上线。**整条合并链路他一眼都没看过代码**——代码审查已经不是人的工位。

这就是这篇导读想告诉你的：**聚星研发中心已经不是 2023 年那个手工作坊。它正在变成一座工厂。**

## 二、如果你没来过聚星研发中心，它长这样

类比一下：**聚星研发中心 = 一座全自动的超级工厂**。

想象那种你在纪录片里见过的、最现代化的汽车或者电子产品工厂——里面有 200 多种不同专业的机器人，它们自己焊接、自己涂装、自己装电池、自己下线测试。人做的不是拧螺丝，而是**设计工位、维护机器人、抽检成品、判断要不要造新车型**。

把聚星往里头套：

| 工厂组件 | 聚星对应 |
|---|---|
| 厂区 | 聚星研发中心（你们办公室） |
| 产线（每条产一款车型） | 产品线（Skills / 消消乐 / Agent 主线 / 插件 ……） |
| 原料进货口 | 用户反馈、市场信号、运营输入 |
| 专业机器人（200 种） | **9 种 agent 角色模板**（CEO / Eng Mgr / Staff Engineer / Designer / QA / Verify / Release / Reviewer / Doc） |
| 工位操作协议（SOP） | **skill folder**（每个 agent 的"工位手册"：何时做、先查什么、不做什么、做完验什么） |
| 工厂档案室 / 产品知识库（事实） | **wiki**（Karpathy 式 Global-Brain：收藏 API 是什么、账单系统怎么结构、谁维护过） |
| 工厂的传送带、夹具、MES | **harness**（agent 运行环境 + CLAUDE.md + eval 集） |
| 质检站 | agent 自动互审 + 评测员按评分表（rubric）打分 |
| 门店反馈抽检 | 合并上线后再抽样（post-merge sampling · 看线上用户真实反馈） |
| 工位负责人 | 杨洋 / 朱旭 / 杨芳 / 娜娜 / 美娜 |
| 工厂工程师（管工装设备） | **Harness owner（徐伸）** |

> 📌 这 9 个模板里，**代码流水线走其中 7 个**（CEO / Eng Mgr / Staff Engineer / Reviewer / Verify / QA / Release）。Designer 和 Doc 是另外两条专用线——主文档 §5.3 里说的"7 角色流水线"就是这 7 个。

工厂圈有一句老话：**"It's the machine that builds machines."**
工厂本身是一台"造机器的机器"。聚星研发中心也一样：**agent 在生产产品，Harness 在生产 agent 的工作环境**。

## 三、为什么工厂要这样改

两个事实，一句话就能说明白。

**事实一：焊接（写代码）这件事变不要钱了。**
过去工厂 80% 的成本在人工焊接。现在机器人焊接 5 分钟搞定一台车的骨架。什么最贵？**设计新车型、判断这个新车型值不值得量产、下线以后客户用得爽不爽**——这些事机器人做不了，只有人做得了。

**事实二：上游零件升级越来越快，但产线吸收不过来。**
Anthropic、OpenAI 每个季度都在放出更强的零件（新模型）。**问题不是零件不够好，是产线跟不上**——新零件堆在仓库吃灰，就等于我们的竞争力天天缩水。

工程圈流传一条公式，说得很直白：

> **进步速度 = 迭代次数 × 每次迭代的进步幅度。**

把它翻译成聚星版：

> **聚星研发工厂的北极星目标，是让我们的产品化节奏，完全发挥模型的节奏。**

不是"加 AI 提效"那么小。是整个工厂组织方式的重构。

## 四、一条需求在工厂里的 24 小时

好，现在把镜头拉回周三下午那条客服留言。**让我们跟着"用户要收藏达人"这条需求，看它在工厂里走完一圈。每一个工位上都有一个你认识的人。**

### ⏰ 14:07 · 入料口 —— 美娜的工位

**美娜是这座工厂最前沿的传感器。**

她不写代码、也不设计界面。她每天和 200 条用户工单打交道，把它们分类、去重、打标签，找出哪些是真需求、哪些是个例。

**这就是全渠道 kickoff 的第一步**——需求不是谁在群里喊一嗓子就进来，是有人把噪声收敛成信号。

美娜这次上报前，做了两件事：
1. 翻了上周工单，确认"收藏"关键词从 3 条跳到 12 条，**信号稳定**；
2. 在标记备注里写"Skills 用户为主，原型页触发，非个例"，**给下游节省 5 分钟甄别**。

她的"工位手册"里沉淀着这类收敛规则：**"同一诉求 < 5 条 = 噪音，5-10 条 = 观察，> 10 条 = 上报。"** 这条规则是她过去半年摔过几次跤沉淀下来的。

> 📌 美娜这套收敛规则固化进一个 skill folder（她负责维护，但放在全工厂共用的 skill 索引里，任何 agent 都能按 task 自动发现）。下次 **Doc agent** 要生成工单周报时，这条规则会被自动加载进它的上下文。

### ⏰ 14:08 · 方向拍板 —— 杨洋的工位（需求 owner）

杨洋看到群消息，5 秒钟拍板。

杨洋做的**唯一一件事**是 L3 判断：
- 这事进哪个版本？**→ Skills 7.8.1。**
- 优先级？**→ P0，挤掉原本排在第 3 的"收藏夹排序"。**
- 边界？**→ 只做最基础的添加/移除，排序留下一轮。**

**杨洋不去想怎么实现、不去看代码。** L1（需求有没有意义）和 L2（怎么拆任务）不是他的工位——那是朱旭和他的 agent 要做的。

杨洋的活只有一个字：**挑**。挑哪个做、不做哪个、什么时候做。这是**品味的高杠杆位**。

> 📌 这对应 §7.4 的三层判断。你只要记住"杨洋管值不值得做，不管怎么做"就够了。

### ⏰ 14:10 → 15:35 · 产品线一号工位 —— 朱旭开工

朱旭接到 @ 后，**没有打开 IDE**。他打开的是一条由 5 个机器人工位组成的产线：

**1. CEO 机器人（3 分钟）**
"'Skills 接入收藏'这个需求的边界是什么？可能有几种做法？哪种值得先跑？"
CEO agent 给了 3 方案。朱旭选了"复用现有收藏 API + Skills 侧加参数"——不换架构，当天能收口。

**2. Eng Manager 机器人（15 分钟）**
拆成 5 个子任务：参数解析、收藏夹 id 映射、防重复提交校验、错误码、埋点上报。朱旭改了 2 条放行。

**3. Staff Engineer 机器人（1 小时）**
这个 agent 用**多疑型人格**（paranoid persona）——主动怀疑自己的实现、边写边质疑自己。

激活时它按任务路由匹配到 Skills 的 **skill folder**（操作协议），SKILL.md 里 `pre-check`（预检清单）字段列了两个必查的 **wiki 页**（事实本体）：`wiki/concepts/收藏API.md` 和 `wiki/concepts/账单系统.md`。两个 wiki 页自动被加载进上下文——第二个里第一句写着：

> "**防重复提交的那个字段（业界叫"幂等 key"）和账单系统绑死，修改前必须先出一份架构决策记录（ADR）评估影响。**"

**这条约束在 agent 动手前就是它的默认输入**。它没机会去尝试重写——直接走了"参数层加一层映射"的路，没碰接口本体。**朱旭全程不在场，杨芳全程不在场，群里没有一条消息**。

> 📌 **这是 skill × wiki 两层接力的胜利**，不是人的警觉，也不靠群里问同事。
> - **skill folder** 定"协议"（什么时候做、去查哪个 wiki 页、不做什么、做完验什么）——由 Harness owner 管
> - **wiki** 放"事实"（接口现状、依赖关系、谁维护过）——由领域 owner 管
> - agent 按协议自动拉事实，不需要人来中转
>
> 如果"账单系统绑防重字段"这条事实只存在某个人脑子里或某条群消息里，今天就是个事故。所以聚星在 §8.3 把 skill 和 wiki 严格分开：事实不许进 skill folder，否则 skill 会膨胀成事实库的冗余复制品，双边要改就漂了。

**角色模板（CEO/Eng Mgr/...）全厂共享，每条产线在此之上累积自己领域的 skill folder 和 wiki 页——但整个工厂共用一套 skill 索引，agent 按 task 能跨产线发现到对应的 skill folder，不靠人告诉它**。通用 × 特化——这是工厂能扩展到多产品线的关键。

### ⏰ 15:28 → 15:35 · 质检工位 —— Reviewer + Verify

**Reviewer 机器人（2 分钟）**：对照**评分表**（rubric）打分，报出 2 条 P1、3 条 P2 级别的问题。P3 的琐碎小毛病自动压掉。按规则它把反馈以"可接受 / 延后 / 反驳"三选一发回给 Staff Engineer agent；两个 agent 来回迭代 3 轮，全部收敛。
**Verify 机器人（5 分钟）**：把**正常使用流程**（happy path）端到端自动跑一遍，Skills 既有的**标杆用例**（golden cases）全过。

两个 agent 互相满意以后，Release 机器人按**上线清单**自动推一轮**灰度**（canary）。**整条合并链路里朱旭没看一眼代码，也没抽检任何 P1 问题**——那是 agent 之间自己收敛的事，不是人的杠杆位。

> 📌 **这就是"合并前不做人工审"（0 pre-merge human review）的落地**。人在合并前环节**不出现**。人的两个杠杆位在别处：**提案（pitch）阶段定评分表**、**事后合并上线后再抽样**（post-merge sampling，见下面次日场景，以及主文档 §5.3 / §8.5）。

### ⏰ 15:45 → 16:30 · 体验打磨工位 —— 娜娜

朱旭在产线上跑机器人的同时，**娜娜**（设计师）在另一个工位上做她的事。

她没看代码，也不问朱旭进度。她看的是：
- 收藏按钮放在原型卡片的哪个位置不挡信息？
- 第一次用这个按钮的用户，要不要引导？
- 跟 Skills 现有的视觉语言协调吗？

娜娜之前在**插件 7.7.1** 上线时沉淀过一条规则：
> "新按钮第一次出现，tooltip 加 1.2 秒延迟——不然用户来不及注意到。"

这条规则她**固化进了一个 skill folder**（她负责维护，放在全工厂共用的 skill 索引里）。今天她让 Designer agent 自动加载这条规则，生成的设计稿**第一版就对了**，不用来回改。

**娜娜跟朱旭几乎没开会**。**skill folder 就是跨工位的异步协作接口**——有人写进去，别人的 agent 按索引自动读出来，各自靠 agent 落地。

> 📌 这是工厂里最隐形、但最值钱的一种配合：**每个人沉淀下来的东西，都会变成别人 agent 的输入**。

### ⏰ 次日凌晨 02:15 · 工装维护工位 —— 徐伸

**徐伸是这座工厂的工程师——Harness owner。**

他不管任何一条产线的代码。他管的是**工装、传送带、MES、质检站**——也就是所有产线**共用**的那套东西：harness 运行环境、CLAUDE.md 主规约、评测用例集、评分表模板、互审机制。

凌晨 02:15，harness 的自动日报给徐伸发了条告警：

> "Skills 产线今天的 Reviewer agent 误报 1 次：把一条正常实现判成 P1。"

徐伸打开 Skills 的评分表文件，加了一条误报样本，重新跑一遍评测，确认评分表被收敛回来。

**如果没有他，产线上的机器人会慢慢变笨。** 评分表会逐步漂移、skill folder 会无序膨胀、评测用例集会陈腐。

> 📌 这是为什么说 Harness owner 是"工厂里最需要懂工厂的人"——他不生产车，他让这座工厂能**持续生产**下去。按主文档 §6.2 的分阶段演进：**先兼任、再触发式升级为专职**——跨场景能力开始重复、兼任 owner 被业务挤走等信号出现时，自然升专。是条件触发，不是日历触发。

### ⏰ 次日 09:30 · 合并上线后再抽样（post-merge sampling）—— 杨洋的另一侧杠杆位

昨天下午两个 agent 互相满意后自动合并、Release 机器人推了灰度，晚上跑得稳、夜里全量上线。**整条路径上没有人在合并前审过任何代码**。

次日早上杨洋打开**上线后抽样**面板，按 §5.3 的规则随机抽了 3 条昨天合并的代码看。"收藏"这条他皱了下眉：

> "收藏上限到了，只返错误码，没给友好提示。"

这不是程序 bug——Reviewer agent 按当前评分表判得没错。**但"用户会懵"属于 L3 的品味判断**，评分表里原来没有这一条。

杨洋**没去改那份已经上线的代码**（这一条不严重，重写成本不划算）。他做的是把"用户可见错误必须带友好提示"写进 Skills 的评分表。下次 Reviewer agent 再看到类似实现，自己会判 P1，自己会让 Staff Engineer 修。

> 📌 **这是 §8.5 "品味 → 评分表 → 评测"（taste → rubric → eval）管线的一次实操**：人的一次品味判断，固化成下次 agent 自己能用的规则。人不回到合并前做审查；人把自己的品味升级成组织资产。

### ⏰ 次日 10:00 · 闭环 —— 美娜的工位，反馈回来了

回到美娜的工位。她翻当天的工单：
- 昨天：12 条"不能收藏"。
- 今天：**0 条**。
- 新冒出 2 条"收藏能不能按时间排序"——**新的、更细的信号**。

她按她的收敛规则打标（暂时归入观察）。

**这条需求的一生，从她开始，在她这里闭环合上。**

## 六、三色规则标签 —— 教你怎么读文档

执行规程里所有数字、规则，都会带一个三色标签。你看到时不要慌，按标签读就行：

- 🟨 **【目标态】** = 工厂升级到这里就爽，但现在到不了。是年底要达到的状态。
- 🟢 **【当前默认】** = 今天就能按这个做，什么都不用额外配。
- 🟦 **【试运行例外】** = 现阶段先在试验场小跑，跑顺了再向前收。

举 3 个例子：

| 话题 | 🟢 当前默认 | 🟦 试运行例外 | 🟨 目标态 |
|---|---|---|---|
| 并行原型数 | 入门 3-5 / 合格 5-8 | 第一次至少 3 个 | 熟手 8-15 个 |
| Agent 互审 | Claude Code 内置互审 | 升级到结构化评分表 | 7 角色全链路 |
| 全渠道 kickoff | 已有渠道保持 | 试开 1 个新渠道 | 飞书/Linear/Gmail 全渠道 |

**认错标签是最常见的误读来源**。别人说"我们现在应该 X"时，先问一句：这是 🟨🟢🟦 哪个？

## 七、你手头的项目属于哪一档

也是三色，不过这次是产品线的节奏分档：

| 档位 | 特征 | 聚星真实例子 | 当前默认切入深度 |
|---|---|---|---|
| 🟢 **Light** | 小 bug 修复 / 文档 / 单人独立 | 插件 7.7.1 文档 · TTShop 开户提审 | 一段提案（pitch）+ §5.3 基础互审即可，其他机制不强制 |
| 🟡 **Standard** | 已上线模块增量 · 多角色 · ≤ 2 周 | **消消乐 XXL0312 多邮箱**（已上线） · Skills 7.8.1 | + 提案 / 规格书 + 至少 3-5 并行原型 + 每周双复盘 |
| 🔴 **Heavy** | 新产品线 · 用户可见大改版 | agent 主线 · 网红经纪人 · 老平台 Skill 化迁移 | Process A + B 全量 + 标杆用例回归；【目标态】向 7 角色流水线靠 |
