## 1. 已对齐设计口径

第一版是独立 Web 产品，核心形态仍然是：

```text
AI 商单经纪人 / 商单工作台
```

产品不把“商单数量”当核心价值，而是帮助达人把真实机会推进成可判断、可报价、可交付、可收款的合作。

### 1.1 三个顶层模块

```text
发现商单
我的商单 / 工作台
社区
```

- 发现商单：达人主动找可申请机会，路径包括精选、推荐、搜索。
- 工作台：处理自己的商单流程，包括开发、谈判、合作、结算和历史记录。
- 社区：看接单经验、案例复盘、报价和沟通方法，不直接处理商单。

### 1.2 工作台的三段互斥流程

工作台里最重要的业务阶段是互斥的，不允许同一条商单同时属于开发和谈判。

| 阶段 | 进入条件 | 用户要解决的问题 | 主要动作 | 退出条件 |
|---|---|---|---|---|
| 商单开发 | 只有一方表达合作意向，还没有形成双方来回沟通 | 这是不是值得打开的机会？我要不要申请/回复？ | 回复邀约、申请商单、补资料、收藏、不感兴趣 | 双方至少各有一封消息后进入商单谈判 |
| 商单谈判 | 双方至少各有一封消息，正在讨论预算、交付、rights、付款等条件 | 怎么把机会谈成条件清楚、价格合理、风险可控的合作？ | 追问、报价、议价、确认条款、不感兴趣 | 双方确认合作后进入合作商单 |
| 合作商单 | 价格和关键条款已经确认，合作成立 | 怎么按时交付、减少返工、记录收款？ | 管理交付、审核、发布、收款、归档 | 完成结算和沉淀案例 |

消息方向规则：

```text
广告主邀约，达人未回复                 -> 商单开发
达人申请/主动 pitch，广告主未回复       -> 商单开发
广告主邀约 + 达人回复                  -> 商单谈判
达人申请/主动 pitch + 广告主回复        -> 商单谈判
双方确认价格和关键条款                  -> 合作商单
```

公开 Offer 的特殊规则：

```text
系统识别公开 Offer 不等于广告主已经邀约达人。
达人提交申请后，如果广告主还没回复，仍然属于商单开发。
```

负向处理规则：

```text
开发中和谈判中的商单都可以标记为“不感兴趣”。
默认动作是忽略 / 归档 / 优化识别规则，不默认展示或发送“拒绝合作”回复。
```

### 4.1 商单开发模块

展示“单向机会”，不展示报价回应率。

信息优先级：

```text
主数字：开发中的机会数量
解释：还没聊起来，先判断要不要回复或申请
辅助：一行展示邀约待回复 / 申请等品牌；来源只做轻量提示
具体推荐：放在下方“推荐先处理”列表，不在卡片里重复展示
入口：处理邀约 -> 全部商单
```

空态：

```text
商单开发
暂无待打开机会

同步 Gmail 或去发现商单后，系统会把可申请、可回复的机会放到这里。
[同步 Gmail] [去发现商单]
```

### 4.2 商单谈判模块

展示“双向沟通”，不展示今天识别了多少商单。

信息优先级：

```text
主数字：谈判中的商单数量
解释：已经聊起来，重点确认价格、交付和付款
辅助：一行展示待你回复 / 等品牌
报价回应率：作为侧边栏或分析指标，不放在谈判卡片里和“待推进”混在一起
入口：继续谈判 -> 沟通中
```

空态：

```text
商单谈判
还没有谈判中的商单

当品牌回复你的申请，或你回复品牌邀约后，这里会跟踪报价、还价和条款确认。
[处理开发机会]
```

### 4.3 本月接单金额

收入口径：

```text
接单金额 = 已进入合作的现金 fee
不含样品、佣金、未确认报价
收款记录 = 用户手动记录的已到账金额
```

文案避免使用“已确认收入”这种财务含义强、容易误解为到账核验的表达。

### 5.1 开发阶段的二级动作

点击立即联系后，系统根据来源和信息完整度推荐动作：

```text
品牌邀约待回复 -> 追问信息 或 发送报价
公开 Offer / 发现商单申请 -> 申请商单
红人主动 pitch -> 发送 pitch / 补资料
```

一旦形成双方至少各一封消息，商单进入“商单谈判”。

## 6. 沟通中 / 商单谈判

沟通中只展示已经形成双向沟通的商单。

```text
+--------------------------------------------------------------------------+
| 沟通中                                                                   |
+--------------------------------------------------------------------------+
| 已通过平台发送追问或报价的商单 · Gmail 3                                  |
| [全部判断] [建议联系] [谨慎] [不建议]                                     |
|                                                                          |
| +---------------------------------------+ +-----------------------------+ |
| | 4399 Game Launch UA Video             | | NovaSkin Beauty Device       | |
| | 阶段: 等品牌确认                      | | 阶段: 品牌返回报价           | |
| | 已报价: $3,000                        | | 我方 $600 / 品牌返回 $450    | |
| | 下一步: 48 小时后 follow-up           | | 下一步: 拆分 Spark Ads 报价   | |
| | [立即联系] [不感兴趣] [等待品牌回复]  | | [立即联系] [不感兴趣] [继续谈] |
| +---------------------------------------+ +-----------------------------+ |
+--------------------------------------------------------------------------+
```

谈判阶段的核心信息：

```text
对方最后说了什么
我方已给过什么价格/条件
当前卡在哪个条款：预算、交付物、rights、付款、时间
下一步是谁处理：你 / 品牌
是否命中核心指标：品牌接受报价或返回报价
```

谈判阶段动作：

```text
追问 / 发送报价 / 继续谈 / follow-up / 不感兴趣
```

说明：

```text
谈判中标记“不感兴趣”代表达人选择忽略该机会或不再继续推进。
默认不展示“拒绝合作”回复，不默认给品牌发送拒绝邮件。
```

## 9. 发现商单

发现商单是供达人主动申请的入口，申请后进入“我的商单 / 商单开发”。

```text
+--------------------------------------------------------------------------+
| 发现商单                                                                 |
+--------------------------------------------------------------------------+
| [精选] [推荐] [搜索]                                                      |
|                                                                          |
| 精选：平台人工挑选或高可信商单                                           |
| 推荐：基于账号数据和偏好推荐                                             |
| 搜索：达人按品牌、品类、平台、预算搜索                                   |
|                                                                          |
| 推荐和搜索共用筛选条件：平台 / 品类 / 报酬 / 样品 / 地区 / rights 风险    |
+--------------------------------------------------------------------------+
```

申请流程：

```text
达人查看商单
-> 复用创作者资料：频道介绍、账号数据、代表案例、联系方式
-> 补本次个性化打招呼信息
-> 提交申请
-> 进入工作台的商单开发：已申请待回复
```

未连接社媒账号时，可以看到基础摘要，但完整预算、申请竞争强度、匹配解释和报价依据保持受限。

## 11. 收藏与不感兴趣

### 11.1 收藏

收藏是“稍后再看”，不是活跃流程。

```text
+--------------------------------------------------------------------------+
| 收藏                                                                     |
+--------------------------------------------------------------------------+
| Phone Case TikTok Campaign                                                |
| 收藏原因: 信息不完整 / 等预算 / 等账号数据                                |
| [立即联系] [不感兴趣] [编辑备注]                                          |
+--------------------------------------------------------------------------+
```

### 11.2 不感兴趣

不感兴趣是负向处理，不是邮件拒绝流程。

```text
+--------------------------------------------------------------------------+
| 不感兴趣                                                                 |
+--------------------------------------------------------------------------+
| Unknown Brand Free Product Collab                                         |
| 原因: [x] 只给样品  [ ] 不匹配  [ ] 预算低  [ ] 像诈骗  [ ] 其他          |
|                                                                          |
| [恢复到全部商单]                                                          |
+--------------------------------------------------------------------------+
```

可选原因用于后续优化识别规则：

```text
来源质量
预算质量
品类匹配
rights / 付款风险
误识别
```

默认不提供“发送拒绝”主按钮。后续如果要支持礼貌拒绝，也应作为手动二级能力，而不是忽略的默认动作。

## 12. 自动化边界

默认确认后发送：

```text
首次联系品牌
申请商单
发送报价
接受关键条款
涉及价格 / 付款 / rights / whitelisting / raw files 的回复
```

可设置自动：

```text
低风险 follow-up
简单确认收到
规则明确的状态提醒
```

可自动处理：

```text
识别商单
提取字段
识别风险
生成草稿
提醒 follow-up
归档状态
更新记录
```

忽略 / 不感兴趣：

```text
可以自动从活跃流程移除并记录原因。
不默认向品牌发送拒绝消息。
```

## 13. 核心状态草案

```text
来源进入
  |
  +-- Gmail 邀约 / 聚星邀约 / 公开 Offer / 发现商单
        |
        v
商单开发（单向意向，互斥阶段）
  |
  +-- 邀约待回复
  |     brand -> creator
  |
  +-- 已申请待回复
  |     creator -> brand
  |
  +-- 收藏
  |
  +-- 不感兴趣
  |
  +-- 双方至少各有一封消息
        |
        v
商单谈判（双向沟通，互斥阶段）
  |
  +-- 待你回复
  |
  +-- 等品牌确认
  |
  +-- 品牌返回报价 / counteroffer     <- 核心指标命中
  |
  +-- 品牌接受报价                   <- 核心指标命中
  |
  +-- 不感兴趣（忽略，不默认发拒绝）
  |
  +-- 双方确认价格和关键条款
        |
        v
合作商单
  |
  +-- 内容制作
  |
  +-- 品牌审核
  |
  +-- 待发布 / 已发布
  |
  +-- 待结算
  |
  +-- 已完成
```

约束：

```text
商单开发和商单谈判必须互斥。
收藏和不感兴趣不计入活跃开发/谈判。
合作商单不再计入商单谈判。
接单金额只统计已进入合作的现金 fee。
```

## 14. 可测性与验证口径

设计落代码时必须能被测试验证：

```text
1. 阶段归类函数可以独立单测：
   - 单向品牌邀约 -> development
   - 单向达人申请 -> development
   - 双方各有一封消息 -> negotiation
   - 收藏 / 不感兴趣 -> inactive
   - 确认合作 -> collaboration

2. 工作台概览组件测试：
   - 商单开发只展示主数量、轻量状态和来源提示
   - 商单谈判只展示主数量、待你回复 / 等品牌
   - 推荐商单列表承接具体优先处理，不在 KPI 内重复
   - 来源提示属于开发模块，不属于谈判模块

3. 交互测试：
   - 处理邀约进入全部商单
   - 继续谈判进入沟通中
   - 开发中和谈判中的商单都可以标记不感兴趣
   - 不感兴趣不会触发拒绝邮件编辑器

4. E2E 验证：
   - 概览首屏三张指标卡顺序正确
   - 沟通中卡片可继续联系，也可不感兴趣
   - 合作中 / 待结算不回流到谈判指标
```
