# Postiz 用户路径体验地图与聚星优化方案

日期：2026-06-23
更新：2026-06-24，一致性清理版
目的：用 Postiz 作为主参考，优化聚星 Skill 及相关入口的新用户路径，提高首次激活、复用和付费转化。

## 1. 核心结论

聚星要优化的不是“让用户知道我们有 Skill / MCP / API”，而是让新用户尽快完成一个可信的业务闭环：

```text
理解 AI 能推进哪类达人营销工作
-> 完成账号连接和自检
-> 跑出第一个低风险业务结果
-> 通过直达链接回到聚星 SaaS 对象继续管理
-> 在对象价值和额度消耗清楚后产生复用和付费理由
```

Postiz 最值得借鉴的是产品结构：

```text
AI runtime -> SaaS workspace -> first task -> SaaS object -> repeat / pay
```

聚星不能照搬 Postiz。Postiz 的核心业务是社媒排程，首次任务可以接近“创建 post”；聚星 Skill 覆盖达人发现、分析、建联、谈判、监控和运营对象管理，首次任务应优先是 read / analyze / draft，不应是 send、schedule、批量导出等高风险动作。

一句话方案：

```text
把 /skills 从“工具入口页”升级为“AI 达人营销启动页”：
首屏卖完整达人营销链路，Quick Start 按真实 Skill/CLI 跑通第一个结果，Skill 输出必须带回聚星 SaaS 的直达链接。
```

建议 Hero：

```text
H1：让 AI 7×24 推进达人营销全链路
Subcopy：从达人发现、分析、建联谈判到效果监控，聚星把数据、流程和结果沉淀回 SaaS 工作台。
```

备选更稳妥版本：

```text
H1：让 AI 持续推进达人营销全链路
Subcopy：发现达人、判断合作价值、准备建联动作，并把结果沉淀回聚星工作台。
```

## 2. Postiz 能证明什么

### 2.1 可借鉴点

Postiz 首页先卖业务结果，而不是先卖协议。已验证首屏主文案是 `Run your social media on autopilot with AI agents`，副文案强调自动 plan / generate / schedule 到 30+ social networks，并在 visual calendar 中 review / edit。

Postiz 首页不是完全隐藏 runtime。它列出 OpenClaw、Hermes、Claude、ChatGPT、Codex、Cursor，也提供 `I need agentic scheduling` / `I need normal scheduling` 分流。但首屏主 CTA 仍是 `Start a 7-day trial for $0`，说明 runtime 是信任和分流信息，不是首页主任务。

Postiz runtime 页服务高意图用户，而不是替代首页：

| 页面 | 作用 | 已验证 Quick Start |
|---|---|---|
| `/agent` | AI Agents CLI 承接页 | install skill -> set API key -> `postiz integrations:list` -> create post |
| `/codex` | Codex 承接页 | install skill -> set API key -> `codex "list my social media integrations"` -> create post |
| `/mcp` | Connector / MCP 承接页 | copy MCP link -> add to ChatGPT / Claude / Cursor -> plain English task |

最值得借鉴的是 first task 顺序：Postiz Codex 示例先 `list my social media integrations`，再 create post。它先确认账号和上下文，再执行业务动作。

### 2.2 不应照搬点

- 不照搬首次激活前信用卡卡墙。聚星当前更需要降低激活门槛，先让用户跑出一次 creator analysis 或 creator search。
- 不照搬 API key 作为默认主路径。聚星 CLI 已有 browser login，API key dashboard 应是 fallback。
- 不照搬一句 “human in the loop” 处理写操作风险。聚星已有 dry-run、validate、preview、apply、explicit approval 等安全模型，应直接沿用。
- 不把 Postiz 的 social publishing first task 迁移成聚星 first task。聚星 first task 应从达人读取、分析、搜索开始。

## 3. 聚星真实约束

以下事实来自 `noxinfluencer_skills` 和 `kol_claw` 当前文档与代码。

### 3.1 Skill 是 agent-first

Skill 的原则是：用户不手动操作 CLI，agent 自己运行命令，只在登录、注册、授权、订阅等必须浏览器完成的动作上给用户 URL。

所以 Quick Start 不能设计成“用户逐行复制命令”。正确方向是：

```text
用户在 Codex / Claude Code / OpenClaw / Hermes 里说业务目标
-> agent 运行 CLI
-> 用户只处理浏览器登录/授权
-> agent 返回结论、证据、下一步和 SaaS 直达链接
```

### 3.2 账号连接是 browser login 优先

CLI 的默认 Quick Start 是：

```bash
noxinfluencer login
```

它会打开 NoxInfluencer 浏览器登录，复用 SaaS session，创建或复用 API key，并保存到本地。手动 API key dashboard 只是 fallback。

这会影响页面设计：`/skills` 不应把 API key 作为主 CTA，也不应让用户先理解 key 再理解业务价值。

### 3.3 自检能力已经存在

Quick Start 应直接复用现有能力：

```bash
noxinfluencer doctor
noxinfluencer quota
noxinfluencer schema --all
noxinfluencer schema <cmd>
```

含义：

- `doctor` 检查 CLI、配置、连通性和 auth。
- `quota` 展示 Skill quota，并可通过 `action.url` 引导续费或升级。
- `schema --all` 检查 command tree 是否完整。
- `schema <cmd>` 让 agent 运行前发现参数，不靠页面硬编码所有 flags。

### 3.4 当前能力覆盖完整达人营销链路

当前 Skill/CLI 已覆盖：

- 达人发现：`creator search`、`creator lookalikes`、`creator search-filter*`
- 达人分析：`creator profile/audience/content/cooperation`
- 外部标识读取：`--url` 或 `--platform --channel-id`
- 联系信息：`creator contacts`，仅在用户明确要可见或导出联系方式时使用
- 视频监控：`monitor list/create/add-task/tasks/history/summary`
- 业务对象：`campaign`、`collection`、`crm`、`email`、`message`、`product`、`export`
- 品牌监控：`brand-monitor ...`
- MCP：当前 Remote MCP Phase 1 以 read-only tools 为主

因此 `/skills` 的价值表达不应停留在“达人发现、分析、名单沉淀”，而应表达完整链路，但写法要避免承诺无人监管的全自动执行。

## 4. 目标体验

第一版不需要定死所有页面路径，但需要统一激活口径：

```text
Skill activation =
auth complete
+ self-check success
+ first low-risk task success
+ object/deep link returned
```

推荐体验阶段：

| Stage | 用户问题 | 产品回答 | 设计重点 |
|---|---|---|---|
| 入口理解 | 这能帮我做什么 | AI 推进达人营销链路 | Hero 讲业务结果，不讲协议 |
| 开始方式 | 我怎么用 | Web / Skill runtime / MCP / API 分层承接 | 首页轻分流，细节进独立页 |
| 连接账号 | 能不能跑起来 | browser login 优先，key fallback | `noxinfluencer login` 优先 |
| 自检 | 是否连对了 | doctor、quota、schema | 返回清楚失败原因和 URL |
| First task | 第一件事做什么 | creator URL analysis 或 creator search | read / analyze 优先 |
| SaaS handoff | 结果去哪了 | 返回 creator / collection / monitor / draft 直达链接 | 避免用户进入平台后迷路 |
| Repeat / pay | 为什么回来或付费 | 保存更多、监控更多、触达更多、额度解释 | 在对象上下文里解释升级 |

最重要的问题：

```text
用户第一次成功之后，是否自然知道下一步打开哪里、继续做什么、为什么值得回来。
```

## 5. 页面与路径方案

### 5.1 `/skills` landing

目标：让用户 5 秒内知道“聚星 Skill 能帮我推进达人营销工作”，并启动第一个低风险任务。

建议首屏信息顺序：

```text
Hero：让 AI 7×24 推进达人营销全链路
Proof：现有账号复用、真实数据、可回到 SaaS 工作台、安全预览
Task cards：分析达人 URL / 找达人 / 建候选名单 / 监控效果 / 准备建联
Runtime entry：Add to your AI，进入 Codex / Claude Code / OpenClaw / Hermes / MCP 说明
CTA：让 AI 跑第一个达人分析
Secondary：查看安装方式 / 进入聚星工作台
```

任务卡建议：

| Task | 页面文案 | First task 类型 |
|---|---|---|
| Analyze creator URL | 粘贴达人主页，AI 给出合作判断 | read / analyze |
| Find creators | 说出品类、地区和平台，AI 找候选达人 | read / search |
| Build shortlist | 对比候选人并保存到名单 | analyze / save draft |
| Start outreach | 基于已选达人准备建联任务 | draft / preview |
| Monitor performance | 追踪合作视频表现和趋势 | read / monitor |

避免：

- 首屏把 API key 作为主 CTA。
- 首屏堆 MCP、REST、CLI 等协议说明。
- 首次激活前要求信用卡。
- 首屏承诺无人监管的自动发送、自动谈判或自动导出。

### 5.2 Runtime Quick Start

Quick Start 不应是统一手动命令模板，而应是“用户 prompt + agent expected actions”。

通用 activation loop：

```text
1. 安装对应 runtime 的 noxinfluencer skill
2. 用户在 runtime 里提出业务目标
3. agent 运行 doctor / quota / schema
4. 缺登录时 agent 运行 login，用户在浏览器完成登录/授权
5. agent 运行第一个 read / analyze 任务
6. agent 返回 verdict、证据、creator_id / object_id、SaaS 直达链接和下一步
```

示例：

```text
Prompt：检查我的 NoxInfluencer Skill 是否可用，并完成登录和额度检查。
Agent actions：doctor -> login（如未登录）-> quota -> schema --all（如需确认命令树）

Prompt：分析这个 YouTube 达人是否值得合作，并给出证据和下一步：
https://www.youtube.com/channel/...
Agent actions：schema creator.profile -> creator profile --url <url> -> 按需要继续 profile / audience / content / cooperation
```

Runtime 差异只按事实和机制解释：

| Runtime | 页面重点 |
|---|---|
| Codex | terminal / IDE 中通过 Skill 执行 CLI，用 prompt 触发 agent 自检和 creator analysis |
| Claude Code | 说明 Skills CLI 安装路径；避免把 CLI Skill 和 MCP connector 混写 |
| OpenClaw | 强调 Add to your AI、skill eligibility、agent 负责运行 CLI |
| Hermes | 只写已验证的 install / inspect，不承诺未验证的 long-running 场景 |
| MCP | 单独讲 OAuth / connect / list tools / quota / read-only first task |
| REST API | 放到 `/developer-api`，服务系统集成和开发者，不作为普通 Skill first value |

### 5.3 SaaS handoff

Dashboard 已经是聚星 SaaS 平台，不是一个新建的 key / usage 控制台。真正 gap 是：Skill 运行后用户可能找不到结果在哪里继续处理。

优先方案：

1. Skill / CLI / MCP 响应里返回对象直达 URL。
2. 错误和额度不足时使用 `action.url` / `action.hint`。
3. `/skills/dashboard` 继续承担 setup、key fallback、usage / billing，不作为 first value 终点。
4. 后续可在 SaaS 首页或 Skills dashboard 增加最近 AI 结果，但不是第一阶段核心。

Deep link 映射：

| Skill 结果 | SaaS 目标 |
|---|---|
| creator profile / analysis | 达人详情页或可打开的 creator URL |
| creator shortlist | collection draft / collection detail |
| monitor add-task / summary | monitor project / task detail |
| outreach draft | email task / message thread / draft detail |
| export task | export detail / download link |
| quota exceeded | usage-billing / plan upgrade |
| auth required | login / signup / oauth authorize |

Agent 输出应尽量像这样：

```text
已完成达人分析：Avery Cole，结论为 viable with risks。
关键证据：受众地区匹配，近 30 天内容稳定，但合作价格信号偏高。
creator_id：...
打开聚星继续查看：<creator deep link>
下一步：加入名单、继续查受众、创建监控、准备建联草稿。
```

### 5.4 Usage / billing

Usage 页面不应只是账单表，应解释“本月用 Skill 完成了什么”和“剩余额度还能做什么”。

第一版保留这些信息即可：

- 本月 creator analysis / search / monitor / export / outreach draft 次数。
- 已保存 creator、collection、monitor、draft 数。
- 最近一次消耗和剩余额度。
- 升级后能继续做的具体动作。

付费提示要绑定对象和下一步：

```text
你已保存 18 个候选达人。继续批量分析、导出 contacts 或创建监控，需要升级对应额度。
```

## 6. 实施优先级

### P0：最短激活闭环

1. 重写 `/skills` Hero、proof、task cards 和 CTA。
2. Quick Start 改为真实 Skill/CLI activation loop：`doctor`、`login`、`quota`、`schema`、creator URL analysis。
3. 默认 first task 选 creator URL analysis 或 creator search，不把 send / schedule / export 作为首次任务。
4. Skill 输出带 creator_id / object_id / action.url / deep link。
5. 错误恢复复用现有 CLI `action.url` / `action.hint`。

### P1：runtime 承接页

1. Codex Quick Start。
2. Claude Code Quick Start。
3. OpenClaw Quick Start。
4. Hermes install / inspect Quick Start。
5. MCP connector Quick Start。

### P2：复用和商业化

1. SaaS 首页或 Skills dashboard 增加最近 AI 结果入口。
2. 对象页标记 “由 Skill 创建 / 更新”。
3. Usage 页面从额度表升级为工作价值解释。
4. 在对象上下文里出现升级提示，而不是激活前卡墙。

## 7. 最小指标

第一版只需要能判断新用户是否完成激活，不要过早做复杂埋点体系。

| Event | 用途 |
|---|---|
| `skill_landing_view` | 入口曝光 |
| `skill_task_select` | 选择任务意图 |
| `skill_runtime_select` | 选择 runtime |
| `skill_auth_complete` | 登录或授权完成 |
| `skill_self_check_success` | doctor / quota / schema 自检成功 |
| `skill_first_task_success` | 首个低风险任务成功 |
| `skill_deep_link_returned` | 返回 SaaS 直达链接 |
| `skill_deep_link_opened` | 用户打开 SaaS 对象 |
| `skill_upgrade_intent` | 产生升级意图 |

必要维度：

- `runtime`：web、codex、claude_code、openclaw、hermes、mcp、api。
- `task_type`：creator_url_analysis、creator_search、shortlist_save、monitor_summary、outreach_draft、export。
- `region`：domestic、overseas。
- `source`：GA / UTM / referrer / ClawHub / skills.sh / runtime attribution，能取多少取多少，不补假数据。

## 8. 证据边界

已验证：

- Postiz 首页、`/agent`、`/codex`、`/mcp` 的公开页面结构和首屏文案。
- Postiz 首页主 CTA、agentic / normal scheduling 分流、runtime 列表。
- 聚星 Skill 的 agent-first 原则。
- 聚星 CLI 的 browser login、doctor、quota、schema、自描述能力。
- 聚星写操作已有 dry-run、validate、preview、apply、explicit approval 模型。

未验证：

- 未输入 Postiz 信用卡，未购买或创建真实 workspace。
- 未连接真实 Postiz 社媒账号。
- 未在 Codex、Claude Code、OpenClaw、Hermes 中执行真实 Postiz 工具调用。
- 未逐项确认聚星 SaaS 对象的稳定 deep link。

## 9. 需要确认的问题

1. Hero 是否允许使用 `7×24`。如果担心过度承诺，建议用备选版 `让 AI 持续推进达人营销全链路`。
2. `/skills` 首屏 Primary CTA 是“让 AI 跑第一个达人分析”，还是更直接的 “Add to your AI”。前者偏业务转化，后者偏 runtime 安装。
3. 第一版 runtime 承接页是否只做 Codex、Claude Code、OpenClaw、MCP，Hermes 是否先降级为安装说明。
4. creator、collection、monitor、email task、export、usage-billing 的稳定 deep link 是否已存在。若不存在，P0 需要先定义 URL 规范。
5. First task 默认选 creator URL analysis 还是 creator search。URL analysis 输入成本更低，search 更能体现数据能力。
