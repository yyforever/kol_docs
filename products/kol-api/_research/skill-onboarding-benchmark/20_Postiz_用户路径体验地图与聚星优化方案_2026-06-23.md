# Postiz 用户路径体验地图与聚星优化方案

日期：2026-06-23
调研对象：Postiz 官网、Agent CLI、Codex、MCP、Pricing / Trial、GitHub、Docs
目的：用 Postiz 作为主参考，重新组织聚星 Skill 及相关入口的新用户转化方案。

## 1. 核心判断

聚星当前要优化的不是“让用户知道我们有 Skill / MCP / API”，而是让用户完成一个闭环：

```text
理解 AI 能帮我完成哪件达人营销工作
-> 选择适合自己的使用环境
-> 成功连接账号和上下文
-> 跑出一个低风险、可信、可保存的业务结果
-> 回到聚星对象继续管理
-> 在对象价值清楚后产生复用和付费理由
```

Postiz 最有价值的地方，是它把“外部 AI runtime”包装成 SaaS 的新入口，但又让结果回到 Postiz 的对象系统。用户不是为了安装一个工具而来，而是为了让 AI 代替自己完成社媒排程；工具只是路径，SaaS object 才是留存和商业化的载体。

聚星应该学习这个模型，但不能照搬 Postiz 的具体做法：

- 学习：业务结果先行、runtime-specific onboarding、read-first first task、结果回写到 SaaS object。
- 不学：首次激活前信用卡卡墙、API key-in-URL 作为主路径、只用一句 human-in-loop 提醒治理写操作风险。

一句话方案：

```text
把 /skills 从“工具入口页”改成“AI 达人营销任务启动页”，把 Quick Start 从“安装说明”改成“跑出第一个可保存结果”，把 dashboard 从“key/usage 页面”改成“AI 结果的 system of record”。
```

## 2. Postiz 证明了什么

Postiz 的业务不是达人营销，但它证明了一个对聚星很关键的产品结构：

```text
Landing 承诺业务结果
-> Runtime 页面降低特定环境的上手成本
-> CLI / MCP / API 连接同一个 SaaS workspace
-> First task 先 discovery，再 action
-> 结果进入 SaaS object
-> Object 承接复访、协作、用量和付费
```

### 2.1 它不是先卖技术协议，而是先卖业务进展

Postiz 首屏是 `Run your social media on autopilot with AI agents`。它没有先解释 MCP、CLI、API，而是让用户先理解：

- AI agent 能帮我 plan / generate / schedule。
- 它能覆盖 30+ social networks。
- 我还可以在 visual calendar 里 review / edit。
- 我可以用 OpenClaw、Hermes、Claude、ChatGPT、Codex、Cursor。

这说明用户第一眼需要的是“我能完成什么工作”，不是“你支持哪些协议”。

聚星映射：

- 首屏主语应该是“AI 帮你完成达人发现、分析、名单沉淀和后续动作”。
- Codex / Claude Code / OpenClaw / Hermes 应作为使用环境出现，不应成为价值主语。
- MCP / REST API 应作为高级接入路径，不应抢普通新用户首屏。

### 2.2 它按用户使用环境重写入口，而不是复制同一份文档

Postiz 有 `/agent`、`/codex`、`/mcp`。这些页面复用同一底层能力，但每页都用对应 runtime 的语言解释：

| 页面 | 用户理解方式 | 本质 |
|---|---|---|
| `/agent` | AI agents 可以通过 CLI 管理社媒 | agent runtime 承接 |
| `/codex` | Codex 可以读取 skill 并在 CLI / IDE 中执行 | Codex-specific onboarding |
| `/mcp` | ChatGPT / Claude / Cursor 可通过一个 link 连接 | hosted / MCP client onboarding |

聚星映射：

- Codex 页应强调“在终端或代码工作流中完成达人分析和对象保存”。
- Claude Code 页应强调“结合文件、上下文、持续任务做运营分析”。
- OpenClaw / Hermes 页应强调“chat app / long-running agent / heartbeat 场景”。
- API / MCP 页应强调“系统集成、自动化工作流、fallback key”，不要和普通 Skill 首屏混写。

### 2.3 它的 first task 是 read-first，而不是一上来执行高风险动作

Postiz Codex 示例先让用户执行：

```bash
codex "list my social media integrations"
```

然后才是创建和排程内容。这是非常关键的 onboarding 模式：先证明已经连上账号、能读到对象、上下文正确，再进入写操作。

聚星映射：

- 第一条命令应该是 status / quota / workspace / brand / available objects。
- 第一个业务任务应是低风险 read / analyze / search。
- outreach、send、apply、schedule、bulk export 等动作不适合作为首次任务。

### 2.4 它把 AI 结果落回 SaaS object

Postiz 的对象包括 integrations、posts、draft / schedule state、calendar、media、analytics。用户回来的理由不是“再看文档”，而是继续管理这些对象。

聚星映射：

| Postiz object | 聚星对应 object |
|---|---|
| integrations / connected accounts | brand / workspace / auth context |
| posts / drafts / scheduled posts | creator candidate / outreach draft / campaign draft |
| calendar | monitor / campaign timeline |
| media | creator profile / content evidence |
| analytics | monitor summary / campaign performance / usage |

如果聚星 Skill 只在 AI client 里返回一段文本，不保存对象，用户路径就断了。

## 3. 聚星转化问题重定义

聚星的新用户转化不能只看“是否访问 `/skills`”或“是否拿到 API key”。真正的问题是：用户是否从第一次访问走到“我得到了一个有用、可信、可继续管理的达人营销对象”。

所以转化漏斗应从页面漏斗升级为工作流漏斗：

```text
访问入口
-> 选择任务 / runtime
-> 连接账号和上下文
-> 自检成功
-> 首个低风险任务成功
-> 保存对象
-> 打开 dashboard 对象
-> 继续下一步或产生付费意图
```

这套漏斗同时覆盖：

- Web / dashboard 手动体验。
- Codex / Claude Code / OpenClaw / Hermes 运行路径。
- MCP / REST API 高级路径。
- 后续 ChatGPT / Claude connector 路径。

不同入口可以不同，但第一次成功必须落到同一套聚星对象和 usage 体系。

## 4. 聚星目标体验地图

| Stage | 用户心智 | 产品应该回答 | 主要风险 | 指标 |
|---|---|---|---|---|
| 1. 看到入口 | 这对我有什么用 | AI 能完成具体达人营销任务 | 首屏讲技术，用户看不懂 | landing view、CTA click |
| 2. 选择开始方式 | 我该点哪个 | 按任务和 runtime 分流 | 入口按内部模块堆叠 | intent select、runtime select |
| 3. 连接账号 | 能不能跑起来 | browser login / OAuth-first、自检 | 要 key、跳转失败、上下文不明 | setup start、auth complete |
| 4. 自检 | 是否连对了 | brand、workspace、quota、available tools | 用户不知道失败原因 | status check success |
| 5. First task | 第一件事做什么 | 低风险 creator analysis / search / compare | 任务太难、太慢、太危险 | first task success |
| 6. First value | 结果是否可信 | 结构化结果、理由、证据、下一步 | 只返回文本，缺少解释 | result viewed、second action |
| 7. Object handoff | 结果去哪了 | 保存为 candidate / collection / monitor / draft | dashboard 无痕迹 | object saved、object opened |
| 8. Repeat / pay | 为什么回来或付费 | 继续管理、消耗解释、升级理由 | 没看到价值先收费 | return visit、quota hit、upgrade intent |

优秀 PM 会优先盯住一个问题：

```text
用户第一次成功之后，是否自然产生“我要回来继续看这个对象 / 继续推进这个任务”的理由？
```

如果没有这个理由，landing、docs、runtime 页面再精致，也只是一次性体验。

## 5. 产品架构原则

### 5.1 价值层：先任务，后协议

首屏和 onboarding 先回答任务：

- 找达人。
- 分析达人 URL。
- 对比候选达人。
- 保存名单。
- 总结监控变化。
- 生成 outreach draft。

协议和工具只作为承接：

- Web。
- Codex。
- Claude Code。
- OpenClaw。
- Hermes。
- MCP。
- REST API。

### 5.2 路径层：多入口，同一对象

用户可以从不同入口进来，但不能产生不同孤岛：

```text
Web preview
Codex command
Claude Code task
OpenClaw conversation
MCP / API call
-> 聚星 creator / collection / monitor / draft / usage object
```

Dashboard 是 system of record。所有路径的结果都应该能回到 dashboard 查看、继续、分享、付费。

### 5.3 安全层：先 read，再 draft，最后 execute

聚星涉及达人联系、邮件发送、合作申请、内容发布、批量导出等高风险动作，不能只靠“请 human in the loop”一句提醒。

默认动作顺序：

```text
read / analyze
-> draft / preview
-> save object
-> confirm
-> execute
```

新用户 first task 应尽量停在 read、analyze、draft，不直接 execute。

### 5.4 商业化层：先价值，再付费

Postiz 的卡墙适合强付费过滤，不适合聚星当前的激活目标。聚星应让用户先完成 1-2 个高确定性的 first value，再在对象上下文里解释升级：

- 想保存更多 creator。
- 想批量分析。
- 想持续 monitor。
- 想导出 contacts。
- 想执行 outreach / send / apply。
- 剩余额度不足。

## 6. 页面与路径方案

### 6.1 `/skills` landing

目标：让用户在 5 秒内知道“聚星 Skill 能帮我完成什么达人营销工作”，并选择一个开始方式。

建议信息顺序：

```text
Hero：AI 帮你完成达人发现、分析与名单沉淀
Proof：真实数据覆盖、主站账号复用、免费额度、dashboard 可管理
Task cards：分析达人 URL / 找达人 / 对比候选 / 保存名单 / 总结监控
Runtime entry：Codex / Claude Code / OpenClaw / Hermes / Web / API
Result preview：creator card、fit score、collection draft、monitor summary
CTA：开始第一个任务
Secondary：查看 runtime Quick Start / API docs
```

不建议：

- 首屏把 API key 作为主入口。
- 首屏堆工具列表。
- 首屏让用户先选 MCP / REST / CLI 这种技术分类。
- 在 first value 之前要求信用卡。

### 6.2 Runtime Quick Start

目标：每个 runtime 都能让用户跑通同一套 activation loop，而不是读一份安装文档。

统一模板：

```text
1. 安装 / 打开对应 runtime
2. 登录聚星
3. 运行 status check
4. 运行低风险 first task
5. 保存为聚星对象
6. 打开 dashboard 继续
7. 失败时按 auth / quota / input / service 分类恢复
```

不同 runtime 的解释重点：

| Runtime | 文案重点 | 第一条命令 / prompt |
|---|---|---|
| Codex | 在终端 / repo 环境中完成结构化任务 | 检查登录状态、brand、workspace、quota |
| Claude Code | 结合文件和上下文做运营分析 | 分析 creator URL 并生成候选卡片 |
| OpenClaw | 在 chat app 里低门槛执行 | 找一组达人并保存 collection draft |
| Hermes | 长期运行和周期任务 | 总结 monitor 变化，先生成摘要，不自动触达 |
| MCP / API | 高级集成 | list tools、quota、sample call、object result |

### 6.3 First value 设计

第一版不要把 first value 做成“功能最全”，而要做成“最容易成功、最容易理解、最容易保存”。

优先级：

| Priority | First value | 原因 | 约束 |
|---|---|---|---|
| P0 | Creator URL analysis | 输入成本低，结果直观，适合外部 runtime | 需要展示数据时间和可信度 |
| P0 | Creator search with reasons | 展示核心数据能力 | 控制结果数量、成本和等待 |
| P0 | Save collection draft | 建立 dashboard handoff | 草稿质量要可编辑 |
| P1 | Creator comparison | 展示分析深度 | 输入需要更清晰 |
| P1 | Monitor summary | 适合复访和已有用户 | 新用户可能没有 monitor |
| P2 | Outreach draft | 展示闭环 | 只能 draft，不自动发送 |

### 6.4 Dashboard handoff

Dashboard 需要承接 AI runtime 的结果，不应只是 key、usage 或空控制台。

Dashboard 首页建议模块：

- 最近一次 AI 运行结果。
- 最近保存的 creator candidates / collections / monitors / drafts。
- 当前 quota 和本月消耗。
- 继续下一步 CTA。
- Runtime 接入状态。
- 常见失败恢复。

对象页建议显示：

- 来源：哪个 runtime、哪个 prompt、哪个用户。
- 数据依据：来源时间、关键指标、confidence。
- 下一步：继续分析、加入 collection、创建 monitor、生成 outreach draft。
- 费用：本次消耗和下一步预计消耗。

### 6.5 Usage / billing

Usage 页面要从“账单表”变成“工作价值解释”。

建议展示：

- 本月完成了多少 creator analysis / search / comparison。
- 保存了多少 creator / collection / monitor / draft。
- 哪些任务消耗最多。
- 剩余额度还能做什么。
- 升级后能多做什么。

付费提示应绑定具体对象和下一步，而不是抽象套餐：

```text
你已保存 18 个候选达人。
继续批量分析和导出 contacts 需要升级。
```

### 6.6 API / MCP / Developer path

API / MCP 是重要路径，但不应成为普通新用户首屏默认路径。

定位：

- 给高级用户、自建工作流、团队系统集成。
- 提供 OAuth-first，API key 作为 fallback。
- 结果仍应写入 usage 和 object history。

关键原则：

- 不把 key-in-URL 作为主路径。
- 不让用户先拿 key 再理解业务价值。
- 示例 API call 应返回可保存或可追踪对象，而不是一次性 JSON。

## 8. 实验与指标

### 8.1 Activation 定义

不要只看 `/skills` 页面访问，也不要只看 API key。建议统一定义：

```text
Skill activation =
auth/context 成功
+ 首个低风险任务成功
+ 至少一个结果被保存或打开
```

总激活按入口合并：

```text
Total activation =
Web first value
+ Runtime first successful call
+ MCP / API first successful call
+ Dashboard object continuation
```

### 8.2 最小事件集

| Event | 说明 |
|---|---|
| `skill_landing_view` | 访问 `/skills` |
| `skill_intent_select` | 选择任务或 runtime |
| `skill_setup_start` | 开始安装 / 登录 / OAuth |
| `skill_auth_complete` | 登录或 token 成功 |
| `skill_status_check_success` | brand / workspace / quota / tools 自检成功 |
| `skill_first_task_start` | first task 发起 |
| `skill_first_task_success` | first task 成功返回 |
| `skill_object_saved` | 保存 candidate / collection / monitor / draft |
| `skill_object_opened` | 打开 dashboard 对象 |
| `skill_quota_explained` | 展示本次消耗和剩余额度 |
| `skill_upgrade_intent` | 点击升级或增购 |

必要维度：

- runtime：web、codex、claude_code、openclaw、hermes、mcp、api。
- task_type：creator_url_analysis、creator_search、creator_compare、collection_save、monitor_summary、outreach_draft。
- region：domestic、overseas。
- user_type：brand 优先。
- acquisition_source：GA / UTM / referrer / cli attribution，能取多少取多少，不补假数据。

### 8.3 实验 backlog

| Priority | Experiment | 成功信号 | 风险 |
|---|---|---|---|
| P0 | `/skills` 首屏改为业务结果 + task cards | CTA click、task card click、setup start | 文案过度承诺 |
| P0 | Quick Start 先做 status / quota / workspace self-check | first command success、support issue 下降 | 自检结果不稳定 |
| P0 | Creator URL analysis 作为默认 first task | first task success、second action | 数据覆盖不足 |
| P0 | First result 保存为 candidate / collection draft | object saved、object opened、return visit | 草稿质量差 |
| P1 | runtime-specific Quick Start | setup start、auth complete、runtime first call | 维护成本上升 |
| P1 | Usage 页面解释还能做什么 | quota page visit、upgrade click | cost estimate 不准 |
| P1 | 高风险动作统一 staged draft | draft created、confirm rate、execute error | 流程变长 |

## 10. 最终取舍

聚星对 Postiz 的正确用法是抽象模型，不是复制页面：

- 用 Postiz 证明 `AI runtime -> SaaS object` 是可行的新用户路径。
- 用 Postiz 证明 runtime-specific onboarding 比泛泛 docs 更容易理解。
- 用 Postiz 证明 first task 应先 discovery，再 action。
- 不用 Postiz 证明信用卡卡墙适合聚星。
- 不用 Postiz 证明 API key-in-URL 适合聚星。
- 不用 Postiz 的 social publishing first task 替代聚星的达人营销 first value。

下一步应该进入聚星自身方案设计，而不是继续扩竞品：

1. 输出 `/skills` landing 信息架构。
2. 输出 Codex / Claude Code / OpenClaw / Hermes Quick Start。
3. 用现有 Skill 能力验证 creator URL analysis 和 creator search 的 first task 成功率。
4. 设计 dashboard saved-object 承接。
5. 补最小埋点，确保 web、runtime、MCP / API 能合并统计。
