# Postiz 用户路径体验地图与聚星优化方案

日期：2026-06-23
调研对象：Postiz 官网、Agent CLI、Codex、MCP、Pricing / Trial、GitHub、Docs
目标：以 Postiz 为当前主参考，重新梳理聚星 Skill 及相关入口的新用户转化路径，形成可落地的页面、引导、指标和实验方案。

## 1. 结论摘要

Postiz 最值得聚星学习的不是社媒排程能力，而是它把一个 SaaS 产品包装成“可以被外部 AI runtime 调用的业务系统”的方式：

- 首屏直接说清业务结果：让 AI agent 自动计划、生成、排程社媒内容。
- 明确列出可用 runtime：OpenClaw、Hermes、Claude、ChatGPT、Codex、Cursor。
- 把不同用户意图分开：agentic scheduling、normal scheduling、developers、automation teams。
- 为不同 runtime 单独写页面：`/agent`、`/codex`、`/mcp` 使用同一底层能力，但文案按使用环境重写。
- First task 先做 read / discovery：Codex 示例先 `list my social media integrations`，再创建内容。
- 结果回到 SaaS 对象：integrations、posts、draft / schedule、calendar、media、analytics。

聚星不能照搬的部分也很明确：

- Postiz 的 `Start for $0` 进入带信用卡的 trial/paywall，聚星不应在首次激活前设置卡墙。
- Postiz 的 MCP URL 直接包含 API key，聚星不应把 key-in-URL 作为普通用户主路径。
- Postiz 的写操作治理偏弱，主要靠 human-in-loop 提醒；聚星涉及 outreach、send、apply、schedule 等动作时，需要 preview / draft / confirm / dashboard review。

聚星新用户路径的核心优化方向应是：

```text
先让用户理解“AI 能帮我完成哪件达人营销工作”
-> 再按使用环境选择进入网页、Codex、Claude Code、OpenClaw、Hermes、API / MCP
-> 先跑 read-first / low-risk first task
-> 把结果保存为聚星对象
-> 从对象继续引导下一步、用量、升级和复访
```

## 3. Postiz 用户路径拆解

### 3.1 Awareness：第一眼卖的是业务进展，不是技术协议

Postiz 首页没有先解释 MCP、CLI 或 API，而是直接说“让 AI agents 帮你跑社媒”。这降低了用户理解成本，因为用户先看到的是业务进展：

```text
我要增长社媒
-> AI agent 可以帮我 plan / generate / schedule
-> 我还能在 calendar 里 review / edit
```

首屏同时放了：

- 大标题：业务结果。
- runtime 名称：OpenClaw / Hermes / Claude / ChatGPT / Codex / Cursor。
- 渠道图标：30+ social networks。
- Product Hunt 奖项：信任背书。
- 单一 CTA：Start a 7-day trial for $0。

对聚星的启发：

- `/skills` 首屏不能只说 “MCP / CLI / API / AI Agent”。
- 首屏应该先说业务结果，例如 “在 Codex / Claude Code / OpenClaw 中发现、分析并沉淀适合你品牌的创作者”。
- runtime 名称要出现，但它是承接使用环境，不是价值主语。

### 3.2 Interest：把用户按意图分流，而不是按内部模块分流

Postiz 首页有两个显性意图：

- `I need agentic scheduling`
- `I need normal scheduling`

后面又把用户分成：

- Agentic：从 Claude、ChatGPT、Codex、OpenClaw prompt Postiz。
- Developers：OAuth2、SDK、Public API。
- Automation Teams：n8n、Make.com workflows。

这不是简单导航，而是在回答“我是哪类用户，我该从哪里开始”。

对聚星的启发：

- 聚星不应按内部资产先分 `Skill / MCP / API / Docs / Dashboard`。
- 应先按用户任务分：
  - 我想让 AI 帮我找达人。
  - 我已经有达人链接，想快速分析。
  - 我想把结果保存成名单或项目。
  - 我想在 Codex / Claude Code / OpenClaw / Hermes 里使用。
  - 我想用 API / MCP 接入自己的工作流。
- 内部资产是承接路径，不是首屏分类标准。

### 3.3 Setup：不同 runtime 单独解释，底层能力保持一致

Postiz 为不同使用环境做了不同页面：

| 页面 | 主诉求 | 用户看到的第一件事 |
|---|---|---|
| `/agent` | Social Media CLI for AI Agents | CLI / OpenClaw / Claude 等 agent 可用 |
| `/codex` | Social Media CLI for Codex | Codex 读 `SKILL.md`，在 CLI / IDE 中执行 |
| `/mcp` | Social media MCP server for your AI | 一个 link 连接 ChatGPT / Claude / Cursor / any assistant |

它没有给每个 runtime 重新造业务逻辑，而是复用同一套 Postiz workspace、API、integrations、posts、analytics。

对聚星的启发：

- 聚星需要 runtime-specific 页面或 Quick Start tab，但底层必须复用同一套 Skill / MCP / quota / object。
- Codex 页面应该强调：在仓库/终端里完成达人营销任务，返回结构化结果。
- Claude Code 页面应该强调：在开发或运营工作流中持续执行、生成可审查结果。
- OpenClaw / Hermes 页面应该强调：chat app / long-running agent / heartbeat 场景、自动化边界和失败恢复。
- MCP / API 页面应该强调：给系统集成或自建 workflow 用，不应抢普通新用户主路径。

### 3.4 Activation：first task 应该先 read-first，而不是一上来写操作

Postiz 的 Codex 示例先让用户执行：

```bash
codex "list my social media integrations"
```

然后才是：

```bash
codex "post 'Hello from Postiz Agent!' to Twitter, schedule for tomorrow at 9am"
```

这说明优秀的 external runtime onboarding 不应第一步就让用户做高风险动作。它应该先让用户确认：

- 已经连上账号。
- workspace / integration 可见。
- agent 能读取对象。
- 下一步有明确候选。

对聚星的 first task 建议：

| First task | 风险 | 为什么适合新用户 |
|---|---:|---|
| 分析一个 creator URL 是否适合我的品牌 | 低 | 用户容易理解，输入成本低，有即时结果 |
| 找 10 个某品类创作者并解释推荐理由 | 低 | 展示核心数据能力，但要控制成本和等待 |
| 对比 3 个达人，输出 fit score 和风险 | 低 | 体现分析深度，容易保存成对象 |
| 总结一个已有 collection / monitor | 低 | 适合已有用户复访和留存 |
| 创建 outreach draft，不发送 | 中 | 能展示业务闭环，但必须停在 draft |

不建议作为新用户第一任务：

- 直接发送邮件。
- 直接申请合作。
- 直接排程或发布外部内容。
- 直接导出大量 contacts。
- 先让用户复制 API key 再调用接口。

### 3.5 First value：结果必须落到聚星对象

Postiz 的优势不是“AI 说了一段话”，而是结果能进入：

- integrations
- posts
- draft / schedule state
- calendar
- media
- analytics

这让用户有回访理由，也让付费升级有上下文。

聚星对应的对象应是：

- creator candidate
- creator comparison
- collection draft
- campaign draft
- monitor summary
- outreach draft
- email task draft
- usage / quota history

体验上应避免：

```text
用户在 agent 里得到一段文本
-> 文本不保存
-> 用户不知道下一步去哪
-> dashboard 没有痕迹
-> 付费页没有上下文
```

目标状态：

```text
agent 生成结果
-> 返回可打开的聚星对象链接
-> dashboard 展示该对象和下一步 CTA
-> usage / quota 能解释本次消耗
-> 后续升级与对象价值绑定
```

### 3.6 Monetization：Postiz 的卡墙不适合聚星首次激活

Postiz 从 `Start for $0` 进入 trial 页面后，第一屏直接要求：

- 选择 plan。
- 输入 card number、expiration、CVC、country、ZIP。
- 点击 `Pay $0 Today - Start your free trial!`。

它的好处：

- 过滤低意向用户。
- 付费计划和试用预期清楚。
- 对已经被 landing 说服的人，商业化路径短。

它的问题：

- 阻断首次体验。
- 对还没看到 first value 的用户很重。
- 不适合聚星当前需要优化的新用户激活阶段。

聚星建议：

- 首次激活前不要要卡。
- 免费额度先支撑 1-2 个清晰 first value。
- 付费触发应发生在用户看到对象价值之后，例如：
  - 想保存更多 creator。
  - 想导出 contacts。
  - 想持续 monitor。
  - 想批量分析。
  - 想发送 / schedule / apply。
  - quota 即将用完。

### 3.7 Retention：从一次性调用变成可复用工作流

Postiz 的 retention hook 是 social calendar、scheduled posts、analytics 和多平台账号状态。用户回来不是为了看文档，而是为了继续管理对象。

聚星应设计自己的 retention hook：

- collection：继续筛选、比较、导出、分享。
- monitor：每天/每周查看变化。
- campaign：推进达人名单、触达、状态。
- outreach draft：审查、修改、发送。
- usage：看本月额度、消耗最多的工具、升级建议。

对新用户路径来说，第一次结果必须自然引出这些 hook。

## 4. 聚星体验地图

### 4.1 当前目标用户

优先用户不是泛开发者，而是有明确达人营销任务、愿意在 AI client 里完成工作的品牌/运营/增长用户。

典型心智：

- 我已经在用 Codex / Claude Code / OpenClaw / Hermes。
- 我想让 AI 帮我找达人、分析达人、整理名单、监控效果。
- 我不想先读一堆 API 文档。
- 我担心账号、额度、数据可信度和误操作。
- 如果结果有用，我愿意回到 SaaS dashboard 继续管理。

### 4.2 体验地图

| Stage | 用户问题 | 聚星应提供 | 关键卡点 | 核心指标 |
|---|---|---|---|---|
| Awareness | 这东西帮我做什么 | 业务结果、样例输出、runtime 名称、信任背书 | 首屏讲技术、看不懂 | landing CTA click、scroll depth |
| Intent selection | 我该从哪里开始 | 按任务和 runtime 分流 | 入口太多、按内部模块分类 | runtime tab click、task card click |
| Setup | 怎么连上我的账号 | browser login / OAuth-first、自检命令、quota 显示 | 先要 key、登录失败、代理/跳转失败 | setup start、auth complete |
| First run | 第一条命令跑什么 | low-risk first task、示例 prompt、预计耗时/额度 | 第一条任务太危险或太抽象 | first successful call |
| First value | 我得到了什么 | 结构化结果、解释依据、下一步建议 | 只返回文本，不知道是否可信 | result viewed、second prompt |
| Object handoff | 结果去哪了 | collection / monitor / draft 链接 | dashboard 无痕迹 | object saved、object opened |
| Repeat | 下次为什么回来 | usage history、saved objects、monitor changes | 一次性体验结束 | return visit、object reopened |
| Monetization | 为什么现在付费 | 与对象价值绑定的升级提示 | 没看到价值先付费 | quota hit、upgrade click、payment start |

### 4.3 优秀 PM 会先问的 8 个问题

这里不从功能清单出发，而从用户进展出发：

1. 用户第一次进来时，是否在 5 秒内知道“AI 可以帮我完成哪件达人营销工作”？
2. 用户是否能不用理解 API key / MCP / quota 术语，就选到自己的开始方式？
3. 用户第一条任务是否低风险、低输入成本、能稳定成功？
4. 用户看到结果时，是否知道结果为什么可信、下一步该做什么？
5. 结果是否变成聚星对象，而不是消失在聊天记录里？
6. 用户失败时，是不是能自己判断卡在 auth、quota、输入、权限还是服务错误？
7. 付费提示是否发生在用户已经感受到对象价值之后？
8. 不同入口的激活指标是否能合并成一个真实的 activation，而不是各自看局部漏斗？

## 5. 聚星优化方案

### 5.1 `/skills` landing 改造方向

首屏建议结构：

```text
H1：让你的 AI 在聚星里完成达人发现、分析与名单沉淀
Subcopy：在 Codex、Claude Code、OpenClaw、Hermes 或网页中，输入品牌和目标，得到可保存的达人结果、分析理由和下一步动作。
Primary CTA：在 AI 客户端中开始
Secondary CTA：先看网页示例
Trust：真实数据覆盖、主站账号复用、免费额度、可回到 dashboard 管理
```

首屏不建议：

- 把 `API key` 作为主 CTA。
- 把 `MCP / CLI / REST API` 放在最醒目的价值位置。
- 一上来堆工具列表。
- 在用户看到结果前引导购买。

可以借鉴 Postiz 的视觉组织：

- 黑底或强对比首屏可以保留，但必须服务于“AI agent 帮我完成工作”的心智。
- 展示 runtime logos / names，但不要让它们抢走业务结果。
- 展示 3-4 个任务卡，而不是一屏 20 个工具。
- 用可视化结果预览替代抽象文案，例如 creator card、fit score、collection draft。

### 5.2 Runtime 入口设计

建议建立统一但不混写的入口：

| Runtime | 页面重点 | 第一条推荐 prompt |
|---|---|---|
| Codex | 在终端 / repo 环境运行，适合结构化分析和保存对象 | `检查我的聚星 Skill 登录状态、品牌上下文和剩余额度` |
| Claude Code | 类似 Codex，但更强调持续工作流和文件上下文 | `分析这个达人主页，并生成可保存的候选卡片` |
| OpenClaw | chat app / agent workflow，适合低门槛使用 | `帮我找 10 个适合某品类的达人，并保存成 collection draft` |
| Hermes | 如果偏 long-running / automation，应强调周期任务和失败暂停 | `每周总结我的 monitor 变化，先生成摘要，不自动触达` |
| MCP / API | 给高级用户和系统集成 | `列出可用工具、quota 和示例调用` |

统一规则：

- 先 auth / status / quota。
- 再 read / analyze / search。
- 再 save object。
- 最后才进入 draft / execute。

### 5.3 Quick Start 改造方向

Quick Start 不应只是安装说明，应按“跑出第一个业务结果”组织。

建议结构：

```text
Step 1：选择你的使用环境
Step 2：登录并自检
Step 3：运行低风险 first task
Step 4：打开聚星对象链接
Step 5：继续下一步或查看额度
```

每个 runtime 都应提供：

- 一条 status prompt。
- 一条 first value prompt。
- 一条 save object prompt。
- 一个常见失败恢复入口。

示例：

```text
先检查我是否已经登录聚星 Skill，告诉我当前 brand、workspace、剩余额度和可用工具。
```

```text
分析这个 YouTube/TikTok 达人主页是否适合 [品牌/品类]，输出推荐理由、风险点，并保存为候选达人草稿。
```

```text
把刚才推荐的达人保存成一个 collection draft，并给我聚星 dashboard 链接。
```

### 5.4 Dashboard 承接设计

Dashboard 不是只展示 API key。它应该承接 agent 跑出来的结果。

建议首页模块：

- 最近一次 AI 运行结果。
- 最近保存的 creators / collections / monitors / drafts。
- 当前 quota 和本月消耗。
- 继续下一步 CTA。
- 常见失败与恢复。
- Runtime 接入状态。

对象详情页应显示：

- 来源：来自哪个 runtime / prompt / user。
- 数据依据：达人数据、时间、置信度。
- 下一步：继续分析、加入 collection、生成 outreach draft、创建 monitor。
- 费用：本次消耗多少，继续操作预计多少。

### 5.5 Usage / billing 改造方向

Postiz 的 pricing 是 plan-first。聚星当前更应 value-first：

```text
先免费跑出结果
-> 再解释本次消耗
-> 再展示继续做更多同类任务需要什么额度
```

usage 页面建议从“账单表”升级为“工作价值解释”：

- 本月完成了多少次 creator analysis。
- 保存了多少 creator / collection / monitor。
- 哪些工具消耗最多。
- 剩余额度还能完成什么。
- 升级后能多做什么。

### 5.6 非 Skill 路径也要统一

虽然当前重点是 Skill，但其他路径不能互相打架：

- 网页体验：给非 runtime 用户先体验 creator search / analysis / collection。
- API / MCP：给高级用户和系统集成，但不要抢普通用户首屏。
- Dashboard：作为所有路径的 system of record。
- Docs：服务 setup 和故障恢复，不承担首屏教育。
- Sales / custom API：继续承接大客户，但不应该阻断自助试用。

统一原则：

```text
不管用户从哪里开始，第一次成功都应该落到同一套聚星对象和 usage 体系。
```

## 6. 建议的新用户路径候选

### 6.1 Skill / runtime 主路径

```text
/skills landing
-> 选择任务或 runtime
-> runtime-specific Quick Start
-> browser login / OAuth-first
-> status / quota / workspace self-check
-> low-risk first task
-> 保存为聚星对象
-> dashboard 打开对象
-> 下一步 / quota / upgrade
```

### 6.2 网页手动路径

```text
/skills landing
```

## 8. 指标口径建议

不要只看 `/skills` 页面漏斗。Skill 的真实 activation 应按入口面合并：

```text
Total Skill activation =
Web first value
+ Runtime first successful call
+ MCP / API first successful call
+ Dashboard saved-object continuation
```

建议事件：

| Event | 说明 |
|---|---|
| `skill_landing_view` | `/skills` 首屏访问 |
| `skill_intent_select` | 选择任务或 runtime |
| `skill_setup_start` | 开始安装 / 登录 / OAuth |

- runtime：web、codex、claude_code、openclaw、hermes、mcp、api。

- region：domestic、overseas。
- user_type：brand 优先。
- acquisition_source：GA / UTM / referrer / cli attribution，能取多少取多少，不强行补假数据。

## 10. 对 Postiz 的最终判断

Postiz 是当前最适合作为聚星新用户路径重构的主参考，因为它同时满足：

- 产品不是纯 API，而是真实 SaaS。
- 能被网站之外的 AI runtime 使用。
- 有 CLI skill、MCP、Codex、OpenClaw、Public API、dashboard 多入口。
- 有可见对象模型和回访对象。
- 有明确的 landing 心智和客观采用信号。

但聚星只学习它的三个方面：

- `AI agent -> SaaS object` 的包装方式。
- runtime-specific onboarding。
- landing 上把业务结果、runtime 和 trust 放在一起的表达。

聚星不学习它的三个方面：

- 首次激活前卡墙。
- API key-in-URL 作为主路径。
- 只用 human-in-loop 提醒承接写操作风险。
