# Postiz 用户路径体验地图与聚星优化方案

日期：2026-06-23
调研对象：Postiz 官网、Agent CLI、Codex、MCP、Pricing / Trial、GitHub、Docs、聚星 Skill / CLI 本地仓库
目的：用 Postiz 作为主参考，优化聚星 Skill 及相关入口的新用户路径，提高首次激活、复用和付费转化。

## 1. 核心判断

聚星当前要优化的不是“让用户知道我们有 Skill / MCP / API”，而是让新用户尽快完成一个可信的业务闭环：

```text
理解 AI 能替我推进哪类达人营销工作
-> 选择一个适合自己的使用环境
-> 完成账号连接和自检
-> 跑出第一个低风险业务结果
-> 通过直达链接回到聚星 SaaS 对象继续管理
-> 在对象价值和额度消耗清楚后产生复用和付费理由
```

Postiz 最值得借鉴的是产品结构，不是表层文案。它把外部 AI runtime 包装成 SaaS 的新入口，但最终让结果回到 Postiz 的账号、集成、内容、日历和 analytics 对象里。聚星也应该用这个结构，但要按自己的真实 Skill 能力改造：

- Postiz 的业务主线是社媒排程，首次任务可以直接创建 post。
- 聚星 Skill 的业务主线更长，覆盖达人发现、分析、建联、谈判、效果监控和运营对象管理。
- 聚星的 SaaS 工作台已经存在，问题不是缺 key / usage 页面，而是 Skill 运行后用户容易找不到对应的 SaaS 对象和下一步。
- 聚星的新用户第一步应优先跑 read / analyze / draft 类任务，不能把 send、schedule、批量导出等高风险动作放在 first task。

一句话方案：

```text
把 /skills 从“工具入口页”升级为“AI 达人营销启动页”：
Hero 卖完整达人营销链路，Quick Start 按真实 Skill/CLI 跑通第一个结果，Skill 输出必须带回聚星 SaaS 的直达链接。
```

## 2. Postiz 证明了什么

### 2.1 首页先卖业务结果，而不是先卖 runtime

已验证 Postiz 首页首屏信息：

- H1：`Run your social media on autopilot with AI agents`
- 副文案：`Plan, generate, and schedule posts automatically to 30+ social media networks ...`
- 首屏列出 `OpenClaw / Hermes / Claude / ChatGPT / Codex / Cursor`
- 主 CTA：`Start a 7-day trial for $0`
- 首页存在 `I need agentic scheduling` / `I need normal scheduling` 分流

关键结论：

Postiz 并不是完全隐藏 runtime，也不是让用户在首页先做复杂 runtime 选择。它的首页主任务仍然是卖“社媒自动化排程”这个业务结果，runtime 只是证明“你的 AI 可以运行它”。真正的 runtime 细节放到独立页面承接。

聚星映射：

- `/skills` 首页主语应是 AI 达人营销业务结果，不是 CLI、MCP、API key。
- runtime 名称可以露出，但不能抢首屏主线。
- 聚星比 Postiz 更需要任务分流，因为达人营销链路比发帖排程更复杂。

建议 Hero 文案：

```text
H1：让 AI 7×24 推进达人营销全链路
Subcopy：从达人发现、分析、建联谈判到效果监控，聚星把数据、流程和结果沉淀回 SaaS 工作台。
```

这个版本保留“7×24”和完整链路，但不把“全自动”说得过满。更强版本可以用于广告或二级标题：

```text
把达人发现、分析、建联、谈判和监控交给 AI 持续推进。
```

### 2.2 Runtime 页面是高意图承接页，不是首页主路径

已验证 Postiz runtime 页面：

| 页面 | 首屏定位 | Quick Start 结构 |
|---|---|---|
| `/agent` | `Social Media CLI for AI Agents` | install skill -> set API key -> `postiz integrations:list` -> `postiz posts:create` |
| `/codex` | `Social Media CLI for Codex` | install skill -> set API key -> `codex "list my social media integrations"` -> create first post |
| `/mcp` | `The social media MCP server for your AI` | copy MCP link -> add to ChatGPT / Claude / Cursor -> plain English task |

Postiz 这样设计的原因：

- 首页面向广义 SaaS 用户，目标是解释业务价值并推动 trial。
- Runtime 独立页面向高意图用户，目标是降低特定环境的安装和连接成本。
- 不在首页展开所有 runtime，可以避免首页变成工具说明书。
- Runtime 页面还能承接 SEO、GitHub、ClawHub、agent marketplace、docs 等外部入口。

聚星映射：

- `/skills` 首页应保留清晰单主线：AI 达人营销业务结果 + 开始第一个任务。
- Codex / Claude Code / OpenClaw / Hermes / MCP / API 应有独立 Quick Start 或深链，但不应让首页变成 runtime 选择器。
- 首页可以放一组 “Add to your AI” 入口，点击后进入对应 runtime 说明。

### 2.3 Postiz first task 的顺序值得借鉴

Postiz `/codex` 的第一条示例不是直接创建内容，而是：

```bash
codex "list my social media integrations"
```

然后才是：

```bash
codex "post 'Hello from Postiz Agent!' to Twitter, schedule for tomorrow at 9am"
```

它的模式是：

```text
先确认账号、集成和上下文
-> 再执行业务动作
```

聚星映射：

```text
先 doctor / quota / schema / workspace 自检
-> 再跑 creator URL analysis 或 creator search
-> 再保存 collection / monitor / draft
```

## 3. 聚星 Skill / CLI 的真实事实

以下事实来自本地仓库：

- `/Users/yangyang/Github/noxinfluencer_skills/README.md`
- `/Users/yangyang/Github/noxinfluencer_skills/skills/noxinfluencer/SKILL.md`
- `/Users/yangyang/Github/noxinfluencer_skills/skills/noxinfluencer/references/marketing-ops.md`
- `/Users/yangyang/Github/kol_claw/cli/README.md`
- `/Users/yangyang/Github/kol_claw/AGENTS.md`

### 3.1 Skill 是 agent-first，不是用户手动 CLI 教程

Skill 文件明确写了：

```text
The user does not operate the CLI. You do. Run commands silently, tell the user the result.
Only share URLs when the user needs to take action in a browser.
```

所以聚星的 Quick Start 不应该设计成“用户逐行复制命令”。正确设计是：

```text
用户在 Codex / Claude Code / OpenClaw / Hermes 里表达业务目标
-> agent 根据 Skill 运行 CLI
-> 只有登录、注册、授权、订阅等必须浏览器完成的动作才给用户 URL
-> agent 返回业务结果、证据、下一步和 SaaS 直达链接
```

### 3.2 真实账号连接路径是 browser login 优先，API key fallback

CLI README 的 Quick Start 是：

```bash
noxinfluencer login
```

该命令会打开 NoxInfluencer 浏览器登录，复用 SaaS 登录 session，创建或复用 non-expiring API key，并保存到本地。

手动 API key 路径只是 fallback：

```text
https://www.noxinfluencer.com/skills/dashboard?utm_source=skill&utm_medium=cli
https://cn.noxinfluencer.com/skills/dashboard?utm_source=skill&utm_medium=cli
```

聚星页面和文案不能把 API key 作为默认主路径，否则会偏离当前产品事实。

### 3.3 真实自检命令不是泛泛的 “status check”

Skill / CLI 已有自描述和诊断能力：

```bash
noxinfluencer doctor
noxinfluencer schema --all
noxinfluencer quota
noxinfluencer pricing
noxinfluencer agent exit-codes
```

因此 Quick Start 应直接使用这些能力：

- `doctor`：检查 CLI、配置、连通性和 auth。
- `schema --all`：确认 command tree 是否包含 `campaign`、`collection`、`email`、`message`、`crm`、`product`、`brand-monitor`、`export`、`agent`。
- `quota`：展示 Skill quota，并用 `action.url` 引导续费或升级。
- `schema <cmd>`：让 agent 自己发现参数，不靠页面硬编码所有 flags。

### 3.4 当前能力已经覆盖完整达人营销链路

Skill/CLI 当前覆盖：

- 达人发现：`creator search`、`creator lookalikes`、`creator search-filter*`
- 达人读取与分析：`creator profile/audience/content/cooperation`
- 直接从外部标识读取：`--url` 或 `--platform --channel-id`
- 联系信息：`creator contacts`，仅在用户明确要可见或导出联系方式时使用
- 视频监控：`monitor list/create/add-task/tasks/history/summary`
- 业务对象：`campaign`、`collection`、`crm`、`email`、`message`、`product`、`export`
- 品牌监控：`brand-monitor ...`
- 反馈：`feedback submit/inbox/get`
- MCP：当前 Remote MCP Phase 1 以 read-only tools 为主

这意味着 `/skills` Hero 不应只写“达人发现、分析、名单沉淀”。更准确的价值范围是：

```text
发现、分析、建联、谈判、效果监控和运营对象管理。
```

页面上可以压缩成：

```text
AI 持续推进达人发现、分析、建联与效果监控。
```

### 3.5 写操作必须按 Skill 现有安全模型表达

Marketing ops 规则已明确：

- 写命令默认 dry-run。
- staged workflow 先 `validate`，再 `preview`，再 `apply --force`。
- `send` / `schedule` 必须确认 task/thread、recipient、sender、time、content。
- 平台内 email outreach 应用 `creator_id` 添加 recipients，不要先调用 `creator contacts`。
- `creator contacts` 只在用户明确需要外部联系方式或导出邮箱时使用。

所以新用户路径不需要设计复杂的“权限分层教育”。更重要的是让用户看到：

```text
AI 可以准备草稿和预览，但真正发送、批量修改、导出前会让你确认。
```

## 4. 聚星转化问题重定义

聚星的新用户路径不能只看“访问 `/skills`”或“拿到 API key”。真正问题是：

```text
用户是否从第一次访问走到一个有用、可信、可继续管理的达人营销对象。
```

当前主要 gap：

1. 首屏价值需要覆盖完整达人营销链路，而不是只讲发现和名单。
2. Quick Start 必须按真实 Skill/CLI 运行方式设计，不是泛泛的 runtime 模板。
3. SaaS dashboard 已经存在，问题是 Skill 输出后要给直达 URL，否则用户进入平台会迷路。
4. 不同 runtime 的文案差异必须来自连接方式和执行机制，不应凭空写“Codex 适合什么、Hermes 适合什么”。
5. API key、usage、billing 是支撑模块，不是新用户 first value。

推荐的统一激活定义：

```text
Skill activation =
完成登录或授权
+ 自检通过
+ 首个低风险 Skill 任务成功
+ 返回可继续操作的聚星对象或直达链接
```

## 5. 聚星目标体验地图

| Stage | 用户心智 | 产品要回答 | 主要风险 | 设计要求 |
|---|---|---|---|---|
| 1. 看到入口 | 这能帮我做什么 | AI 能持续推进达人营销链路 | 首屏只讲工具或 API | Hero 用业务结果表达 |
| 2. 选择开始方式 | 我在哪里用 | Web / Codex / Claude Code / OpenClaw / Hermes / MCP / API 的差异 | 首页变 runtime 说明书 | 首页轻分流，runtime 页深解释 |
| 3. 连接账号 | 能不能跑起来 | browser login 优先，API key fallback | 要用户手动找 key | `noxinfluencer login` 和 dashboard fallback 并列 |
| 4. 自检 | 是否连对了 | doctor、quota、schema、command tree | 失败原因不清 | 返回 action.url / action.hint |
| 5. First task | 第一件事做什么 | creator URL analysis 或 creator search | 任务太难或太危险 | read / analyze 优先 |
| 6. First value | 结果可信吗 | 结论、证据、数据边界、下一步 | 只返回文本 | 结构化结果 + creator_id / object_id |
| 7. SaaS handoff | 结果去哪了 | 打开 creator / collection / monitor / email task | 平台功能太多找不到 | Skill 返回 deep link |
| 8. Repeat / pay | 为什么回来或付费 | 继续分析、保存更多、监控、触达、额度解释 | 先收费后价值 | 在对象上下文解释升级 |

优秀产品经理会先盯住一个问题：

```text
用户第一次成功之后，是否自然知道下一步打开哪里、继续做什么、为什么值得回来。
```

## 6. 页面与路径方案

### 6.1 `/skills` landing

目标：让用户 5 秒内知道“聚星 Skill 能帮我推进完整达人营销工作”，并启动第一个低风险任务。

建议首屏：

```text
H1：让 AI 7×24 推进达人营销全链路
Subcopy：从达人发现、分析、建联谈判到效果监控，聚星把数据、流程和结果沉淀回 SaaS 工作台。
Primary CTA：让 AI 跑第一个达人分析
Secondary CTA：Add to your AI / 查看安装方式
Tertiary：进入聚星工作台
```

首屏证明点：

- 账号复用：使用现有 NoxInfluencer SaaS 账号登录。
- 数据可信：结果带 creator_id、数据时间、关键指标和证据边界。
- 可继续管理：分析结果可进入 creator、collection、monitor、email task 等聚星对象。
- 安全边界：发送、批量修改、导出等动作前先预览并确认。

任务卡建议：

| Task card | 页面文案 | First task 类型 |
|---|---|---|
| Analyze creator URL | 粘贴达人主页，AI 给出合作判断 | read / analyze |
| Find creators | 说出品类、地区和平台，AI 找候选达人 | read / search |
| Build shortlist | 对比候选人并保存到名单 | analyze / save draft |
| Start outreach | 基于已选达人准备建联任务 | draft / preview |
| Monitor performance | 追踪合作视频表现和趋势 | read / monitor |

不建议：

- 首屏把 API key 作为主 CTA。
- 首屏堆 MCP、REST、CLI 等协议说明。
- 首次激活前要求信用卡。
- 首屏承诺无人监管的自动发送、自动谈判或自动导出。

### 6.2 Runtime Quick Start

当前文档原来的统一模板不符合真实 Skill 使用方式。正确做法不是让用户按 runtime 手动复制一串命令，而是给每个 runtime 一个相同的 activation loop，并把具体执行交给 agent。

统一 activation loop：

```text
1. 安装对应 runtime 的 noxinfluencer skill
2. 用户在 runtime 里提出业务目标
3. agent 运行 `noxinfluencer doctor`
4. 缺登录时 agent 运行 `noxinfluencer login`，用户在浏览器完成登录/授权
5. agent 运行 `noxinfluencer quota` 和必要的 `noxinfluencer schema <cmd>`
6. agent 运行第一个 read / analyze 任务
7. 结果返回 verdict、证据、creator_id / object_id、SaaS 直达链接和下一步
```

CLI / Skill runtimes 的 Quick Start 页面应给“用户 prompt”，不是只给底层命令：

```text
Prompt 1：检查我的 NoxInfluencer Skill 是否可用，并完成登录和额度检查。
Agent expected actions：
- noxinfluencer doctor
- noxinfluencer login（如未登录）
- noxinfluencer quota
- noxinfluencer schema --all（如需确认命令树）

Prompt 2：分析这个达人主页是否值得合作，并给出证据和下一步：
https://www.youtube.com/channel/...
Agent expected actions：
- noxinfluencer schema creator.profile
- noxinfluencer creator profile --url <url>
- 按需要继续 profile / audience / content / cooperation
- 返回结论、证据、creator_id、后续可打开的聚星链接
```

按 runtime 的差异应这样解释：

| Runtime | 有证据的差异 | 页面重点 |
|---|---|---|
| Codex | Postiz `/codex` 已验证：Codex 在 terminal / IDE 中通过 skill 执行 CLI；聚星 README 也支持 `--agent codex` 安装 | 给 Codex prompt，让 Codex 自己检查、登录、跑 creator URL analysis |
| Claude Code | 聚星 README 支持 `--agent claude-code` 安装；Postiz MCP 页也给 Claude Code 的 MCP terminal 命令 | 说明 Skills CLI 安装路径，避免把 MCP 和 CLI Skill 混成一套 |
| OpenClaw | 聚星 skill metadata 有 OpenClaw install；Postiz `/agent` 强调 OpenClaw 通过 skill eligibility 和 chat app 执行 | 强调 “Add to your AI”，agent 负责运行 CLI，用户只处理登录授权 |
| Hermes | 聚星 README 只验证了 Hermes Skills Hub install / inspect | 只写安装和发现方式，不要未经验证地承诺 long-running / heartbeat 场景 |
| MCP | Postiz `/mcp` 是 link-based connector；聚星 Remote MCP 当前以 read-only tools 为主 | 单独页面讲 OAuth / connect / list tools / quota / read-only first task |
| REST API | 面向系统集成和开发者，不是普通 Skill first value | 放到 `/developer-api`，不要混进 `/skills` 首屏主路径 |

关键原则：

```text
runtime 文案差异来自安装方式、认证方式、执行环境、是否支持 CLI、是否支持 MCP、是否能返回 deep link。
不要为了显得丰富而给每个 runtime 编不同用户人格。
```

### 6.3 为什么 Postiz runtime 页独立，聚星也应拆但不能照搬

Postiz runtime 页独立的原因：

- Runtime 用户通常从 GitHub、搜索、ClawHub、docs、agent marketplace 进入，意图比首页用户更明确。
- 每个 runtime 的连接机制不同，独立页可以降低安装认知负担。
- 首页要维持强付费 trial CTA，不适合展开大量安装细节。
- Postiz 的核心对象是 post / calendar，首次动作相对单一，付费卡墙能过滤非付费用户。

聚星应借鉴独立页，但不能照搬强付费阻塞：

- 聚星当前更需要降低激活门槛，先让用户跑出一次 creator analysis 或 search。
- 聚星任务链路更长，应先按业务任务分流，再按 runtime 承接。
- 聚星已有 SaaS 工作台，runtime 页的关键不是“创建新控制台”，而是告诉用户结果会回到哪个 SaaS 对象。

建议页面结构：

```text
/skills              业务价值首页
/skills/codex        Codex Skill Quick Start
/skills/claude-code  Claude Code Skill Quick Start
/skills/openclaw     OpenClaw Skill Quick Start
/skills/hermes       Hermes install / inspect Quick Start
/skills/mcp          MCP connector Quick Start
/developer-api       REST API / developer trial
```

### 6.4 Dashboard / SaaS handoff

Dashboard 已经是聚星 SaaS 平台，不应在方案中把它说成 key / usage 页面。真正问题是：平台功能多，Skill 运行后用户可能找不到结果。

解决方案优先级：

1. Skill / CLI / MCP 响应里返回对象直达 URL。
2. 错误和额度不足时使用 `action.url` / `action.hint` 引导。
3. `/skills/dashboard` 继续承担 key / usage / setup 支撑，但不作为 first value 终点。
4. 后续可在 SaaS 首页或 Skills dashboard 加 “最近 AI 运行结果”，但这不是第一阶段核心。

建议 deep link 映射：

| Skill 结果 | 应返回的 SaaS 目标 |
|---|---|
| creator profile / analysis | 达人详情页或可打开的 creator URL |
| creator shortlist | collection draft / collection detail |
| monitor add-task / summary | monitor project / task detail |
| outreach draft | email task / message thread / draft detail |
| export task | export detail / download link |
| quota exceeded | usage-billing / plan upgrade |
| auth required | login / signup / oauth authorize |

Agent 输出示例：

```text
已完成达人分析：Avery Cole，结论为 viable with risks。
关键证据：受众地区匹配，近 30 天内容稳定，但合作价格信号偏高。
已保留 creator_id：...
打开聚星继续查看：<creator deep link>
下一步可以：加入名单、继续查受众、创建监控、准备建联草稿。
```

### 6.5 Usage / billing

Usage 页面不要只做账单表，应解释“本月用 Skill 完成了什么”和“剩余额度还能做什么”。

建议保留关键指标：

- 本月 creator analysis / search / monitor / export / outreach draft 次数。
- 已保存 creator、collection、monitor、draft 数。
- 最近一次消耗和剩余额度。
- 升级后能继续做的具体动作。

付费提示必须绑定对象和下一步：

```text
你已保存 18 个候选达人。继续批量分析、导出 contacts 或创建监控，需要升级对应额度。
```

不要用抽象套餐文案替代业务理由。

## 8. 指标与实验

### 8.1 Activation 定义

不要只看 `/skills` 页面访问，也不要只看 API key。建议统一定义：

```text
Skill activation =
auth complete
+ self-check success
+ first low-risk task success
+ object/deep link returned
```

### 8.2 最小事件集

| Event | 说明 |
|---|---|
| `skill_landing_view` | 访问 `/skills` |
| `skill_task_select` | 选择达人分析、找达人、保存名单、监控等任务 |
| `skill_runtime_select` | 选择 Codex / Claude Code / OpenClaw / Hermes / MCP / API |
| `skill_setup_start` | 开始安装、登录或授权 |
| `skill_auth_complete` | 登录或 token 成功 |
| `skill_self_check_success` | doctor / quota / schema 自检成功 |
| `skill_first_task_start` | first task 发起 |
| `skill_first_task_success` | first task 成功返回 |
| `skill_deep_link_returned` | 返回 creator / collection / monitor / draft / billing 等链接 |
| `skill_deep_link_opened` | 用户打开 SaaS 直达链接 |
| `skill_quota_explained` | 展示本次消耗和剩余额度 |
| `skill_upgrade_intent` | 点击升级或增购 |

必要维度：

- `runtime`：web、codex、claude_code、openclaw、hermes、mcp、api。
- `task_type`：creator_url_analysis、creator_search、shortlist_save、monitor_summary、outreach_draft、export。
- `region`：domestic、overseas。
- `user_type`：brand 优先。
- `source`：GA / UTM / referrer / ClawHub / skills.sh / runtime attribution，能取多少取多少，不补假数据。

### 8.3 实验 backlog

| Priority | Experiment | 成功信号 | 风险 |
|---|---|---|---|
| P0 | `/skills` Hero 改为完整达人营销链路 | CTA click、task select、setup start | 文案过度承诺 |

## 10. 最终取舍

聚星对 Postiz 的正确用法是抽象产品结构，而不是复制页面：

- 学习 Postiz：业务结果先行，runtime 独立承接，first task 先确认上下文，再执行业务动作，结果回到 SaaS 对象。
- 不照搬 Postiz：激活前信用卡卡墙、API key 作为主路径、只用一句 human-in-loop 处理写操作安全。
- 聚星必须强化：完整达人营销链路、agent-first 使用方式、browser login 优先、SaaS deep link handoff、基于对象的复用和付费理由。
