# 01 定位与假设

> 迁移日期：2026-03-13
> 结构来源：历史 PRD 迁移稿，当前结果文档见 [05_PRD_v7.7.0.md](05_PRD_v7.7.0.md)
> 对应产品版本：v7.7.0
> 状态：从源 PRD 提炼，未做业务裁剪

## 结论

聚星选星助手不是独立产品线，而是聚星在浏览器插件选星场景的轻量入口。它服务低预算个人卖家与小团队，核心任务是把“前台刷内容找人”的工作流接入聚星的数据、邮箱、CRM 与后续升级链路。

## 定位

- 产品形态：浏览器插件。
- 目标平台：YouTube / TikTok / Instagram。
- 核心价值：在频道页 / 视频页直接叠加可用数据与操作入口，让用户更快判断“值不值得合作”。
- 差异化方向：`数据增强 + 找相似 + 邮箱获取 + 轻量 CRM + 即时邀约` 一体化闭环。
- 当前归属：按 `products/` 下独立目录管理，与 `kol-xxl` 同级。

## 目标用户

- 角色 A：网红投放执行 / BD 专员。高频使用，负责找人、收邮箱、发邀约、维护状态。
- 角色 B：小团队负责人 / 创业团队老板。中频使用，关注候选池、协作冲突、团队产出和是否升级工具。

## 为什么现在做

- EasyKOL 已验证“浏览器插件 + 低价订阅 + 邮箱能力”的低端市场需求。
- 聚星已有多平台数据、受众分析、邮箱与邮件服务能力，具备切入插件场景的基础。
- 当前聚星 Web 产品偏后台，和用户“边刷边选”的前台行为存在明显割裂。
- 如果不做，低端客群会继续把“找相似 + 邮箱”心智绑定到 EasyKOL 等竞品。

## 核心业务假设

- H1：插件入口能覆盖现有聚星后台难以低成本触达的个人卖家 / 小团队。
- H2：对用户来说，“少切页面、快判断、可立即联系”比更复杂的后台能力更重要。
- H3：只要把列表、邮箱、状态、备注回流到聚星，就能为后续智能营销提供升级基础。
- H4：本期无需做复杂跨产品流程，只要保证数据与账号层兼容即可。

## 业务目标

- 建立“聚星选星助手”的低端插件市场认知。
- 将单个博主的合作判断耗时降低 50% 以上。
- 沉淀可复用的数据资产，统一进入聚星数据层。
- 为智能营销等上层产品预留升级路径。

## 非目标

- 本期不做完整活动管理、预算与合同等重流程。
- 本期不做插件内完整邮件自动化编排，批量发送仍跳转聚星邮件中心。
- 本期不做深度效果归因和 ROI 分析。
- 本期不做细粒度组织权限控制。

## 待验证

- G1 的业务目标数值和品牌目标尚未量化。
- 与 EasyKOL 的具体差异化优先级仍需继续校准。
- 插件能否作为长期独立入口，还是仅作为聚星存量能力的前端壳，仍待后续验证。
