## 1. 背景与目标

当前目标不是一次性建设完整平台，而是从 0 到 1 搭建一个可投入内部运营的 OpenClaw 控制面。

一期 A 的目标是建立如下最小闭环：

`内部录入品牌 -> 自动创建独立 OpenClaw 实例 -> 可查看状态 -> 可升级 -> 可停用/销毁 -> 全程可审计`

设计要求：

- 一期只服务聚星内部人员
- 运行环境基于公司现有 Kubernetes
- 云基础设施基于 UCloud 托管服务
- 品牌实例按独立隔离方式运行
- 设计上必须保留后续 B/C 阶段扩展挂点

## 2. 已确认范围

### 2.1 一期 A 包含的内容

- 聚星内部管理后台
- 品牌租户管理
- OpenClaw 标准运行时封装
- Kubernetes 上的品牌实例创建、销毁、升级、状态查询
- 基础监控、审计日志、配额与 token 限制
- 接公司现有账号登录体系
- 品牌配置导出基础能力

### 2.2 一期 A 明确不交付的内容

- 客户自助后台
- 完整审批流和 onboarding 流程产品化
- 双层代理和 sidecar 出站控制的完整实现
- 完整计费结算系统
- 自动化跨集群迁移编排
- 多地域、多集群复杂调度体系

### 2.3 当前已明确的业务前提

- 一期范围固定为 A
- 当前没有现成 OpenClaw 部署资产，需要从 0 到 1 搭建
- 云厂商为 UCloud
- Kubernetes 已有，可作为运行底座
- 租户创建方式先采用内部人工录入
- 目标规模为 10-50 个品牌实例
- 活跃使用时间主要集中在 8:00-18:00
- 内部后台登录直接接现有账号体系
- 后端研发常用 Java 或 Python

## 3. 方案对比与推荐路线

### 3.1 方案 A：模块化单体控制面（推荐）

将内部后台、租户管理、实例管理、升级、监控汇总、审计、配额策略统一放入一个控制面后端中，通过模块边界隔离职责；实例创建、升级、销毁由后端中的编排模块执行，底层对接 Kubernetes 和 UCloud 托管资源。

优点：

- 一期落地最快
- 对 10-50 个品牌实例的规模足够
- 工程复杂度最低
- 后续可以按模块拆服务

缺点：

- 中后期如果编排复杂度大幅上升，可能需要拆分编排模块

### 3.2 方案 B：控制面 + 多微服务

将租户、编排、监控、审计、配置等拆成多个服务。

优点：

- 服务边界清晰
- 理论上更便于独立扩缩容

缺点：

- 一期工程成本明显过高
- 研发和运维成本高
- 对当前规模是过度设计

### 3.3 方案 C：纯 GitOps 驱动

控制面只负责生成配置，实际部署完全依赖 GitOps 工具执行。

优点：

- 配置审计天然较强
- 适合高度平台化团队

缺点：

- 一期交互链路长
- 状态回传、失败重试和即时运维体验较差
- 不适合从 0 到 1 的最短路径

### 3.4 推荐结论

推荐采用方案 A：模块化单体控制面 + 标准化 OpenClaw Runtime + Kubernetes 编排执行。

原则是：

- 一期优先解决“能稳定开通和管理独立品牌实例”
- 控制面代码按未来可拆服务的边界组织
- 不为了未来假想规模牺牲当前落地速度

## 4. 总体架构

一期 A 采用三层控制面：

- `Admin Web`
  内部运营后台，提供品牌录入、实例开通、升级、停用、状态查看、审计查看
- `Control Plane`
  核心后端，负责租户、实例、任务、配额、审计、登录体系接入，是系统事实来源
- `OpenClaw Runtime on Kubernetes`
  每个品牌一个独立实例，按统一模板部署到 Kubernetes，运行时配置由控制面注入

架构关系如下：

```text
内部运营/管理员
        |
        v
   Admin Web
        |
        v
  Control Plane
        |
        +--> PostgreSQL
        +--> Redis
        +--> Object Storage
        +--> Secret Manager
        |
        v
Kubernetes / OpenClaw Runtime
        |
        v
每品牌独立实例
```

该结构的关键点是控制面共享、品牌运行时隔离。后续客户后台、安全代理、计费系统都只需要在 Control Plane 基础上扩展，不需要推翻一期架构。

## 5. 核心模块职责

建议在一个后端服务内按模块划分：

- `auth-integration`
  对接现有登录体系，完成用户身份解析和角色映射
- `brand-domain`
  品牌主数据、配额策略、品牌状态管理
- `instance-domain`
  实例记录、发布记录、生命周期状态机
- `orchestration`
  对接 Kubernetes 和 Helm，执行创建、升级、销毁
- `runtime-template`
  生成 OpenClaw 运行时配置和挂载内容
- `task-center`
  管理异步任务、重试、并发控制、执行日志
- `audit-observability`
  审计日志、状态汇总、资源与 token 视图
- `admin-api`
  为内部后台提供 API

这样部署仍然简单，但未来需要拆分时，边界已经清晰。

## 6. 数据模型设计

控制面至少需要以下核心数据实体：

- `tenant_brand`
  品牌主体，保存品牌名称、编码、负责人、状态、默认配置、创建人、创建时间
- `brand_instance`
  品牌实例，一期先默认一个品牌对应一个 OpenClaw 实例，保存当前版本、部署状态、namespace、域名、资源规格、最近健康状态
- `instance_release`
  发布记录，保存每次创建、升级、回滚的镜像版本、配置版本、执行结果、执行时间
- `runtime_config_snapshot`
  运行配置快照，保存某次发布时实际注入的配置摘要，供审计、回滚、迁移使用
- `operation_task`
  异步任务表，承载创建、销毁、升级、重启、导出等任务的状态机
- `audit_log`
  审计日志，记录操作人、对象、动作、结果、来源、时间
- `quota_policy`
  品牌配额策略，先支持资源规格、token 上限、功能开关，后续扩展套餐和计费策略

设计原则：

- 品牌主数据、实例状态、发布历史分离
- 所有变更动作可追踪
- 所有实际注入配置可审计
- 为未来客户后台和计费模型预留稳定主键和边界

## 7. 实例生命周期设计

### 7.1 生命周期状态机

- `draft`
  品牌已创建，但尚未创建实例
- `provisioning`
  正在创建 Kubernetes 资源并等待健康检查通过
- `running`
  实例运行正常
- `degraded`
  实例仍可运行，但健康检查或依赖异常
- `upgrading`
  正在升级版本或配置
- `stopped`
  被人工停用，资源可按策略保留
- `failed`
  创建或升级失败，需要人工介入或重试
- `deleting`
  正在销毁
- `deleted`
  已销毁，仅保留元数据和审计记录

### 7.2 关键动作

- `创建实例`
  后台录入品牌资料后生成标准配置，落库后创建异步任务，由编排层部署 namespace、config、secret、deployment、service、ingress
- `升级实例`
  由运营选择目标版本，系统生成发布记录并执行单实例升级；一期保留批量升级入口，但只需先做单实例可靠升级
- `销毁实例`
  默认采用软销毁 + 元数据归档，真正物理清理由二次确认任务执行
- `导出迁移`
  一期只需导出品牌配置与运行元数据包，为后续正式迁移流程预留基础

### 7.3 生命周期约束

- 所有变更动作必须异步执行
- 同一品牌同一时刻只能存在一个排他性任务
- 所有状态变化都必须写入审计日志
- 所有失败都必须保留可读错误原因和重试入口

## 8. OpenClaw Runtime 标准化

由于当前没有任何现成部署资产，一期必须先定义标准运行时。

OpenClaw Runtime 需要统一以下内容：

- Base Image
- 启动命令和入口脚本
- 目录结构和挂载约定
- 健康检查接口
- 日志输出格式
- 配置注入约定
- 版本规则

建议将两个版本号分开管理：

- `runtime version`
  表示 OpenClaw 运行镜像和运行模板版本
- `config version`
  表示品牌业务配置版本

这一步是整个平台能否稳定运行的地基。如果运行时标准化失败，控制面功能再完整也无法形成可持续交付。

## 9. Kubernetes 部署拓扑

### 9.1 资源组织

- 控制面命名空间：
  `openclaw-control-plane`
- 品牌实例命名空间：
  `openclaw-brand-{brandCode}`

每个品牌实例至少包含以下 Kubernetes 资源：

- `Deployment`
- `Service`
- `Ingress`
- `ConfigMap`
- `Secret`
- 可选 `PVC` 或对象存储挂载方案

### 9.2 推荐每品牌独立 namespace 的原因

- 权限边界清晰
- 便于后续扩展 NetworkPolicy、资源配额、日志筛选
- 销毁和迁移边界明确
- 便于未来增强品牌隔离能力

### 9.3 统一标签

建议所有资源统一打以下标签：

- `app=openclaw`
- `brand_code=xxx`
- `managed_by=control_plane`
- `runtime_version=xxx`
- `config_version=xxx`

后续日志、监控、批量升级、计费都依赖这些标签串联。

## 10. 配置与文件注入策略

计划中的关键文件包括：

- `SOUL.md`
- `AGENTS.md`
- `brand_profile.md`
- `MEMORY.md`
- `skills/`
- `openclaw.json`

建议分层处理：

- 文本类静态配置通过 `ConfigMap` 注入
- 凭证类配置通过 `Secret` 注入
- 技能包和模板源保存在对象存储或版本仓库
- 控制面生成每次发布的最终配置快照

设计目标：

- 分离模板源和实例实际注入结果
- 支持审计、回滚和迁移
- 避免运行中实例直接依赖不可追踪的外部模板

## 11. 技术选型建议

### 11.1 推荐组合

- 后端：`Java 21 + Spring Boot`
- 数据库：`PostgreSQL`
- 缓存与队列：`Redis`
- 前端：`React + 管理后台组件库`
- 部署：`Kubernetes`
- 包装：`Docker Image + Helm Chart`

### 11.2 推荐 Java 作为控制面主后端的原因

- 团队已有使用基础
- 企业内部控制台、状态机、权限接入、任务编排场景较适合 Spring Boot
- 对未来长期维护和标准化更友好

Python 可以继续作为集成脚本、运维工具或辅助任务能力使用，但不建议作为一期控制面主服务。

## 12. 一期需要的基础设施前提

至少需要以下资源：

- UCloud 托管 PostgreSQL
- UCloud 托管 Redis
- UCloud 对象存储
- 密钥管理能力

## 13. 一期实施里程碑

### M0：运行基线打通

- 拉通 OpenClaw 源码、容器化方式、启动依赖
- 制作第一个可运行的标准镜像
- 明确探针、目录、日志、配置约定

### M1：运行时标准化

- 定义品牌实例目录结构和配置生成规则
- 完成 Helm Chart 或等价部署模板
- 在测试环境手工部署单个品牌实例

### M2：控制面最小闭环

- 品牌录入
- 实例创建任务
- 实例状态查询
- 审计日志
- 登录接入

### M3：实例运维能力

- 升级
- 停用和销毁
- 重试和失败处理
- 品牌配额与 token 限制基础版

### M4：观测与运营增强

- 品牌维度日志和指标视图
- 告警
- 配置快照
- 导出包

当 M3 完成时，一期已经具备对外服务所需的核心运营能力；M4 用于提升持续运营稳定性。

## 14. 一期成功标准

建议将验收标准写成可验证条目：

- 可以创建至少一个新品牌并自动在 Kubernetes 中拉起独立实例
- 控制面可查询实例状态和最近一次发布记录
- 可以执行实例版本升级并返回可追踪结果
- 可以安全停用或销毁实例
- 所有变更动作均有审计日志
- 品牌配置与实例版本之间可建立追踪关系
- 控制面成功接入现有登录体系并限制内部访问

## 15. 风险与边界

主要风险：

- OpenClaw 运行方式如果不稳定，一期难点将转移到运行时标准化
- 登录体系接入资料不完整会影响联调
- 品牌配置格式如果早期频繁变化，需要及时做模板版本化
- 若一期同时带上 sidecar、安全代理、客户后台，项目复杂度会明显失控

主要边界：

- 一期重点是“可管、可看、可控”
- 不追求一步到位的平台化能力
- 不承诺复杂迁移、精细计费和大规模调度

## 16. 为后续 B/C 阶段预留的扩展挂点

虽然一期只做 A，但设计上应明确保留以下扩展点：

- `tenant_brand` 与 `brand_instance` 分离
  为后续客户后台、一品牌多实例、多环境扩展留空间
- `quota_policy` 独立
  为后续套餐、计费、限流策略留空间
- `runtime_config_snapshot` 独立
  为后续迁移、回滚、审计提供基础
- `task-center` 独立模块
  为后续审批流、批量升级、长流程编排提供基础
- 统一标签和版本号体系
  为后续成本、监控、运营看板扩展做准备
- Control Plane 作为统一事实来源
  为后续客户后台、安全代理、计费系统复用同一套主数据和状态

## 17. 我会如何落地实施

如果交由我来实现，我会按以下顺序推进：

1. 先完成 OpenClaw Runtime 标准化，不跳过运行底座
2. 再做控制面最小闭环，只实现品牌录入、实例创建、状态查询、审计
3. 再补实例升级、停用/销毁、失败重试
4. 最后补观测、导出、配额和 token 限制

这样做的原因是：

- 优先闭环最短路径
- 把最高不确定性前置验证
- 保持每个里程碑都能独立验收

## 18. 仍需补齐的信息

以下信息不影响当前设计方向，但会影响正式实施方案细节：

- OpenClaw 源码来源、许可证和可二次封装边界
- 登录体系的具体接入协议和字段格式
- UCloud 上实际使用的对象存储、密钥管理、镜像仓库产品

- 谁来维护共享 AGENTS、skills、SOUL 的基础模板

## 19. 结论

一期 A 的正确目标不是构建完整平台，而是构建一个可交付、可运营、可扩展的内部控制面。推荐采用：

`模块化单体控制面 + 标准化 OpenClaw Runtime + Kubernetes 上的品牌独立实例`

这条路线可以在保持复杂度可控的前提下，最快形成业务闭环，同时为后续客户后台、安全代理、计费、迁移等能力留下稳定挂点。
