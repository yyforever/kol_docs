## Eval Case 1：会话真实持久化

### 输入

- 可运行的 `kol-agent` prototype
- 用户完成至少一轮会话后刷新或重新进入的结果

### 期望行为

- 能看到之前的会话列表
- 能进入已有会话继续聊
- 历史内容仍存在

### 评判维度

- 持久化真实性
- 继续会话闭环是否成立
- 是否只是前端演示态

## Eval Case 2：实例与账号边界正确

### 输入

- prototype 的账号模型
- 主账号与子账号访问行为

### 期望行为

- 主账号之间实例隔离
- 子账号不是独立实例
- 子账号在共享实例下按 `session` 级隔离继续使用

### 评判维度

- 实例隔离是否符合当前共识
- 子账号边界是否被误实现
- 共享记忆边界是否被擅自改写

## Eval Case 3：聊天软件接入不是纯占位

### 输入

- 主站中的接入管理入口
- 当前接入尝试记录

### 期望行为

至少满足以下之一：

- 已打通一条真实链路，当前优先个人微信
- 尚未打通，但已经有明确阻塞点和替代验证方式

### 评判维度

- 是否存在真实接入尝试
- 是否有清晰阻塞记录
- 是否把未验证接法误写成已打通

## Eval Case 4：OP 支持面没有越界

### 输入

- 当前 `OP` 界面
- `OP` 动作范围说明

### 期望行为

- 存在简单 `OP` 界面
- 有基础治理入口或基础状态
- 首期动作仍保持待收口状态

### 评判维度

- `OP` 是否服务用户侧主线
- 是否把 demo 当成首期承诺
- 是否出现控制面反客为主

## Forbidden Actions

- 把前端演示态或内存态当成“会话已持久化”
- 把子账号实现成独立实例，或擅自改变共享记忆边界
- 把个人微信之外的渠道误写成当前第一优先
- 把未验证的 `UC / UCloud` API 或接入方案写成已确认能力
- 把 `OP` demo 中的按钮和路由直接写成首期承诺
- 重新引入 `llm gateway` 或把 `Hermes` 写成本期实现目标
- 对没有实证的数据补脑，不显式保留 `ToDo`
