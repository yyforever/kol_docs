## Eval Case 1：会话真实持久化

### 输入

- 可运行的 `kol-agent` prototype
- 用户完成至少一轮会话后刷新或重新进入的结果

### 期望行为

- 能看到之前的会话列表
- 能进入已有会话继续聊
- 历史内容仍存在

### 评判维度

- 持久化真实性
- 继续会话闭环是否成立
- 是否只是前端演示态

## Eval Case 2：实例与账号边界正确

### 输入

- prototype 的账号模型
- 主账号与子账号访问行为

### 期望行为

- 主账号之间实例隔离
- 子账号不是独立实例
- 子账号在共享实例下按 `session` 级隔离继续使用

### 评判维度

- 实例隔离是否符合当前共识
- 子账号边界是否被误实现
- 共享记忆边界是否被擅自改写

## Eval Case 3：个人微信真实打通

### 输入

- 主站中的接入入口与连接状态
- 个人微信扫码接入流程
- 客户通过微信发消息的结果

### 期望行为

- 用户可通过扫码完成个人微信接入
- 主站能展示真实连接状态
- 客户通过个人微信发出的消息能进入 `OpenClaw`
- `OpenClaw` 返回结果能回到微信侧

### 评判维度

- 个人微信是否形成真实双向闭环
- 是否把扫码入口误写成已打通
- 是否把其他渠道误写成当前第一优先

## Eval Case 4：主站到 OpenClaw 的落层边界正确

### 输入

- 前后端实现
- 浏览器请求链路
- 主站后端接口与适配层

### 期望行为

- 浏览器只访问主站接口，不直接暴露 `UC / OpenClaw`
- Java 后端承接实例映射、`session` 关系和运行时转发
- 前端 `node` 层继续只承担 SSR 与 `/ws` 代理

### 评判维度

- 状态和关系边界是否落对层
- 是否错误把 `UC / OpenClaw` 集成塞进前端 `node` 层
- 是否存在浏览器直连 `UC / OpenClaw`

## Forbidden Actions

- 把前端演示态或内存态当成“会话已持久化”
- 把子账号实现成独立实例，或擅自改变共享记忆边界
- 把个人微信之外的渠道误写成当前第一优先
- 把扫码入口误写成“已完成微信打通”
- 把未验证的 `UC / UCloud` API 或接入方案写成已确认能力
- 让浏览器或前端 `node` 层直接承担 `UC / OpenClaw` 正式业务集成
- 把 `OP` demo 中的按钮和路由直接写成首期承诺
- 重新引入 `llm gateway` 或把 `Hermes` 写成本期实现目标
- 对没有实证的数据补脑，不显式保留 `ToDo`
