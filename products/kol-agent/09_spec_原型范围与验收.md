# 09 Spec — 原型范围与验收

> 状态：工作材料 v0.1
> 更新：2026-04-24
> 用途：Heavy 输入材料，供实现、评审与 promotion 使用
> 当前生效版若已 promotion，请以 `kol_brain` 中对应 canonical 页面为准

---

## 核心输入输出断言

### 输入

- 主站用户已登录
- 用户从左侧导航进入 `/agent/dashboard`
- 用户选择已有会话或发起新会话
- 用户在主站中发送消息

### 输出

- 请求经 Java 后端正确路由到与主账号绑定的 SaaS 后台 `OpenClaw` Agent Runtime 实例
- 用户看到返回结果
- 会话列表与账号到 `session` 的关系被真实持久化
- 用户可再次进入已有会话继续聊
- `OpenClaw` 侧会话内容可继续读取
- 用户可完成个人微信扫码接入并进入同一条会话链路

## 边界条件

- 每个主账号对应一个 `OpenClaw` 实例
- 主账号之间实例隔离
- 主账号下子账号只做 `session` 级隔离
- 子账号共享实例，也共享 `OpenClaw` 内部记忆
- `OpenClaw` 服务承担 SaaS 后台的 Agent Runtime 服务
- `OpenClaw` 不直接对外暴露
- 主站负责鉴权
- 模型请求默认经 `Sub2API`
- 前端 `node` 层继续只做 SSR 与 `/ws` 代理
- Java 后端负责 `OpenClaw` 集成、实例映射与 `session` 关系持久化
- 大文件只预留对象存储方向，不在本材料里写死供应商和数据边界

## 最小页面流

### 用户侧

1. 用户进入主站中的 `kol-agent` 页面
2. 用户看到会话列表
3. 用户进入已有会话继续聊，或发起新会话
4. 用户发送消息并收到回复
5. 用户离开后再次进入，仍能看到会话列表和会话内容
6. 用户完成个人微信扫码接入，并看到真实连接状态

## 验收条件

### 必须满足

- 主站用户可进入 `kol-agent`
- 存在会话列表
- 存在“进入已有会话继续聊”能力
- 会话内容和列表是持久化的，不是纯前端 mock
- 主站请求能到内部 `OpenClaw` 并返回结果
- 个人微信扫码接入真实打通
- 客户可通过个人微信与 `OpenClaw` 对话
- 浏览器不直接暴露 `UC / OpenClaw`，前端 `node` 层不承接正式业务集成

### 明确不作为本轮验收前提

- `OP` 控制台
- `UC / UCloud` API 全量接入
- `Hermes` 落地实现
- 对象存储最终选型
- 定价和收费模式拍板
