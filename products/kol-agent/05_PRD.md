# 05 PRD — 原型版开工文档

> 状态：草稿 v0.1
> 更新：2026-04-22
> 适用范围：只覆盖 `kol-agent` 原型，不覆盖正式商业版
> 本版以 `2026-04-21` 会议共识和后续澄清为准；若与旧文档冲突，以本文件为准

---

## 一、文档目标

这份 PRD 只回答一个问题：

**为了尽快做出可演示、可验证方向正确性的 `kol-agent` 原型，这一版到底要做什么，不做什么。**

它不是正式商用 PRD，也不试图一次性写清未来所有功能、后台治理动作、定价和底层运维细节。

---

## 二、一句话定义

`kol-agent` 原型版是聚星主站中的一个网红营销 Agent 入口：用户可以在主站里发起和继续会话，系统把请求路由到内部 `OpenClaw` 实例，并为后续聊天软件接入和独立 `OP` 控制台预留路径。

---

## 三、原型目标

### 3.1 这版要验证什么

原型阶段重点验证 4 件事：

1. 主站承载用户侧会话体验是否可行。
2. `UCloud` 托管 `OpenClaw` 作为底层运行时是否能跑通最小闭环。
3. “每个主账号一个 `OpenClaw` 实例，子账号只做 `session` 级隔离”的模型是否能在产品和工程上站住。
4. 聊天软件接入管理是否可以先在主站中占住产品位，并为后续真实接入留出路径。

### 3.2 原型成功标准

满足以下条件即可视为原型达标：

- 主站用户登录后，可以进入 `kol-agent` 页面并发起对话。
- 用户能看到会话列表，并进入已有会话继续聊。
- 主账号和子账号的访问边界符合当前共识：
  - 主账号对应自己的 `OpenClaw` 实例
  - 子账号在同一主账号下仅做 `session` 级隔离
- 系统能把主站请求转发到内部 `OpenClaw` 服务并返回结果。
- 主站中存在“聊天软件接入管理”的原型入口。
- 团队能基于这一版继续调研：
  - `UC / UCloud` 的控制 API
  - 真实聊天软件接入方式
  - 后续 `OP` 控制台范围

## 四、用户与角色

### 4.1 对外用户

- 主站主账号用户
- 主站子账号用户

### 4.2 对内角色

- `OP` 运维人员

说明：

- 这期原型的主线仍是主站用户侧能力。
- `OP` 侧另有一条支持面，但当前按“独立内部控制台”理解，且动作范围先不在本 PRD 中写死。

---

## 五、原型范围

### 5.1 In Scope

### A. 主站用户侧

- `kol-agent` 入口页
- 会话列表
- 进入已有会话继续聊
- 新发起一轮对话
- 展示基本会话内容
- 主站内的聊天软件接入管理入口

### B. 应用服务侧

- 主站鉴权后访问 `kol-agent`
- 主账号到 `OpenClaw` 实例的映射
- 子账号到 `session` 的访问控制
- 把用户消息转发到内部 `OpenClaw`
- 返回模型结果到主站
- 维护最小会话元数据
  - 会话 ID
  - 所属主账号
  - 所属子账号或可见范围
  - 最近消息时间
  - 会话标题或摘要

### C. 原型期必须明确的技术方向

- 底层运行时使用 `UCloud` 托管 `OpenClaw`
- 模型请求默认经 `Sub2API`
- `OpenClaw` 不直接对外暴露
- 主站负责用户鉴权
- 大文件默认按对象存储方向预留，但不在本期固化选型和边界

### 5.2 Out of Scope

以下内容明确不在这版原型里写成必须交付：

- 完整的 `OP` 控制台功能清单
- `UC / UCloud` 控制 API 的完整接入
- 自动化实例编排平台
- `llm gateway`
- `Hermes` 实现
- 完整记忆迁移方案实现
- 对象存储最终选型与持久化边界定稿
- 定价、收费模式、套餐体系
- 正式商用权限系统与审计体系

## 六、核心产品要求

### 6.1 用户侧最小闭环

用户进入主站后，应当能完成以下最小闭环：

1. 进入 `kol-agent`
2. 查看已有会话列表
3. 进入某个已有会话继续聊，或新开一轮对话
4. 发送消息
5. 获得 `OpenClaw` 返回结果

### 6.2 会话管理最小要求

当前只固定两项：

- 会话列表
- 进入已有会话继续聊

其他动作暂不写死，包括但不限于：

- 删除
- 归档
- 搜索
- 重命名
- pin

这些都保留为后续 `ToDo`。

### 6.3 聊天软件接入管理

主站中需要有“聊天软件接入管理”的产品位，但原型阶段不要求把所有渠道真实打通。

当前渠道优先级共识是：

1. 个人微信
2. 企业微信
3. 飞书
4. 钉钉

原型阶段至少要做到：

- 主站中有接入管理入口
- 能表达“未来可接聊天软件”的产品方向
- 为后续真实接入预留配置和状态位置

是否必须在原型阶段打通第一条真实聊天软件链路，本版暂不写死。

### 6.4 账号与隔离

当前必须遵守以下模型：

- 每个主账号一个 `OpenClaw` 实例
- 主账号之间实例隔离
- 主账号下子账号仅做 `session` 级隔离
- 子账号共享同一个实例，也共享记忆

原型不对 `OpenClaw` 内部共享记忆机制做额外覆盖。

## 七、原型信息架构

### 7.1 主站侧页面建议

原型至少需要以下页面或视图：

- `kol-agent` 首页 / 工作区
- 会话列表区
- 当前会话区
- 聊天软件接入管理区

### 7.2 OP 侧支持面

`OP` 侧能力单独理解为“独立内部控制台”。

但这版 PRD 不要求把 `OP` 控制台本身做进原型交付范围，只要求在方案中承认这条支持面存在。

当前只保留治理类别认知，不固化动作清单：

- 租户
- 实例
- 升级
- token 配额
- 迁移
- 审计 / 告警入口

---

## 八、技术实现边界

### 8.1 当前已定边界

- `UCloud` 托管 `OpenClaw` 是当前底层方案。
- `Sub2API` 是默认模型转发路径。
- 主站负责鉴权。
- `OpenClaw` 服务不直接对外暴露。
- 不额外建设新的模型中间层。

### 8.2 当前暂不定边界

以下只保留方向，不做实现承诺：

- 对象存储到底选 `阿里云 OSS` 还是 `UCloud` 对象存储
- 哪些数据必须持久化，哪些只是迁移或备份材料
- `UCloud` 可用控制 API 的范围
- 官方迁移能力、`brainpack`、手工迁移三者最终怎么落

### 8.3 原型阶段可接受的工程取舍

为了尽快做出原型，本版允许：

- 某些实例准备动作先人工或半人工完成
- 聊天软件接入先做占位或半链路验证
- `OP` 支持动作先不自动化

只要不违背已定架构边界，这些都可以接受。

## 九、原型交付物

这版原型建议至少交付：

### 9.1 可演示产品原型

- 主站内的 `kol-agent` 页面
- 会话列表 + 会话详情 + 发送消息
- 聊天软件接入管理入口

### 9.2 最小后端能力

- 主站鉴权后访问 agent 服务
- 找到或绑定主账号对应实例
- 转发消息到内部 `OpenClaw`
- 回传结果
- 记录最小会话元数据

### 9.3 研发结论输出

- `UC / UCloud` 控制 API 能做什么
- 原型阶段聊天软件接入建议先打通哪一条
- `OP` 控制台首期最值得保留的动作是什么

---

## 十、验收标准

原型评审时，用以下标准验收：

### 10.1 功能验收

- 主站用户可进入 `kol-agent`
- 可查看会话列表
- 可进入已有会话继续聊
- 可发送消息并看到返回结果
- 页面上有聊天软件接入管理入口

### 10.2 边界验收

- `OpenClaw` 不对外暴露
- 主站负责鉴权
- 主账号与实例绑定关系清晰
- 子账号不是独立实例，而是 `session` 级隔离

### 10.3 方向验收

- 团队能据此继续判断 `UCloud` 托管方案是否可继续推进
- 团队能据此继续判断真实聊天软件接入怎么切入
- 团队能据此收敛 `OP` 独立控制台的首期动作

---

## 十一、当前 ToDo / 未决问题

这版 PRD 明确保留以下未决问题：

### 11.1 UC / UCloud 侧

- 可用的实例管理 API 有哪些
- 是否支持后续实例接管和控制
- 对存储、迁移、网络边界的支持程度如何

### 11.2 聊天软件接入

- 原型是否必须打通第一条真实渠道
- 如果要打通，个人微信是否就是第一条实施渠道

### 11.3 OP 独立控制台

- 首期最值得先做哪两三类动作
- 哪些动作可以继续依赖人工流程

### 11.4 存储与迁移

- 对象存储的最终供应商
- 持久化边界
- 迁移方案的实际优先顺序是否会受 `UCloud` 能力影响

---

## 十二、这版 PRD 的使用方式

这份 PRD 只用于启动原型。

后续如果方向成立，再继续拆：

- 正式版用户 PRD
- 技术方案文档
- `OP` 独立控制台 PRD
- `UC / UCloud` API 调研结论

在这些结论出来之前，不要把这版原型 PRD 误读成正式商用范围。
