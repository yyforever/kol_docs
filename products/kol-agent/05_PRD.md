## 一、文档目标

这份 PRD 只回答一个问题：

**为了尽快做出可演示、可验证方向正确性的 `kol-agent` 原型，这一版到底要做什么，不做什么。**

它不是正式商用 PRD，也不试图一次性写清未来所有功能、后台治理动作、定价和底层运维细节。

---

## 二、一句话定义

`kol-agent` 原型版是聚星主站中的一个网红营销 Agent 入口：用户可以在主站 `/agent/dashboard` 发起和继续会话，系统把请求路由到承担 SaaS 后台 Agent Runtime 服务的内部 `OpenClaw` 实例，真实持久化账号与会话关系，并打通个人微信扫码接入这条真实链路。

---

## 三、原型目标

### 3.1 这版要验证什么

原型阶段重点验证 4 件事：

1. 主站承载用户侧会话体验是否可行。
2. 作为 SaaS 后台 Agent Runtime 服务的 `UCloud` 托管 `OpenClaw` 是否能跑通最小闭环。
3. “每个主账号一个 `OpenClaw` 实例，子账号只做 `session` 级隔离”的模型是否能在产品和工程上站住。
4. 主站侧是否可以跑通“持久化会话 + 个人微信扫码真实接入 + 正确系统落层”这一版最小原型。

### 3.2 原型成功标准

满足以下条件即可视为原型达标：

- 主站用户登录后，可以从左侧导航进入 `/agent/dashboard` 并发起对话。
- 用户能看到会话列表，并进入已有会话继续聊。
- 用户重新进入页面后，仍能看到已持久化的会话列表和历史内容。
- 主账号和子账号的访问边界符合当前共识：
  - 主账号对应自己的 `OpenClaw` 实例
  - 子账号在同一主账号下仅做 `session` 级隔离
- 系统能经 Java 后端把主站请求转发到内部 `OpenClaw` 服务并返回结果。
- 主站中存在“聊天软件接入管理”的原型入口。
- 原型阶段必须真实打通个人微信扫码链路。
- `OpenClaw` 负责保存会话内容，主站数据库负责保存实例与 `session` 关系。
- 团队能基于这一版继续调研：
  - `UC / UCloud` 的控制 API
  - 个人微信扫码接入方式
  - 后续是否启动独立 `OP` 控制台 workstream

## 五、原型范围

### 5.1 In Scope

### A. 主站用户侧

- 主站左侧导航中的 `Agent` 入口
- `/agent/dashboard` 页面
- 会话列表
- 进入已有会话继续聊
- 新发起一轮对话
- 展示基本会话内容
- 主站内的聊天软件接入管理入口
- 个人微信扫码真实接入

### B. 应用服务侧

- 主站鉴权后访问 `kol-agent`
- 主账号到 `OpenClaw` 实例的映射
- 子账号到 `session` 的访问控制
- 把用户消息转发到内部 `OpenClaw`
- 返回模型结果到主站
- 主站数据库持久化账号、实例和 `session` 关系
- `OpenClaw` 侧持久化会话内容并可继续读取

### C. 原型期必须明确的技术方向

- 底层运行时使用 `UCloud` 托管 `OpenClaw`
- `OpenClaw` 服务承担 SaaS 后台的 Agent Runtime 服务
- 模型请求默认经 `Sub2API`
- `OpenClaw` 不直接对外暴露
- 主站负责用户鉴权
- 大文件默认按对象存储方向预留，但不在本期固化选型和边界

### 5.2 Out of Scope

以下内容明确不在这版原型里写成必须交付：

- 完整的 `OP` 控制台功能清单
- `UC / UCloud` 控制 API 的完整接入
- 自动化实例编排平台
- `llm gateway`
- `Hermes` 实现
- 完整记忆迁移方案实现
- 对象存储最终选型与持久化边界定稿
- 定价、收费模式、套餐体系
- 正式商用权限系统与审计体系

## 六、核心产品要求

### 6.1 用户侧最小闭环

用户进入主站后，应当能完成以下最小闭环：

1. 进入 `kol-agent`
2. 查看已有会话列表
3. 进入某个已有会话继续聊，或新开一轮对话
4. 发送消息
5. 获得 `OpenClaw` 返回结果

### 6.2 会话管理最小要求

当前只固定两项：

- 会话列表
- 进入已有会话继续聊

并且这两项都必须建立在真实持久化之上，而不是只做前端演示态。

其他动作暂不写死，包括但不限于：

- 删除
- 归档
- 搜索
- 重命名
- pin

这些都保留为后续 `ToDo`。

### 6.3 聊天软件接入管理

主站中需要有“聊天软件接入管理”的产品位，且原型阶段必须打通个人微信这条真实渠道。

当前渠道优先级共识是：

1. 个人微信
2. 企业微信
3. 飞书
4. 钉钉

原型阶段至少要做到：

- 主站中有接入管理入口
- 用户可通过扫码完成个人微信接入
- 主站可展示真实连接状态
- 客户通过个人微信发出的消息能进入同一条主站 / `OpenClaw` 链路

如果因技术或外部限制无法在原型阶段打通，必须明确记录阻塞点并触发重塑或止损讨论，而不是静默降级为占位入口。

### 6.4 账号与隔离

当前必须遵守以下模型：

- 每个主账号一个 `OpenClaw` 实例
- 主账号之间实例隔离
- 主账号下子账号仅做 `session` 级隔离
- 子账号共享同一个实例，也共享记忆

原型不对 `OpenClaw` 内部共享记忆机制做额外覆盖。

## 八、技术实现边界

### 8.1 当前已定边界

- `UCloud` 托管 `OpenClaw` 是当前底层方案。
- `Sub2API` 是默认模型转发路径。
- 主站负责鉴权。
- `OpenClaw` 服务不直接对外暴露。
- 不额外建设新的模型中间层。

### 8.2 当前暂不定边界

以下只保留方向，不做实现承诺：

- 对象存储到底选 `阿里云 OSS` 还是 `UCloud` 对象存储
- 哪些数据必须持久化，哪些只是迁移或备份材料
- `UCloud` 可用控制 API 的范围
- 官方迁移能力、`brainpack`、手工迁移三者最终怎么落

### 8.3 原型阶段可接受的工程取舍

为了尽快做出原型，本版允许：

- 某些实例准备动作先人工或半人工完成
- 某些外部接入准备动作先人工或半人工完成
- `UC / UCloud` 控制 API 的具体收口延后到专项调研页

只要不违背已定架构边界，这些都可以接受。

## 九、原型交付物

这版原型建议至少交付：

### 9.1 可演示产品原型

- 主站内的 `kol-agent` 页面
- 会话列表 + 会话详情 + 发送消息
- 聊天软件接入管理入口
- 个人微信扫码真实链路

### 9.2 最小后端能力

- 主站鉴权后访问 agent 服务
- 找到或绑定主账号对应实例
- 转发消息到内部 `OpenClaw`
- 回传结果
- 持久化账号、实例与 `session` 关系
- 通过 `OpenClaw` 读取和续接会话内容

### 9.3 研发结论输出

- `UC / UCloud` 控制 API 能做什么
- 个人微信扫码接入的可行路径与阻塞点
- 当前 prototype 是否值得继续推进，还是需要重塑或止损

## 十、验收标准

原型评审时，用以下标准验收：

### 10.1 功能验收

- 主站用户可进入 `kol-agent`
- 可查看会话列表
- 可进入已有会话继续聊
- 可发送消息并看到返回结果
- 刷新或重新进入后，仍可看到已保存的会话列表和历史内容
- 页面上有聊天软件接入管理入口
- 已打通个人微信扫码这条真实聊天软件链路
- 客户可通过个人微信与 `OpenClaw` 对话

### 10.2 边界验收

- `OpenClaw` 不对外暴露
- 主站负责鉴权
- 主账号与实例绑定关系清晰
- 子账号不是独立实例，而是 `session` 级隔离
- 浏览器不直连 `UC / OpenClaw`
- 前端 `node` 层不承接正式业务状态，Java 后端负责 `OpenClaw` 集成与关系持久化

### 10.3 方向验收

- 团队能据此继续判断 `UCloud` 托管方案是否可继续推进
- 团队能据此继续判断真实聊天软件接入怎么切入
- 团队能据此判断是否值得再启动独立 `OP` 控制台 workstream

### 11.2 聊天软件接入

- 个人微信扫码接入的真实技术路径是什么
- 如果个人微信无法在原型阶段打通，应如何记录阻塞并触发重塑或止损

### 11.3 OP 独立控制台

- 首期最值得先做哪两三类动作
- 哪些动作可以继续依赖人工流程

### 11.4 存储与迁移

- 对象存储的最终供应商
- 持久化边界
- 迁移方案的实际优先顺序是否会受 `UCloud` 能力影响
