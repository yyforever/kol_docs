# 产品调研技能

## 核心原则

1. **不带预设立场** — 品类扫描只是起点，开放心态发现真正的关键问题
2. **迭代式发现** — 默认3轮迭代，每轮基于上一轮发现进行更深入调研
3. **人类在循环中** — 每轮迭代后与人类讨论洞察和下一轮重点
4. **评分维度驱动** — 每轮调研内容与评分权重保持一致
5. **步骤内并发** — 主流程串行，每个步骤内部拆分并发子任务
6. **频繁写入文档** — 每个子步骤完成后立即更新文档
7. **文档呈现最终结论** — 不分轮次，呈现整体一致的最终结论

---

## 多子方向处理

**一个大方向下可能有多个子方向，必须分别完整调研后再汇总。**

### 流程

```
大方向（如02_达人矩阵管理）
├── 子方向A（如CRM）→ 3轮迭代 → 评分A
├── 子方向B（如微网红）→ 3轮迭代 → 评分B  
├── 子方向C（如矩阵平台）→ 3轮迭代 → 评分C
└── 汇总评分：综合A/B/C，给出大方向建议
```

### 规则

1. **每个子方向独立完成3轮迭代**
2. **每个子方向独立评分**
3. **所有子方向完成后，汇总讨论**：
   - 哪些子方向值得做？
   - 是否合并为一个产品？
   - 优先级排序
4. **子方向可能有重叠**，汇总时去重

### 识别子方向

开始调研前，先检查目录下是否有多个子方向文件（如`机会评分_XXX.md`）。如果有，列出并确认调研顺序。

## 调研前必问（Checklist）

开始调研前，先和产品Owner确认：

### 1. 调研主体的已有定位
- 调研主体（如聚星）已经是什么？
- 已经是一站式平台？还是细分工具？

### 2. "机会"的定义
| 方向类型 | 是否机会 |
|---------|---------|
| 对标传统一站式平台 | ❌ 如果调研主体已是一站式 |
| 对标**新形式**一站式（Agent驱动、新流量入口） | ✅ 新形式是机会 |
| 对标**细分品类**（专注某场景的独立产品） | ✅ 细分是机会 |

### 3. 独立产品 vs 功能模块
- 这个方向是要做**独立产品**（需验证细分市场规模）？
- 还是**功能模块升级**（只是迭代，不是新方向）？

### 4. 目标用户的核心工具
- **目标用户现在用什么工具？**
- 这些工具必须作为竞品优先调研
- 例：聚星目标是中国TikTok Shop卖家 → 达密是核心竞品

### 5. 竞品分类
- **直接竞品**：核心业务就是这个场景
- **间接竞品**：综合平台含该功能（功能之一，非主打）
- **本土核心工具**：目标用户实际在用的工具

⚠️ 不能把间接竞品的定价/功能当作细分市场的基准

## ⚠️ 首轮前必须验证（新增）

**开始调研前，先验证对调研主体（如聚星）的理解是否正确！**

### 验证清单

| 问题 | 必须确认 |
|------|---------|
| 调研主体是什么类型的公司/产品？ | 搜索官网、Crunchbase确认 |
| 调研主体的数据来源是什么？ | 官方API？爬虫？用户授权？|
| C1-C6资源的具体规模和能力？ | 不要假设，要验证 |
| 调研主体与平台（TikTok等）的真实关系？ | ISV资质≠独家数据权限 |

### 核心原则

> **从场景需要什么出发，验证资源是否能满足，而不是从资源出发假设能用。**

### 常见错误

| 错误假设 | 正确做法 |
|---------|---------|
| "官方服务商"="有官方数据" | 验证API实际开放哪些数据 |
| "ISV资质"="独家数据权限" | ISV只是可以调用授权数据 |
| "有网红数据库"="有竞品情报能力" | 竞品情报需要的数据可能是盲区 |
| 从资源推场景 | **从场景验证资源** |

### 教训案例

> **04竞品达人情报调研教训**：
> 假设聚星有"TikTok官方数据权限"，基于此推导所有分析。
> 实际上：达人GMV数据API根本不对第三方开放，这个假设从一开始就不成立。
> 
> **正确流程**：先问"竞品情报需要什么数据"→再问"这些数据能获取吗"→最后问"聚星资源有用吗"

### 执行规则（重要！）

**步骤之间：串行**
- 必须等上一步骤**完全完成**后，才能开始下一步骤
- 不允许同时启动多个步骤
- 每个步骤完成后汇报，确认后再进入下一步骤

**步骤内部：并发≥4个子任务**
- 每个步骤内部**必须拆分至少4个并发子任务**
- 步骤0（场景画像）：场景+画像+购买行为+现有做法
- 步骤1（竞品发现）：按场景+本土+新进入者分拆
- 步骤2（竞品深挖）：每个竞品一个子任务
- 步骤3（用户痛点）：按渠道/主题分拆
- 步骤4-6：按具体分析维度分拆

**步骤完成 → 强制文档整合** ⭐新增
- **每个步骤完成后，必须立即整合到目标文档**
- 操作顺序：① 读取已有文档 → ② 整合新发现 → ③ 写入更新
- 不允许跳过文档整合直接进入下一步骤
- 详见下方"文档整合操作规范"

**递进关系**：
- 步骤0（场景画像）→ 后续所有步骤：场景理解指导调研方向
- 步骤0-2（竞品）→ 步骤3（痛点）：了解竞品后才能针对性调研用户评价
- 步骤0-3 → 步骤4（优势）：了解竞争和痛点后才能评估差异化
- 步骤2 → 步骤5（PLG）：参考竞品模式
- 步骤3 → 步骤6（开发）：明确需求后评估难度
- 步骤0-6 → 步骤7（JTBD）：基于所有发现做Job分析
- 步骤0-7 → 步骤8（第一性原理）：挑战假设，验证结论

### 执行规则

1. **启动每轮时，先列出执行清单**（复制上面的清单）
2. **每完成一个步骤，打勾并汇报**
3. **每完成一个步骤，立即整合到目标文档** ⭐新增
4. **不得跳过任何步骤**
5. **不得在未完成前面步骤时执行后面步骤**
6. **不得跳过文档整合直接进入下一步骤** ⭐新增
7. **子Agent任务可以并发，但必须覆盖所有步骤**

### 文档整合操作规范 ⭐新增

**每个步骤完成后，必须执行以下操作：**

```
步骤完成
    ↓
① 读取目标文档（如果存在）
    ↓
② 整合新发现（保持与已有内容一致）
    ↓
③ 写入更新后的完整文档
    ↓
④ 确认文档更新完成
    ↓
进入下一步骤
```

**步骤→目标文档映射**：

| 步骤 | 整合到哪个文档 |
|------|---------------|
| 步骤0: 场景画像 | `XXX_调研报告.md` - 场景画像章节 |
| 步骤1: 竞品发现 | `XXX_竞品分析.md` - 竞品清单章节 |
| 步骤2: 竞品深挖 | `XXX_竞品分析.md` - 竞品详情章节 |
| 步骤3: 用户痛点 | `XXX_痛点分析.md` |
| 步骤4: 聚星优势 | `XXX_调研报告.md` - 聚星优势章节 |
| 步骤5: PLG可行性 | `XXX_调研报告.md` - PLG章节 |
| 步骤6: 开发难度 | `XXX_调研报告.md` - 开发难度章节 |
| 步骤7: JTBD分析 | `XXX_调研报告.md` - JTBD章节 |
| 步骤8: 第一性原理 | `XXX_调研报告.md` - 第一性原理章节 |
| 步骤9: 汇总讨论 | `README.md` - 更新子方向状态 |

**整合原则**：

1. **先读后写** — 必须先读取已有文档，了解已有内容
2. **保持一致** — 新内容的格式、术语与已有内容保持一致
3. **整体性** — 不是简单追加，而是融合成完整的章节
4. **覆盖更新** — 如果是同一章节的更新，覆盖旧内容
5. **保留有效信息** — 不要删除仍然有效的已有内容

**整合示例**：

```
# 步骤2竞品深挖完成后

## 操作1: 读取已有文档
read: XXX_竞品分析.md

## 操作2: 检查已有内容
- 已有竞品：A、B、C
- 本轮新增：D、E

## 操作3: 整合写入
- 保留A、B、C的分析
- 新增D、E的分析
- 更新竞品格局总览
- 确保格式一致

## 操作4: 确认
✅ 竞品分析文档已更新（新增D、E）
```

**禁止行为**：
- ❌ 步骤完成后不更新文档直接进入下一步
- ❌ 不读取已有文档就直接覆盖写入
- ❌ 每次都创建新文件而不是更新已有文件
- ❌ 只在轮次结束时才更新文档

### 3轮迭代的典型节奏

| 轮次 | 重点 | 产出 |
|------|------|------|
| **第1轮** | 广度调研，建立基线认知 | 初步竞品格局、痛点概览、假设清单 |
| **第2轮** | 深度验证，修正偏差 | 补充遗漏竞品、验证关键假设、修正结论 |
| **第3轮** | 精准聚焦，确认结论 | 验证剩余问题、形成最终判断 |

**注意**：每轮内容相同，只是深度和准确度递进。

## 子Agent任务指导原则

### 1. 时效性检查（重要）
- **只看最近1年的信息**（除非是基础事实如公司背景）
- 搜索时加时间限制
- 如果信息来源超过1年，标注"⚠️ 可能过时"

### 2. 要求一手来源
- 功能有/无的判断 → 必须来自官网/产品文档
- 不能只依赖二手横评文章

### 3. 验证"空白"结论
- 如果声称"XX功能空白"，必须：
  - 列出已检查的工具
  - 说明检查的来源（官网、文档、商店描述）
  - 如果是二手来源，标注"待验证"

### 4. 用户原话带来源
- 所有用户痛点原话必须带来源链接
- 不能编造或概括性描述

### 5. 区分事实和推测
- 明确标注哪些是确认的事实
- 哪些是推测或需要进一步验证

## 文档规范

### 核心原则

1. **呈现最终结论，不分轮次** — 文档只保留最终版本，不展示迭代过程
2. **更新而非追加** — 多轮迭代更新同一份文档，而非创建新文件
3. **用前缀区分子方向** — 不用子目录，用命名前缀（如`CRM_`、`微网红_`）

### 文档结构

每个机会方向一个目录，**不用子目录**，用前缀区分子方向：
```
XX_方向名/
├── README.md              # 大方向汇总（所有子方向对比+建议）
│
├── 子方向A_调研报告.md    # 汇总报告（含最终评分）
├── 子方向A_竞品分析.md    # 竞品详细分析
├── 子方向A_痛点分析.md    # 痛点详细分析
│
├── 子方向B_调研报告.md    # 同上
├── 子方向B_竞品分析.md
├── 子方向B_痛点分析.md
│
└── _archive/              # 可选：保留原始碎片
```

### 每个子方向3份文档

| 文档 | 内容 |
|------|------|
| **XXX_调研报告.md** | 汇总：聚星优势、PLG可行性、开发难度、JTBD、第一性原理、**最终评分** |
| **XXX_竞品分析.md** | 所有竞品详细分析汇总 |
| **XXX_痛点分析.md** | 所有用户痛点详细分析汇总 |

### 文档更新时机

- **每个步骤完成后**：立即整合到对应的目标文档 ⭐改
- **先读后写**：整合前必须读取已有文档，保持一致性
- **迭代时覆盖更新**，不追加新文件
- 详见上方"文档整合操作规范"
